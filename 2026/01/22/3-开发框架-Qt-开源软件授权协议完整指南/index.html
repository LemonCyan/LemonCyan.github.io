开源软件授权协议完整指南：GPL、LGPL、MIT、Apache 2.0 与 BSD 深度解析

核心概念：Copyleft 与 Permissive

开源许可证分为两大阵营：

类型 代表协议 核心特征
Copyleft (强传染) GPL v2/v3 衍生作品必须保持相同许可证开源
Weak Copyleft LGPL v2.1/v3 仅库本身保持 Copyleft，链接程序可闭源
Permissive (宽松) MIT, BSD, Apache 2.0 允许闭源商用，仅保留版权声明

MIT License

协议原文核心条款

合规要求清单

行为 是否允许 额外义务
商业使用 ✅ 无
修改代码 ✅ 无需公开修改
闭源分发 ✅ 保留版权声明和 MIT 协议文本
更换许可证 ✅ 衍生品可用任意协议
专利授权 ❌ 无明确专利条款

实际操作

局限性

  • 无专利保护：如果贡献者持有相关专利，MIT 协议不提供隐式专利授权
  • 责任免除仅限软件本身：在某些司法管辖区可能存在法律不确定性

GPL (GNU General Public License)

GPLv2 与 GPLv3 核心差异

GPLv3 条款详解

Section 3: 专利授权

每个贡献者自动授予你：

  • 永久、全球、免版税、不可撤销的专利许可
  • 覆盖范围：制造、使用、销售、进口、转让

Section 10: 自动许可升级

实践意义：使用 GPL-2.0-or-later SPDX 标识符可确保向前兼容。

合规要求对比

要求 GPLv2 GPLv3
源码公开 ✅ 分发二进制时必须 ✅ 同左
保留许可证文本 ✅ ✅
标注修改内容 ✅ ✅
专利授权 ❌ 无 ✅ 显式授权
反 Tivoization ❌ ✅ 消费设备需提供安装信息
与 Apache 2.0 兼容 ❌ ✅

静态链接 vs 动态链接

对于 GPL 代码：

LGPL (GNU Lesser General Public License)

设计目的

LGPL 专为共享库设计，允许闭源程序链接使用 LGPL 库，同时保护库本身的开源性。

动态链接 vs 静态链接合规要求

动态链接 (推荐)

合规义务：

  • 分发 LGPL 协议文本
  • 保留库的版权声明
  • 确保用户可以替换 LGPL 库版本

静态链接 (复杂)

链接方式 你的代码需开源? 额外义务
动态链接 ❌ 提供协议文本，允许用户替换库
静态链接 ❌ 提供目标文件 + 库源码 + 重链接说明

LGPLv2.1 vs LGPLv3 差异

特性 LGPLv2.1 LGPLv3
独立协议 ✅ 单独文档 ❌ GPLv3 附加条款
与 Apache 2.0 兼容 ❌ ✅
反 Tivoization ❌ ✅ 消费设备需提供安装信息

Apache License 2.0

核心优势：显式专利授权

专利报复条款

实践意义：使用 Apache 2.0 软件的企业如果对项目发起专利诉讼，将丧失所有专利权。

合规检查清单

BSD License

变体对比

变体 条款数 特殊限制
0-Clause BSD 0 无任何限制，等同公有领域
2-Clause BSD 2 保留版权 + 免责声明
3-Clause BSD 3 + 禁止使用作者名背书
4-Clause BSD (原始) 4 + 广告条款（已弃用）

2-Clause BSD 原文

BSD vs MIT

对比维度 BSD 2-Clause MIT
条款长度 较长 更简洁
核心要求 相同 相同
专利授权 ❌ 无 ❌ 无
社区偏好 系统软件 (FreeBSD, OpenBSD) Web/Node 生态

实际场景速查表

场景 推荐许可证 原因
商业闭源产品使用开源库 MIT / BSD / Apache 2.0 无 Copyleft 传染
企业级共享库 Apache 2.0 专利保护 + 专利报复
希望代码保持开源 GPL v3 强 Copyleft
库允许闭源程序链接 LGPL v3 弱 Copyleft
嵌入式设备 避免 GPLv3/LGPLv3 反 Tivoization 条款
