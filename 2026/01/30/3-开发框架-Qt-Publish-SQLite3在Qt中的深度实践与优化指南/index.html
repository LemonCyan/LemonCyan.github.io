SQLite3 在 Qt 中的深度实践与优化指南

核心机制与原理

Qt SQL 模块架构

Qt 的 SQL 模块用驱动插件架构设计，通过统一 API 接口支持多种数据库后端。SQLite3 集成通过 QSQLITE 驱动实现。

工作流程：

关键细节：

  • Qt 6.x 内置 SQLite 3.35.5 版本
  • 连接格式：QSqlDatabase::addDatabase("QSQLITE", connectionName)
  • 支持相对和绝对路径

线程安全模型

这是 Qt SQLite 开发最关键的问题：

  • SQLite 引擎：默认 Serialized 模式，线程安全
  • QSqlDatabase：非线程安全，连接句柄不能跨线程使用

错误模式：在 Thread A 创建连接，在 Thread B 中复制使用，导致竞争条件。

核心配置

配置参数说明：

参数 作用 副作用
journal_mode=WAL 日志写入模式，提升并发 增加磁盘占用
synchronous=NORMAL 数据刷新频率 OFF 可能导致断电数据丢失
cache_size=-20000 控制页面缓存大小 值过大会占用过多内存
foreign_keys=ON 启用外键约束 影响写入性能
busy_timeout=5000 锁冲突等待时间 值过大会导致 UI 阻塞

已知限制

1. 并发写入限制

即使在 WAL 模式下，SQLite 也只支持单个写入者。

解决方案：使用写入队列（生产者-消费者模式）。

2. 内存限制

SQLite 的 B-Tree 页面默认为 4KB，大型数据集会导致内存碎片。

优化建议：针对大表使用分页查询。

3. 类型系统限制

SQLite 使用动态类型，与 Qt 的强类型系统存在潜在冲突。

安全访问方式：

4. UI 性能陷阱

QSqlQueryModel 默认仅加载前 256 行数据。

优化策略：

  • 需要总行数时单独执行 SELECT COUNT(*)
  • UI 层使用 canFetchMore() 动态加载

5. 扩展性瓶颈

即便开启 WAL，SQLite 在同一物理文件上仍只能有一个 Writer。

解决方案：考虑升级到 PostgreSQL/MySQL。

不适用场景

  1. 分布式事务：不支持两阶段提交（2PC）
  2. 存储过程：无存储过程概念
  3. 全文搜索：FTS5 功能不如 PostgreSQL
  4. 实时分析：不适合 OLAP 场景
  5. 高并发 Web 后端：写扩展性有限

总结与最佳实践

推荐配置

性能优化

  • 启用 WAL 模式（journal_mode=WAL）
  • 设置缓存大小 20MB（cache_size=-20000）
  • 使用 NORMAL 同步模式（synchronous=NORMAL）
  • 启用内存映射（mmap_size=268435456）

安全性

  • 启用外键约束（foreign_keys=ON）
  • 使用预处理语句防止 SQL 注入

线程安全

  • 遵循”每线程一连接”原则
  • 为每个连接设置 BUSY_TIMEOUT
  • 使用作用域隔离管理连接生命周期

适用场景判断

✅ 推荐使用：

  • 桌面应用程序本地数据存储
  • 嵌入式设备数据记录
  • 移动应用（Qt for Android/iOS）
  • 数据分析和报表工具
  • 单机应用配置和缓存

❌ 不推荐使用：

  • 高并发 Web 应用后端
  • 分布式系统
  • 需要复杂存储过程的业务系统
  • 实时分析（OLAP）
  • 需要分布式事务的场景

技术选型决策树

Qt 与 SQLite3 的结合为单机应用和嵌入式系统提供了高效的本地数据存储方案。通过合理的配置和优化，可以获得接近专业数据库系统的性能。
