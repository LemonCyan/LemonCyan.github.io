SSH 常见故障排查

修改 SSH 配置后无法连接是常见问题，以下是按可能性排序的排查思路。

症状判断

  • **Permission denied (publickey)**：密钥不匹配或未正确部署
  • **Connection closed**：配置错误或被安全机制拦截

常见原因

1. 配置文件语法或逻辑错误

如果使用 root 用户登录，注意 PermitRootLogin 的设置：

  • PermitRootLogin no：完全禁止 root 登录（即使有密钥也不行）
  • PermitRootLogin prohibit-password：禁止密码，允许密钥（推荐）

2. 权限过宽触发 SSH 自我保护

SSH 对密钥文件权限要求严格，权限过宽（如 777）会导致拒绝读取：

3. Fail2ban 误封

测试配置过程中多次失败可能触发 Fail2ban 封禁，参考前文”Fail2ban 入侵防御”章节解封。

  • CPU 占用持续 100%
  • 其他服务响应变慢甚至超时

云安全组

在云服务商控制台确保以下端口不对公网开放：

  • 7890（HTTP 代理端口）
  • 7891（SOCKS5 代理端口）
  • 9090（管理接口端口）

经验教训

  1. 服务器上跑代理服务要格外小心：确认只监听本地地址
  2. 防火墙配置要精细化：不要放行大范围端口
  3. 定期检查服务器状态：监控流量、CPU、连接数
  4. 安全组是第一道防线：云服务器的安全组规则要配置好

六、安全加固优先级

根据实际情况，建议按以下优先级实施：

阶段 时间 行动项
立即 今天 启用 UFW 默认拒绝策略，只开放必要端口
短期 本周 安装并配置 Fail2ban 自动防护
中期 本月 部署 SSH 密钥认证，禁用密码登录
持续 常态 定期更新系统补丁，监控日志，考虑 VPN 访问 SSH

七、快速检查清单

在部署新服务器或审计现有服务器时，可参考以下清单：

  • UFW 已启用且默认策略为拒绝入站
  • 只开放了必要端口（SSH、Web 等）
  • SSH 使用非标准端口或配置了访问限制
  • Fail2ban 已安装并正常运行
  • SSH 已启用密钥认证
  • 已禁用 SSH 密码登录
  • 已禁用 root 直接登录
  • 云安全组规则与本地防火墙规则一致
  • 日志监控已配置（auth.log、ufw.log、fail2ban.log）
