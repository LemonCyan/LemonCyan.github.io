基本概念

鉴于本维基内容及代理领域中涉及的部分概念对许多人而言可能较为陌生，本文旨在以简明易懂的语言对这些概念加以阐释。

IP 地址

IP 地址是用于标识互联网或局域网中某一台设备的唯一地址，借此可实现对该网络或局域网的访问。换言之，IP 地址是网络中设备间传递信息所依赖的标识符；因其内含设备的地理位置信息，故能确保设备彼此可见并实现通信。

目前存在两种 IP 版本：IPv4 与 IPv6。

IPv4（IP 版本 4）

IPv4 地址由若干段组成，各段之间以“点”分隔，格式为 w.x.y.z，每段取值范围为 0 至 255。例如，一个典型的 IPv4 地址为：54.78.11.14。

IPv6（IP 版本 6）

IPv6 是新一代 IP 地址体系，基于十六进制（即基数为 16）表示法，由八段组成，各段之间以冒号 : 分隔。例如：2017:c569:54ed:f141:ec49:0567。

在该版本中，某些位置允许省略连续的零段。例如，对于地址 2017:c569:0000:0000:0000:0567，可将连续的零段压缩为双冒号 ::，最终简化为：2017:c569::0567。

HTTP

HTTP 是互联网的基础协议，亦为万维网信息交互的核心机制：通过超链接与文本等元素，将文档与其他资源相互关联，使用户可在浏览器中仅凭点击或轻触屏幕即可便捷访问。该协议采用 URL（统一资源定位符），即通常所说的“域名”进行寻址——相关定义见下文。

HTTP 协议通过“握手”机制完成信息的收发：客户端与服务器之间需交换若干请求与响应报文，方能启动并终止一次数据交互过程。试想，客户端（即浏览器）如同一位欲获取某网站信息的访客：首先须向该网站服务器发送请求；随后等待服务器回应。若服务器上存在所请求的信息，则会回传同意传输的响应；接着客户端再次发出请求，要求获取具体数据内容。此时服务器接收请求后，将所需数据拆分为若干小数据包，并依据握手协议依次发送至客户端。此协议下的客户端与服务器通信，通常经由 80 端口进行。

HTTPS

HTTPS 实质上是 HTTP 协议的安全增强版本。由于 HTTP 协议本身不加密传输内容，极易被劫持与滥用；一旦第三方（如互联网服务提供商）能够接入信息传输路径，便可观测全部通信内容。而 HTTPS 则对网站与用户间交换的信息实施加密处理——即便他人截获通信数据，亦无法解读其内容，因信息已被加密，唯持有解密密钥者方可查看。该协议借助 SSL 或 TLS 中的一种实现通信加密，相关内容后文详述。

HTTP Upgrade

Upgrade 是 HTTP 协议中的一个字段，客户端可借此请求服务器切换至所列协议之一（或多个），按优先级降序排列。借此，客户端可请求服务器升级至更高版本的 HTTP 协议，从而提升连接安全性。服务器可基于任意理由拒绝该请求，此时必须视同未收到 Upgrade 头部予以响应。若服务器支持该功能，则 HTTP 连接将被升级至更高版本。HTTP Upgrade 在实际操作中，于直接使用连接前，完整执行 HTTP/1.1 的 Upgrade 请求与响应流程。其建立单向通道的方式与 WebSocket 类似，可被多数反向代理及 CDN 透明转发，且无需处理 WebSocket 协议本身的诸多复杂问题。

域名

通俗而言，所谓“域名”，即您在浏览器地址栏中输入以访问任一网站的地址。域名通过 IP 地址与网站主机相联结；在互联网世界中，每一台服务器均以其唯一的 IP 地址标识；而域名则通过一种名为 DNS（Domain Name System，域名系统）的服务映射至对应 IP 地址。

SSL

SSL 协议利用公钥与私钥对信息进行加密：经其加密的数据，须由另一端使用相应密钥解密还原。该协议的安全性体现在信息源与目的地两端——加密与解密操作均在两端完成。采用此协议加密数据的网站，通常通过 HTTPS 协议与客户端通信。为使您的网站支持安全的 HTTPS 连接，须获取一张有效的 SSL 证书。在此协议下，服务器与客户端间交换的所有信息均被加密，第三方无法窥见其内容；即便数据被截获，亦因加密而无法直接读取，除非拥有本次安全会话所用的解密密钥——而此密钥极难获取，故该协议在实践中几乎不可攻破。

TLS

TLS 是基于 SSL 协议构建的安全协议，实为 SSL 的改进版本；尽管二者差异细微，核心机制大体一致。相较而言，TLS 采用更强的加密算法，在公钥生成与消息认证方面更为安全高效。更准确地说，TLS 是一套用于加密客户端与服务器间 Web 通信的密码学算法集合（如 uTLS、ALPN、allowInsecure）。重要提示：“用户身份识别依赖于 TLS 握手过程中的消息字段。”

通过对 TLS 流量的分析，审查系统可推测用户所访问的网站及其服务器信息（如 IP、地理位置等）。因此，我们必须审慎配置 TLS 算法，确保审查系统无法获取域名信息（尤其是 SNI 字段）。

  • uTLS：模拟浏览器自身指纹。“为避免单一指纹导致的误判风险，建议启用多种指纹并随机化选取。”
  • ALPN：预先协商客户端与服务器间的应用层协议，从而加速数据加载与 TLS 加密建立过程。
  • allowInsecure：禁止使用伪造证书。

Xray

Hiddify 面板依托 Xray 平台绕过审查系统；事实上，Hiddify 官方亦推荐使用 Xray。该平台于 2020 年发布，系知名 V2Ray 的改良与增强版本；凭借持续更新与缺陷修复，Xray 已逐步取代 V2Ray，成为主流平台。Xray 不仅完整兼容所有 V2Ray 协议，更新增了 VLESS 等现代协议。

VMess

VMess 曾为 V2Ray 与 Xray 子类中最早期的协议之一，其信息包编码与发送机制依赖系统时间戳；然而，因算法存在固有弱点，该协议易被审查系统识别，故强烈建议在可行情况下避免使用。

VLESS

VLESS 协议系 VMess 的优化版本：其已修正 VMess 的全部缺陷；加之加密方式不同、负载更轻，使得同一服务器可承载比 VMess 更多的并发用户。

Trojan

Trojan 协议是另一种高效可靠的绕过审查手段。自 2017 年推出以来，Trojan 一直与 VLESS 协同使用，构成突破审查系统的强大组合。

REALITY

REALITY 可替代传统 TLS，不仅可消除服务器端 TLS 指纹特征，仍保留前向保密等安全特性；同时可有效抵御证书链攻击，整体安全性超越常规 TLS。此外，无需购买域名或配置 TLS 服务器，即可指向任意网站，操作更为便捷；并能实现真正的 TLS 行为——即向中间人明确指定 SNI。

注：尽管 Hiddify 全面支持 Xray 所有协议，我们仍建议优先选用 REALITY、VLESS 与 Trojan 等更具鲁棒性的协议。
