Linux C 进程线程与进程间通信 (IPC) 核心知识点整合

在 Linux 操作系统中，进程与线程是任务调度的最核心单元，而进程间通信 (IPC, Inter-Process Communication) 是实现多任务协作的根本基石。

1. Linux 进程与线程的底层概念与内核管理

1.1 进程与线程的定义及本质区别

  • **进程 (Process)**：是操作系统分配资源的基本单位，是程序的一次动态执行过程。
  • **程序 (Program)**：是静态的指令集合，保存在磁盘物理介质上，无运行状态概念。
  • Linux 进程的四大要素：
    1. 独立的虚拟用户空间：用于存放用户态的代码和数据。
    2. **进程控制块 (PCB)**：内核中的 struct task_struct，是内核调度的基本单元。
    3. 独立的内核堆栈：用于保存该进程在内核态运行时的上下文与压栈信息。
    4. 一段可执行的代码程序：可由多个进程并发执行。
  • **线程 (Thread)：是 CPU 任务调度的基本单位。在 Linux 底层，线程被实现为轻量级进程 (LWP, Light Weight Process)**。
    • 核心区别：线程只拥有前三个要素，没有第四个要素（不拥有独立的虚拟用户空间，而是共享父进程的虚拟用户空间）。
    • **内核线程 (Kernel Thread)**：完全没有用户空间，只运行在内核态（如 ksoftirqd 等）。
    • **用户线程 (User Thread)**：共享父进程的虚拟用户空间（代码段、数据段、堆区等）。

1.2 Linux 进程的三大分类

  1. **交互式进程 (Interactive Process)**：
    • 特点：由 Shell 启动，频繁与用户发生交互，大部分时间处于等待用户操作的阻塞状态。
    • 调度要求：要求非常低的调度延迟（一般在 50~150 毫秒以内），当有输入时必须能够被瞬间激活。
    • 典型代表：Shell 控制台、文本编辑器（Vim）、图形化应用。
  2. **批处理进程 (Batch Process)**：
    • 特点：不需要用户交互，通常在后台运行，主要占用 CPU 算力或执行大量 I/O 读写。
    • 调度要求：响应速度要求不高，其运行常受到调度器的 CPU 时间片额度限制。
    • 典型代表：GCC 编译器、数据库搜索引擎、科学计算程序。
  3. **实时进程 (Real-time Process)**：
    • 特点：对系统调度有极度严格的时间限制要求。
    • 调度要求：绝对不能被低优先级的进程所阻塞，必须在指定的极短时间内（毫秒或微秒级）做出反应。
    • 典型代表：音视频流播放程序、机器人/工业控制程序。

1.3 静态优先级、动态优先级与实时调度

Linux 系统中，进程的调度次序与优先级紧密关联：

  • **静态优先级 (Static Priority)**：
    • 数值范围：100（高优先级）到 139（低优先级），数值越大优先级越低。默认新建进程的静态优先级为 120。
    • 特性：被称为“静态”是因为它不会随着时间自行改变，内核也不会修改它，用户只能通过系统调用 nice() 或 setpriority() 来手动微调（nice 值范围 -20 到 19，对应优先级 100 到 139）。
    • **Base Time Quantum (基础时间片)**：由静态优先级决定。
      • 当静态优先级 < 120 时：时间片 (ms) = (140 - 静态优先级) * 20
      • 当静态优先级 >= 120 时：时间片 (ms) = (140 - 静态优先级) * 5
  • **动态优先级 (Dynamic Priority / Counter)**：
    • 特性：内核调度器实际参看的指标。它本质上是当前时间片内**剩余的 CPU 运行时间量 (Counter)**。
    • 机制：只要进程占有 CPU 运行，其 Counter 值就随着时钟滴答递减。当递减为 0 时，内核会设置重新调度标志。
    • 计算公式：Dynamic Priority = max(100, min(Static Priority - bonus + 5, 139))（其中 bonus 值为 0~10，与进程的平均睡眠时间/交互性成正比，睡眠越久的进程 bonus 越大，动态优先级就越高，从而获得快速响应）。
  • **实时优先级 (Real-time Priority)**：
    • 实时优先级只对实时进程有效（系统权值基础常数为 1000）。Linux 在调度实时进程时，会将其实时优先级与 counter值相加得到最终权值。非实时进程的优先级远低于实时进程，因此实时任务总是能抢占普通任务优先执行。

1.4 Linux 进程创建与 task 克隆的底层剖析

在 Linux 内核中，不管是创建进程的 fork()、vfork()，还是创建线程的 clone()，其底层最终执行的都是同一个内核通用函数 **do_fork()**（新版本内核为 _do_fork()）。

do_fork() 的五个核心克隆步骤：

  1. **复制进程控制表 (PCB)**：调用 copy_process -> dup_task_struct()，复制父进程的 task_struct 结构体内容。
  2. 分配独立的内核堆栈：调用 alloc_thread_info_node()，在内核中申请一个页面大小（通常为 8KB）的物理空间，用于存放子进程的内核堆栈及 thread_info 结构。
  3. 初始化子进程内核堆栈：调用 setup_thread_stack()，将父进程的内核堆栈框架拷贝过来，并进行子进程专属初始化。
  4. 修改寄存器上下文（决定 fork 返回值）：
    • 子进程在复制得到的内核栈中，保存系统调用的寄存器结构 pt_regs（对应系统调用入口 SAVE_ALL 压入的寄存器状态）。
    • 将子进程寄存器结构中的 ax 寄存器值强行修改为 0：childregs->ax = 0;。这就是为什么子进程中 fork() 返回值为 0 的底层物理根源！
  5. 设定子进程的首次运行起点：
    • 将子进程 task 结构体中的调度栈顶 sp 设为 childregs 的物理地址：p->thread.sp = (unsigned long)childregs;
    • 将子进程首次被调度运行的指令指针 ip 设为内核函数 ret_from_fork 的地址：p->thread.ip = (unsigned long)ret_from_fork;
    • 执行起点：因此，当子进程第一次获得 CPU 被唤醒时，它不会从 main 开头执行，而是直接跳转到 ret_from_fork 处开始运行，随后通过 pt_regs 恢复寄存器并返回用户空间，继续执行 fork() 之后的下一行代码。

1.6 进程打开文件关联结构：fs_struct 与 files_struct

每个进程的 task_struct 内部，都有两个关键的结构体指针用于维护文件访问信息：

  1. **struct fs_struct *fs**：
    • 作用：保存了进程当前所处的物理工作目录（CWD）、文件系统根目录（Root）、以及新建文件时的缺省权限掩码 **umask**。
  2. **struct files_struct *files**：
    • 作用：存放该进程正在使用的所有文件描述符的集合。
    • 核心字段：
      • fd 数组指针：指向一个 struct file * 类型的数组。用户态传入的文件描述符（0, 1, 2 等整数）就是该数组的下标索引。
      • file 结构体：数组中每一项指向一个表示打开文件的 struct file。其中 f_mode 标识打开读写模式，f_pos 记录当前文件读写偏移位置（lseek 改变的就是它），f_inode 指向文件在虚拟文件系统（VFS）中的索引节点，f_ops 指向具体文件操作的函数指针集合。

1.7 进程虚拟内存与请求调页 (Demand-Paging) 机制

在 Linux 中，运行二进制程序创建进程时，内核绝不会立刻把整个程序的代码和数据全部一次性装入物理内存，因为这会产生极大的物理内存浪费。

  • 请求调页 (Demand-Paging) 机制：
    1. 当进程分配虚拟内存时，内核只在其 mm_struct 的虚拟内存区域列表（VMA）中，新建一个 struct vm_area_struct 结构体来描述这块新分配的虚拟地址范围，而不分配实际物理页面。
    2. 此时，对应的虚拟页表项（PTE）被标识为“有效但物理数据不存在”。
    3. 当进程在运行中第一次尝试读写这个虚拟地址时，CPU 硬件的 MMU（内存管理单元）会因为查不到合法的页表物理映射，自动触发一个 **Page Fault (缺页异常/缺页中断)**。
    4. 内核的缺页异常处理程序被激活，它会在 VMA 列表中查找该引起故障的虚拟地址是否合法：
      • 如果合法：内核会自动在物理内存中申请一页空闲物理内存帧，建立页表映射项（PTE），并将磁盘或交换分区上的代码数据读取到该页中。随后，恢复被中断的 CPU 指令重新执行，进程即可正常读写。
      • 如果不合法（比如访问了没有权限的受保护区域，或越过了栈底）：内核会向当前进程发送一个 SIGSEGV 信号，也就是大家开发中经常遇到的 Segmentation fault (段错误) 并直接强退程序。

2. 进程间通信 (IPC) 概述

2.1 为什么需要进程间通信？

  1. 模块化与低耦合：利用进程相对独立的物理隔离特点，方便将复杂系统拆分为多个组件进行开发、调试，降低系统级耦合。
  2. 临界资源保护：防止多个进程同时修改共享的数据（临界资源），需要同步和互斥机制。
  3. 内核与用户空间交互：驱动程序或系统内核需要向用户态进程通知硬件状态或事件。

2.2 主流 IPC 通信方式对比

IPC 方式 亲缘限制 双工模式 物理介质 核心特点
无名管道 (Pipe) 必须有亲缘关系 半双工 (单向) 内存缓冲区 简单高效，生命周期随进程结束而销毁
有名管道 (FIFO) 无限制 半双工 (单向) 内存缓冲区 + 磁盘节点 拥有文件路径名，在文件系统中可见
信号 (Signal) 无限制 异步单向通知 内核软中断 唯一的异步通信方式，只能传递有限的信号值
消息队列 (MSG) 无限制 全双工 内核链表 自动分包，支持按类型选择性接收，避免临界区冲突
共享内存 (SHM) 无限制 全双工 物理内存映射 效率最高的通信方式，但需要额外的同步/互斥锁配合
信号量集 (SEM) 无限制 N/A 内核计数器 配合共享内存使用，实现多进程间的同步与互斥

3. 管道通信：无名管道与有名管道

3.1 无名管道 (Anonymous Pipe)

核心机制

无名管道是基于文件描述符的通信通道，数据只存在于内存中，不写入磁盘。它是一个经典的半双工通道：fd[0] 固定用于读管道，fd[1] 固定用于写管道。

API 使用

  • 参数：包含两个整数的数组。成功后，fd[0] 存储读端描述符，fd[1] 存储写端描述符。
  • 返回值：成功返回 0，失败返回 -1。

管道读写核心特性

  1. 读阻塞：当管道中没有数据时，read 操作会阻塞等待，直到有数据写入或写端全部关闭（返回 0，即 EOF）。
  2. 写阻塞：管道是有容量限制的（Linux 默认一般为 64KB）。如果读端不取走数据，写端将管道写满后，write 操作会一直阻塞。
  3. SIGPIPE 信号（Broken Pipe）：如果在读端全部关闭的情况下，尝试向管道写入数据，内核会向写进程发送 SIGPIPE 信号，默认会导致进程异常终止。
  4. 原子性：向管道写入小于 PIPE_BUF（Linux 下通常为 4096 字节）的数据能保证操作的原子性，防止多进程并发写入时数据交错。

实用场景：父子进程通信与标准 I/O 重定向

3.2 有名管道 (FIFO / Named Pipe)

核心机制

有名管道解决了无名管道只能用于亲缘进程的限制。它在文件系统中创建一个特有的管道文件节点（不占用实际磁盘物理扇区），使任意不相关的进程都可以通过普通的 open/read/write 文件操作读写该管道。

API 使用

  • 参数：
    • filename：管道文件路径。
    • mode：八进制权限标志（如 0666）。
  • 返回值：成功返回 0，失败返回 -1。若文件已存在，会触发 EEXIST 错误（通常在调用前用 access(filename, F_OK) 检查是否存在）。

FIFO 的打开行为与读写限制

  1. 打开同步阻塞：
    • 如果只读方式（O_RDONLY）打开有名管道，会阻塞直到另一个进程以只写方式（O_WRONLY）打开它。
    • 反之，只写方式打开也会阻塞直到有进程以只读方式打开。
    • 若要实现非阻塞，可以在 open 时传入 O_NONBLOCK 标志。
  2. 不支持文件定位：有名管道属于先进先出队列，绝不支持 lseek() 随机定位读写。
  3. 内容瞬时性：当所有持有有名管道文件描述符的进程关闭后，管道中的数据被立刻清空。

4. 信号机制 (Signal)

4.1 信号的本质

信号是软件层次上对硬件中断机制的一种模拟，属于异步通信。内核随时可以因为某些事件（如除零、段错误、定时器到期、用户按下 Ctrl-C）向进程投递信号，进程会暂停当前执行流转去处理信号。

4.2 常见系统信号一览表

信号名 信号值 触发原因 默认操作
SIGHUP 1 终端控制进程结束（挂断终端） 终止进程
SIGINT 2 用户按 Ctrl-C 终止进程
SIGQUIT 3 用户按 Ctrl-\ 终止进程 + 产生 Core
SIGILL 4 进程企图执行非法指令 终止进程
SIGFPE 8 致命算术错误（如除以 0） 终止进程
SIGKILL 9 强制杀死进程指令 立即终止 (不可忽略/捕捉)
SIGALRM 14 alarm() 定时器到期 终止进程
SIGTERM 15 优雅终止程序指令（kill 默认信号） 终止进程
SIGCHLD 17 子进程暂停或退出 忽略该信号
SIGSTOP 19 暂停进程执行 暂停进程 (不可忽略/捕捉)

4.3 信号发送与捕捉 API

1) 发送信号：kill() 与 raise()

2) 定时器与阻塞挂起：alarm() 与 pause()

  • alarm 注意事项：
    • 一个进程只能同时存在一个活动闹钟。重复调用 alarm() 会取消上一个闹钟，并返回上一个闹钟的剩余时间。
    • 不能与 sleep() 混用，因为许多系统的 sleep() 在底层也是通过 alarm 和 pause 组合实现的。

3) 注册信号处理函数：signal()

  • handler 参数可选值：
    • SIG_IGN：忽略此信号。
    • SIG_DFL：执行系统默认行为。
    • 自定义函数指针（回调函数）：原型必须为 void handler(int sig_num)。

5. 共享内存 (Shared Memory)

5.1 核心机制

共享内存直接将同一块物理内存映射到不同进程的虚拟地址空间中。进程读写该内存块就像读写本进程的局部变量一样，不需要经过内核数据的拷贝，因而成为 Linux 下最高效的 IPC 通信手段。
由于缺乏内核层的读写互斥保护，多进程同时读写共享内存时，必须配合信号量或互斥锁来避免数据损坏。

5.2 System V 共享内存 API 族

1) 获取唯一的 IPC 键值：ftok()

通过指定的文件路径名和工程 ID 组合计算出一个唯一的 key_t 值，用于作为 IPC 对象（共享内存、消息队列等）的初始化标识符（类似于域名的作用）。

2) 申请/创建共享内存：shmget()

  • 参数：
    • key：ftok 返回的标识符。
    • size：共享内存区字节数（一般是系统页大小 4096 的倍数）。
    • shmflg：控制权限（如 0666），可按位或 IPC_CREAT（不存在则创建） and IPC_EXCL（已存在则报错）。
  • 返回值：成功返回共享内存段的 ID (shmid)，失败返回 -1。

3) 映射共享内存：shmat()

  • 参数：
    • shmaddr：一般传 NULL，由内核自动选择合适的虚拟地址进行映射。
    • shmflg：传 SHM_RDONLY 表示只读，传 0 表示可读写。
  • 返回值：成功返回映射到本进程的虚拟首地址，失败返回 (void *)-1。

4) 解除映射：shmdt()

  • 作用：断开当前进程与共享内存的连接。解除后，该虚拟地址不再可用，但内核中的共享内存块不会被删除。进程结束时系统会自动执行解除。

5) 控制/删除共享内存：shmctl()

  • **参数 cmd**：
    • IPC_STAT：获取共享内存属性。
    • IPC_RMID：标记删除该共享内存段。
  • 注意：调用 IPC_RMID 时，只有当所有映射了该共享内存的进程都解除映射后（即 nattch 归 0），该共享内存才会在内核中被真正物理释放。

5.3 终端诊断指令

  • **ipcs -m**：显示当前系统中所有共享内存段的 ID、大小、连接进程数等信息。
  • **ipcrm -m shmid**：手动删除指定的共享内存段。

6. 消息队列 (Message Queue)

6.1 核心机制

消息队列是存储在内核中的一个消息链表，通过消息队列 ID 进行唯一标识。

  • 与管道对比：
    1. 消息边界清晰：消息队列以独立的“消息包”为单位投递，自带大小信息，不会出现管道那种读写流字节拼接的粘包问题。
    2. 类型选择性接收：发送的消息必须指定类型（long mtype）。接收进程可以选择性地只接收特定类型的消息，而忽略其他类型。这极大地简化了多对一或多对多通信的设计。
    3. 自带同步机制：自带读写锁保护，不需要程序员考虑底层的临界区数据交错污染。

6.2 System V 消息队列 API 族

1) 创建或打开队列：msgget()

2) 发送消息：msgsnd()

  • **参数 msgp**：指向必须以 long mtype 开头的用户自定义结构体：
  • **参数 msgflg**：若设为 0，写满时阻塞等待；若设为 IPC_NOWAIT，写满时立刻报错返回，errno 设为 EAGAIN。

3) 接收消息：msgrcv()

  • **参数 msgtyp**：
    • 0：获取队列中第一个可用的消息。
    • > 0：只获取类型等于 msgtyp 的第一个消息。
    • < 0：获取类型小于或等于 abs(msgtyp) 中最小类型的第一个消息。

4) 控制/删除队列：msgctl()

  • 使用 IPC_RMID 作为 cmd 可以立刻将消息队列在内核中销毁。

5) 诊断命令：ipcs -q 与 ipcrm -q msqid。

7. 信号量集 (Semaphore)

7.1 核心机制

System V 信号量本质上是一个内核计数器，用于控制多个进程对临界区公共资源的访问。它是多线程中 sem_t 机制在多进程层面的扩展实现。
主要操作是 P 操作（申请资源，值减 1，值为 0 时阻塞） 和 V 操作（释放资源，值加 1，唤醒阻塞进程）。

7.2 信号量 API 族

1) 创建/打开信号量集：semget()

  • nsems：该信号量集里包含的信号量个数。

⚠️ 死锁克星：SEM_UNDO 标志

当一个进程执行 P 操作锁定了资源后，如果因为非正常崩溃、意外退出等原因未能执行 V 操作释放锁，将导致其他所有进程永久死锁。

  • 解决方案：在 semop 结构体的 sem_flg 中带上 SEM_UNDO。内核会记录当前进程的动作，一旦该进程意外终止，内核会自动撤销其所有未完成 of P 操作（相当于自动执行了 V 操作），从而完美避免死锁。
