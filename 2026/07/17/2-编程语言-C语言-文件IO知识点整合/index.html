Linux C 系统级文件 I/O 核心知识点整合

在 Linux 操作系统中，系统一切皆文件。文件是以字符序列构成的信息载体。根据这一核心理念，Linux 将所有磁盘上的普通文件、目录、硬件设备、网络套接字等均当做“文件”来处理，可以使用统一的系统调用接口进行交互，从而极大地简化了系统设计。

1. Linux 文件系统与文件描述符基础

1.1 Linux 的文件分类与类型字符

在 Linux 系统下，任何设备或数据都以文件形式呈现。文件主要分为以下 7 种类型（对应 ls -l 列表输出的第一个标识字符）：

标识字符 文件类型 英文名称 物理特性与用途描述
- 普通文件 Regular File 存储普通的文本数据、源码、可执行二进制程序或多媒体二进制流。
d 目录文件 Directory 专门存放文件名与 inode 节点映射关系的特殊文件，相当于“文件夹”。
l 符号链接 Symbolic Link 类似于 Windows 的快捷方式，其物理内容仅存放另一个目标文件的绝对/相对路径。
p 命名管道 FIFO/Pipe 专用于本地进程间半双工通信的数据通道（数据只保存在内核缓冲区中，不写盘）。
s 套接字 Socket 专用于跨主机或本地进程间双向网络通信的虚拟接口。
c 字符设备 Char Device 代表键盘、鼠标、串口终端等按字节流顺序读写、不支持随机寻址定位的硬件接口。
b 块设备 Block Device 代表硬盘、U 盘等按扇区块大小读写、支持随机寻址定位与缓存机制的物理存储硬件。

1.2 Linux 设备驱动的三大类及其区别

在 Linux 操作系统中，所有外设和硬件资源都要通过驱动程序进行统一管理。根据设备的物理特性与内核接口的不同，Linux 系统中的设备驱动被划分为以下三大类：

  1. 字符设备 (Character Device)
    • 特点：数据传输是以字节流形式按顺序串行依次读取，不支持随机寻址定位（例如不能进行 lseek() 寻道）。
    • 文件映射：在 Linux 文件系统中被映射为特殊文件节点（首字符为 c），在 /dev/ 目录下占据结点空间。
    • 操作接口：用户空间可以使用与普通文件相同的系统调用接口（open(), read(), write(), close()）对其进行基础读写。
    • 常见设备：键盘、鼠标、串口终端（UART）、蜂鸣器、LCD 显示器、触摸屏等。
  2. 块设备 (Block Device)
    • 特点：数据的访问可以按任意随机顺序进行，但每次读写物理操作都必须以固定的数据块（如 512 字节或 4KB）为单位，且内核在底层为其维护了专用的高速缓冲区和 I/O 调度队列以优化物理读写频次。
    • 文件映射：在文件系统中映射为块设备特殊文件（首字符为 b）。
    • 操作接口：内核通过文件系统间接提供给用户空间，虽然可以用 open() 等文件接口访问，但更多是通过挂载（mount）到目录树下后以常规文件系统进行读写。
    • 常见设备：机械硬盘、SSD 固态硬盘、U 盘、SD 卡等存储介质。
  3. 网络设备 (Network Device)
    • 特点：专门针对网络数据包的发送和接收而设计，不通过文件系统中的物理节点进行表示和挂载。
    • 操作接口：其与内核的通信方式完全不同于字符和块设备。网络设备没有对应的文件描述符，用户空间必须通过专门的套接字 (Socket) 接口及 socket/bind/sendto/recvfrom 系统调用与网络设备驱动进行包收发，内核在底层使用 struct sk_buff (套接字缓冲区) 传递数据。
    • 常见设备：以太网网卡（WIFI、网线网卡）、环回网卡（loopback）等。

1.3 文件描述符 (File Descriptor, FD)

内核使用文件描述符来区分和引用特定进程打开的文件。

  • 物理本质：文件描述符是一个非负整数，实际上是进程控制块（PCB）中打开文件描述符表的一个数组索引值，该索引指向内核中进程打开文件记录表。
  • 参数传递：每当进程通过系统调用打开或创建文件时，内核会返回该文件的描述符。随后的所有读写操作都必须将该描述符作为参数传入，由内核映射到真实的物理文件结构上。

默认打开的三个标准流

当任何一个 Linux 进程启动时，系统都会默认自动为其打开 3 个标准流，分别占用前三个文件描述符：

  • **0：标准输入 (stdin)**，对应的 POSIX 宏定义为 **STDIN_FILENO**。
  • **1：标准输出 (stdout)**，对应的 POSIX 宏定义为 **STDOUT_FILENO**。
  • **2：标准错误 (stderr)**，对应的 POSIX 宏定义为 **STDERR_FILENO**。

1.3 不带缓冲的 I/O 特征

系统级文件 I/O 函数（如 open, read, write, lseek, close）通常被称为**不带缓冲的 I/O (Unbuffered I/O)**。

  • 物理特性：不带缓冲是指每一次 read 或 write 操作都会直接触发一个底层的系统调用，由内核直接读写磁盘/设备，而不经过用户空间的用户级缓冲区（如 C 库中 printf / fread 维护的缓冲区）。
  • 标准定义：这些函数不属于 ANSI C 标准库，而是由 POSIX 标准（IEEE Std 1003.1）定义，具有极高的平台兼容性，能方便地移植到任何 POSIX 平台上，也是实现多路 I/O（select/poll/epoll）、套接字编程的唯一基石。

2. 文件创建与控制：open 与 close 系统调用

2.1 open() 打开/创建文件

1) 头文件与函数原型

  • 参数：
    • pathname：指向欲打开的文件路径名字符串（绝对或相对路径）。
    • flags：控制文件打开方式的位掩码标志。
    • mode：仅当使用 O_CREAT 标志创建新文件时才需要传入，用于指定该新文件的物理权限位。
  • 返回值：成功时返回分配的最小可用非负整数文件描述符，失败返回 -1，并设置相应的 errno 错误码。

2) flags 参数常用取值（按位或 | 运算组合）

  • 互斥的基础打开模式（三者必须且只能选其一）：
    • **O_RDONLY**：以只读方式打开文件。
    • **O_WRONLY**：以只写方式打开文件。
    • **O_RDWR**：以可读写方式打开文件。
  • 创意与状态辅助标志：
    • **O_CREAT**：若文件不存在，则自动创建该文件，此时必须指定第三个参数 mode。
    • **O_EXCL**：常与 O_CREAT 联用，用于执行独占性检查。如果文件已存在，则 open 报错失败（errno 设为 EEXIST）。这能防止在多进程环境下出现竞争创建文件的安全漏洞。
    • **O_TRUNC**：如果文件已存在，且以可写模式（只写/读写）成功打开，则会将文件长度清空为 0，舍弃原有的全部内容（相当于覆盖写）。
    • **O_APPEND**：以追加模式打开文件。每次执行 write 时，读写位置都会自动定位到文件的末尾。
  • 不常用的特殊控制标志：
    • **O_NOCTTY**：如果打开的文件代表终端设备，不会将该终端作为此进程的控制终端。
    • O_NONBLOCK / O_NDELAY：以非阻塞（无等待）方式打开。对于管道、FIFO、字符设备或套接字，若读写无数据会立即返回而不阻塞进程。
    • **O_SYNC**：以同步 write-through 方式打开，每次 write 都必须等数据物理写入磁盘后函数才返回。
    • **O_NOFOLLOW**：如果 pathname 指向的是一个符号链接，则 open 失败。
    • **O_DIRECTORY**：如果 pathname 指向的不是目录，则 open 失败。

3) 新建文件权限：mode 权限位与 umask

第三个参数 mode 控制新建文件的初始权限，但该权限并不是最终权限，而是受到系统安全进程掩码 umask 的限制。

  • 计算公式： 例如：若传入 mode = 0777 (即 S_IRWXU | S_IRWXG | S_IRWXO)，假设系统的 umask 值为 0022，则最终创建的文件的真实权限为：0777 & (~0022) = 0755（即所有者可读写执行，组用户及其他用户可读执行）。
  • 标准权限常量：
    • S_IRWXU (00700)：所有者读写执行。
    • S_IRUSR / S_IREAD (00400)：所有者可读。
    • S_IWUSR / S_IWRITE (00200)：所有者可写。
    • S_IXUSR / S_IEXEC (00100)：所有者可执行。
    • S_IRWXG (00070)：同组用户读写执行。
    • S_IRWXO (00007)：其他用户读写执行。

2.2 close() 关闭文件

1) 头文件与函数原型

  • 作用：断开文件描述符 fd 与内核对应文件结构的映射，释放该描述符以供后续 open 使用。
  • 磁盘物理写入：在关闭文件前，操作系统会将由于延迟写而暂存在缓冲区的数据物理写回磁盘，并释放相关的内核内存资源。
  • 返回值：成功返回 0，失败返回 -1。
  • 规范提示：虽然进程退出时内核会自动关闭所有打开的描述符，但在编写长期运行的服务程序时，必须主动关闭不用的描述符，否则会导致描述符泄露（达到进程或系统描述符上限后将无法再打开任何新文件/套接字）。

3. 文件读写操作：read 与 write 系统调用

3.1 read() 读取数据

1) 头文件与函数原型

  • 参数：
    • fd：已打开文件的文件描述符。
    • buf：指向存储读出数据的用户空间缓冲区指针。
    • count：指定期望读取的最大字节数。
  • 返回值：其类型为有符号整型 ssize_t。
    • **> 0**：返回实际读取到的字节数。该值可能小于 count（例如读到文件尾、管道中数据不足、或者被信号打断）。
    • **0：代表读取到了文件尾 (EOF)**，或无可读取的数据。
    • **-1**：代表读取出错，错误原因记录在 errno 中。

2) 经典读取错误码

  • **EINTR**：当 read 在阻塞等待输入时，进程接收到了一个系统信号，导致读取操作被提前打断，返回 -1。
  • **EAGAIN**：在使用非阻塞 I/O (O_NONBLOCK) 时，若当前输入缓冲区没有任何数据可读，read 不会阻塞进程，而是立刻返回 -1，并将错误码设为 EAGAIN，提示程序稍后重试。

3.2 write() 写入数据

1) 头文件与函数原型

  • 参数：
    • buf：指向存放待写入数据的用户空间缓冲区指针（由于只读，加了 const）。
    • count：指定期望写入的字节数。
  • 返回值：
    • 成功时返回实际写入文件的字节数。若返回值与 count 不等（往往小于 count），多是由于磁盘已满、文件大小超限等原因。
    • 失败时返回 -1，并设置相应的 errno 错误码。

4. 文件读写位置定位：lseek 系统调用

4.1 lseek() 定位读写偏移量

每一个已打开的文件在内核中都会记录一个当前的读写偏移量（CFO, Current File Offset）。默认打开时指向文件开头（若以 O_APPEND 打开则指向文件尾）。读写操作会使偏移量自动增加。lseek 可手动定位和调整此偏移量。

1) 头文件与函数原型

  • 参数：
    • offset：以字节为单位的偏移量。正数代表向后移，负数代表向前移。
    • whence：偏移量的相对参考基准点。
  • 返回值：成功时返回定位后的新读写偏移量位置（相对于文件开头的字节偏移数），失败返回 -1。

2) whence 相对基准点取值

  1. **SEEK_SET**：将偏移量直接设置为相对于文件开头的 offset 字节。
  2. **SEEK_CUR**：将偏移量设置为当前 CFO 位置加上 offset。
  3. **SEEK_END**：将偏移量设置为文件物理结尾长度加上 offset。

[!IMPORTANT]
读写偏移量 (CFO) 的内核控制规则：

  1. 独立偏移规则：每执行一次 open() 或 fopen() 开启文件，内核就会为其分配和记录一个独立的当前文件位置偏移量。因此，如果多个线程/进程同时通过独立的 open 调用去操作同一个物理文件，它们拥有各自独立、互不干扰的读写位置。
  2. 共享读写偏移：针对同一次 open() 返回的同一个文件描述符（FD），读和写操作共同影响并共享这同一个 CFO 偏移量。例如：读取 100 字节后，此时 CFO 向后移动 100 字节；紧接着在该描述符上进行 write 写入，写入的数据将位于该 100 字节之后，而不会从头覆盖。
  3. 管道/套接字不可定位：所有的管道类文件（命名管道 FIFO、匿名管道 Pipe）和网络通信套接字（Socket）属于流式接口，绝对不支持设置文件偏移量。如果对这类文件描述符调用 lseek() 或 fseek()，函数会直接返回错误 -1，且全局 errno 会被设为 **ESPIPE**。
  4. 文件空洞的产生：lseek 允许将偏移量定位到超出当前文件物理结尾的后方（例如 lseek(fd, 1000, SEEK_END)）并紧接着执行 write 写入操作。这会在物理磁盘上产生一个**文件空洞 (File Hole)**。文件空洞部分的字节在逻辑上被解释为 \0 填充，但不占用实际的物理磁盘扇区存储，这对于快速初始化巨型虚拟文件（如虚拟机磁盘镜像）非常重要。

6. 标准 C 库缓冲文件 I/O 函数 (C Standard Library)

系统调用（如 open/read/write）是不带缓冲区的 I/O。而标准 C 库提供的文件操作接口是带用户空间缓冲区的高级 I/O 封装，使用 FILE * 文件流指针来管理文件，相比系统调用更具跨平台可移植性，且在频繁小字节读写时效率更高。

6.1 文件流的打开与关闭

1) fopen()：打开文件流

  • 函数原型：FILE *fopen(const char *path, const char *mode);
  • 参数说明：
    • path：欲打开的文件的路径及文件名。
    • mode：流形态（打开模式），取值如下：
      模式 对应系统调用 flag 组合 语义说明
      r O_RDONLY 打开只读文件，文件必须存在。流定位在文件开头。
      r+ O_RDWR 打开可读写文件，文件必须存在。流定位在文件开头。
      w O_WRONLY | O_CREAT | O_TRUNC 打开只写文件。若存在则长度清 0 覆盖，若不存在则新建。
      w+ O_RDWR | O_CREAT | O_TRUNC 打开可读写文件。若存在则长度清 0 覆盖，若不存在则新建。
      a O_WRONLY | O_CREAT | O_APPEND 附加只写模式。若不存在则新建；若存在，写入的数据自动追加到文件尾。
      a+ O_RDWR | O_CREAT | O_APPEND 附加可读写模式。若不存在则新建；若存在，写入的数据自动追加到文件尾。
  • 二进制模式 b：上述模式后可追加字符 b（如 rb, wb+），明确告诉库以二进制形式打开。在 Linux/POSIX 系统中会忽略该字符（二进制与文本文件在 Linux 下物理结构一致），但在 Windows 系统下有重要作用（用于防止 \n 与 \r\n 的自动转换）。
  • 新建文件权限：由 fopen 新建的文件默认权限为 0666（可读写，即 S_IRUSR|S_IWUSR|S_IRGRP|S_IWGRP|S_IROTH|S_IWOTH），该权限会参考系统的 umask 值进行掩码处理。
  • 返回值：成功返回指向 FILE 结构体的流指针，失败返回 NULL 并设置 errno。

2) fclose()：关闭文件流

  • 函数原型：int fclose(FILE *stream);
  • 说明：关闭先前由 fopen 打开的文件流。在关闭前，会自动将用户缓冲区中残留的数据全部强行刷入物理文件（即隐式调用 fflush），并释放系统文件资源。
  • 返回值：成功返回 0，失败返回 EOF（通常为 -1），错误代码存在 errno 中。

6.2 各种类型数据的读写接口

1) 格式化 I/O 函数

  • **fprintf**：向指定文件流格式化写入数据。
    • int fprintf(FILE *stream, const char *format, ...);
  • **fscanf**：从指定文件流格式化输入（读取）数据。
    • int fscanf(FILE *stream, const char *format, ...);

2) 字符级 I/O 函数

  • **fgetc / getc**：由文件中读取一个字符。
    • int fgetc(FILE *stream);
    • int getc(FILE *stream); // 注：getc() 与 fgetc() 作用完全相同，但 getc 在标准库中通常被实现为宏定义，而非真正的函数调用，执行效率略高。
    • 返回值：成功返回读取到的字符（强转为 int ），若读到文件尾或发生错误则返回 **EOF**。
  • **fputc / putc**：将一指定字符写入文件流。
    • int fputc(int c, FILE *stream);
    • int putc(int c, FILE *stream); // 注：putc() 通常也是宏定义。
    • 返回值：成功返回写入字符，失败返回 EOF。
  • getchar / putchar 宏：
    • getchar() 等价于调用 getc(stdin)（从标准输入读）。
    • putchar(c) 等价于调用 putc(c, stdout)（输出到标准输出）。

3) 行级（字符串）I/O 函数

  • **fgets**：从文件中读取一行字符串。
    • char *fgets(char *s, int size, FILE *stream);
    • 机制：读取字符存入 s 指向空间，直到遇到换行符 \n、文件尾 EOF 或读取了 size - 1 个字符为止。最后自动在末尾追加 \0 结束符（换行符也会被存入数组中）。
    • 返回值：成功返回首地址 s，发生错误或已到文件尾且未读到数据时返回 NULL。
  • **fputs**：将指定的字符串写入文件。
    • int fputs(const char *s, FILE *stream);
    • 返回值：成功返回写入字节数，失败返回 EOF。
  • ⚠️ 废弃安全警示：gets() 绝对禁止在工程中使用：
    • char *gets(char *s); 试图从标准输入读取一行，但它不接收长度限制参数，容易遭遇输入缓冲区溢出导致栈破坏，触发安全漏洞。必须使用 fgets(s, sizeof(s), stdin) 彻底替代！
  • **getline (Linux C 动态分配读取行 - GNU 扩展)**：
    • 启用宏与头文件：#define _GNU_SOURCE（必须写在最顶端），#include <stdio.h>。
    • 函数原型：ssize_t getline(char **lineptr, size_t *n, FILE *stream);
    • 运行特征（自动动态伸缩）：
      • 如果 *lineptr 为 NULL 且 *n 为 0，getline() 会在内部自动调用 malloc() 申请合适大小的内存，并将首地址写入 *lineptr，缓冲区容量大小写入 *n。调用者在使用完毕后，必须手动调用 free(*lineptr) 释放内存以避免内存泄漏！
      • 如果传入的缓冲区大小 *n 不够存放整行数据，getline() 会自动在底层执行 realloc() 扩容，并更新 *lineptr 与 *n。
    • 返回值：成功时返回实际读入的字符个数（包括末尾的换行符 \n），但不计算字符串终止符 \0。读取发生错误或读到文件尾（EOF）时返回 -1。

4) 数据块（二进制）I/O 函数

对于结构体、数组等复合二进制块的读写，使用数据块接口：

  • **fread()**：从流中读取数据块。
    • size_t fread(void *ptr, size_t size, size_t nmemb, FILE *stream);
  • **fwrite()**：将数据块写入流。
    • size_t fwrite(const void *ptr, size_t size, size_t nmemb, FILE *stream);
  • 参数含义：
    • ptr：存放读写数据的缓冲区内存首地址。
    • size：每个数据块的字节大小。
    • nmemb：欲读写的数据块数量。
    • stream：目标文件流指针。
  • 返回值：返回**实际成功读/写的完整数据块数量 (nmemb)**。
    • ⚠️ 注意：如果 fread 返回的值小于请求的 nmemb，说明发生了读取不足。可能是到达了文件尾（EOF），也可能是发生了文件错误。必须通过 feof(stream) 或 ferror(stream) 进一步判定具体状态。

6.3 文件流定位与状态控制

1) fseek()：移动流读写位置

  • 原型：int fseek(FILE *stream, long offset, int whence);
  • 功能：功能类似于系统调用 lseek。参数 whence 可选 SEEK_SET（开头）、SEEK_CUR（当前位置）、SEEK_END（文件尾）。
  • 对比差异：
    • lseek 成功时会直接返回定位后的绝对偏移地址；
    • **fseek 成功时仅返回 0，失败返回 -1**。因此如果想获取定位后的当前位置，必须额外调用 ftell()。

2) ftell()：获取流当前绝对位置

  • 原型：long ftell(FILE *stream);
  • 返回值：成功返回当前流相对于文件开头的字节偏移量，失败返回 -1L。

3) rewind()：重设位置为开头

  • 原型：void rewind(FILE *stream);
  • 说明：将流重置到开头，相当于调用 (void)fseek(stream, 0, SEEK_SET)。

4) feof()：检查流是否到达文件尾

  • 原型：int feof(FILE *stream);
  • 说明：如果先前有读操作越过文件尾，则返回非零值（真），否则返回 0。

5) clearerr()：清除错误与 EOF 标志

  • 原型：void clearerr(FILE *stream);
  • 说明：复位清除流的错误状态旗标以及 EOF 结束标志。

6) fflush()：强制刷新输出缓冲区

  • 原型：int fflush(FILE *stream);
  • 说明：强迫将文件流的用户缓冲区中尚未物理写盘的数据写回物理文件。若参数为 NULL，则刷写所有打开的文件流。

6.4 系统调用与标准库接口的互转

1) fileno()：从文件流获取文件描述符 fd

  • 原型：int fileno(FILE *stream);
  • 返回值：返回该 FILE * 文件流底层正在关联使用的物理**文件描述符 (file descriptor)**。

2) fdopen()：将文件描述符 fd 转换为文件流指针

  • 原型：FILE *fdopen(int fildes, const char *mode);
  • 说明：将系统调用获得的 fd 转换为标准 C 库能操作的 FILE * 流指针。参数 mode 的流形态必须和原先 fd 打开时的读写模式（如 O_RDONLY / O_RDWR）相契合。

6.5 高级：设置文件流缓冲区策略

在某些极高性能场景或无缓冲的日志记录场景下，我们可能需要定制缓冲区大小：

  • **setbuf()**：设置缓冲区。
    • void setbuf(FILE *stream, char *buf);
    • 如果 buf 指向用户申请的长度为 BUFSIZ 的空间，则开启完全缓冲；如果 buf 为 NULL，则关闭缓冲区变为无缓冲。
  • **setvbuf()**：高级控制缓冲区。
    • int setvbuf(FILE *stream, char *buf, int mode, size_t size);
    • 参数 mode 种类：
      • **_IONBF (No Buffer)**：无缓冲区，任何读写立刻调用底层系统调用写盘（例如标准错误流 stderr）。
      • **_IOLBF (Line Buffer)**：行缓冲区，遇到换行符 \n 时或缓冲区满时才写盘（例如控制台终端交互流 stdout/stdin）。
      • **_IOFBF (Full Buffer)**：完全缓冲区，只有当缓冲区被装满或显式刷新时才物理写盘（例如普通的磁盘文件读写）。

7. Linux 文件属性与元数据获取 (stat 系列 API)

在 Linux 中，文件的元数据（大小、所有者、权限、修改时间等）保存在磁盘的 inode 节点中。应用程序可以通过 stat 函数簇获取这些属性。

7.1 stat()、fstat() 与 lstat()

1) 头文件与函数原型

  • 核心差异：
    • stat 会追踪符号链接，获取的是链接指向的真实目标文件的属性。
    • lstat 不会追踪符号链接，获取的是符号链接文件本身的元属性。
    • fstat 通过已打开的文件描述符 fd 直接获取其对应的文件属性。

3) 使用 st_mode 宏判断文件类型

st_mode 是一个 16 位的位图，高 4 位用于表示文件类型。POSIX 定义了专用宏进行快速判定：

  • **S_ISREG(m)**：是否为普通文件。
  • **S_ISDIR(m)**：是否为目录文件。
  • **S_ISLNK(m)**：是否为符号链接。
  • **S_ISFIFO(m)**：是否为管道文件。
  • **S_ISSOCK(m)**：是否为套接字文件。
  • **S_ISCHR(m)**：是否为字符设备。
  • **S_ISBLK(m)**：是否为块设备。

9.1 权限与存在性判定：access()

用于检查当前进程是否具有访问某个文件的特定权限，或者确认文件是否存在。

  • 参数 mode 的取值：
    • **F_OK**：检查文件是否存在。
    • **R_OK**：检查是否可读。
    • **W_OK**：检查是否可写。
    • **X_OK**：检查是否可执行。
  • 返回值：满足检查条件返回 0，否则（权限不足或文件不存在）返回 -1。

9.2 临时文件创建：tmpfile()

自动在系统的临时目录（如 /tmp）下创建一个唯一的、处于读写状态的临时二进制文件流。

  • 机制与安全：该临时文件在其关闭 (fclose()) 或程序退出时，会被内核自动从磁盘上物理删除，不会留下垃圾数据，非常安全。

9.3 Linux C 错误处理机制：errno、strerror 与 perror

在 Linux 系统编程中，大部分系统调用和库函数在执行出错时会返回特定的负值（如 -1 或 NULL），并将全局错误标识变量 errno 设置为对应的错误状态码。

1) 全局错误号 errno 规范

  • 头文件：#include <errno.h>，该头文件定义了符号 errno 以及各种以 E 开头的错误常量（如 EACCES 代表权限不足，ENOENT 代表文件不存在）。
  • 必须遵循的两个硬性规则：
    1. 仅在返回值明确指示失败时校验：如果没有发生出错，errno 的值不会被后续成功的系统调用清零。因此，只有当函数返回值（如返回 -1）表示出错时，再去检查 errno 才有意义。
    2. 不会被系统调用自动置 0：任何系统调用/标准库函数在出错时都会将 errno 设置为一个非零常量，但它们永远不会将其置为 0。
  • **多线程/线程安全 (Thread Safety)：在现代支持多线程（POSIX 线程）的环境中，errno 被定义为一个宏，其实质是一个线程局部变量 (Thread-Local Variable)**。每个线程都拥有专属的 errno 副本，不同线程之间互不干扰，从而确保了多线程环境下的调用安全性。

2) 错误描述符转字符串：strerror()

将 errno 整数值映射为具体的英文文本描述字符串。

  • 函数原型：char *strerror(int errnum);（头文件 <string.h>）
  • 线程安全警告：**strerror() 不是线程安全的**！它在内部通常会使用一个静态缓冲区来保存对应的错误信息字符串并返回其指针。如果多个线程并发调用 strerror()，该静态缓冲区的内容就会被互相覆盖。
  • 线程安全替代方案：多线程环境下应强制使用 POSIX 线程安全版本 **strerror_r()**：
    int strerror_r(int errnum, char *buf, size_t buflen);

3) 输出错误到标准出错流：perror()

基于当前的 errno 值，直接在标准错误输出设备（stderr）上打印一条格式化的出错信息并返回。

  • 函数原型：void perror(const char *msg);（头文件 <stdio.h>）
  • 输出格式：msg 指向的字符串 + 冒号 + 空格 + 当前 errno 对应的错误文本 + 换行符。
