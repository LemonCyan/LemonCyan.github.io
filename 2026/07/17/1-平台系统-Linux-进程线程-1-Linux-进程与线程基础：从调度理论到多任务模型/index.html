Linux 进程与线程基础：从调度理论到多任务模型

Linux 进程与线程基础：从调度理论到多任务模型

在 Linux 编程的世界里，搞高并发只有两条路：多进程或者多线程。很多新手一上来就手搓代码，结果经常掉进锁竞争死锁、僵尸进程成群、或者主线程死了子线程跟着夭折的深坑里。本文咱们就剥开操作系统的外壳，从最底层的调度理论开始，一直聊到多任务控制的实战避坑。

一、 大局观：多进程 VS 多线程，怎么选？

在选择多任务并发模型时，我们必须跳出“多线程性能一定好”的思维定势。在 Linux 系统下，进程和线程的创建与调度开销差异没有想象中那么悬殊。我们通过两者的优缺点来做一次深度面对面。

1. 多进程模型 (Multi-process)

每个进程都拥有独立的 4GB 虚拟内存地址空间（在 32 位系统下，其中 03GB 是用户空间，3G4GB 是内核空间）。

  • 【爽点/优点】：
    1. 强物理隔离：每个进程拥有自己独立的地址空间和页表。哪怕子进程因为内存越界、野指针崩溃，父进程及其他子进程依然稳如老狗，绝不连累他人。
    2. 开发简单与免锁设计：由于内存空间物理隔离，不同进程之间无法直接读写对方的数据，因此几乎不需要加复杂的互斥锁，彻底告别死锁烦恼。
    3. 横向扩展容易：多进程程序非常容易进行多机部署和分布式改造，可以通过增加服务器或多核 CPU 线性提升吞吐量。
  • 【坑点/缺点】：
    1. 跨进程数据通信（IPC）繁琐：因为进程间存在 4GB 的物理隔离屏障，想要传递数据必须使用管道、共享内存、信号量、消息队列等复杂的跨进程通信机制，开发成本较高。
    2. 物理内存开销大：每个进程都有自己独立的数据段、堆栈、文件描述符表等。在创建数千个并发任务时，物理内存很快就会被吃光。
    3. 进程切换开销大：当进程发生切换时，CPU 不仅要保存寄存器现场，还需要切换页表（修改 CR3 寄存器），这会导致 CPU 的 TLB（页表缓冲区）全部失效。当新进程运行时，会发生大量的 TLB 未命中，导致性能抖动。

2. 多线程模型 (Multi-thread)

多线程是同一个进程内部的并发模型。所有线程共享同一个进程的虚拟内存空间和资源。

  • 【爽点/优点】：
    1. 内存无缝共享：同一个进程内部的所有线程直接共享全局变量区、堆区。线程间传递复杂数据只需要传一个指针，不需要任何跨进程通信，速度极快。
    2. 创建与销毁开销极低：创建线程不需要重新分配页表、文件描述符表等，只是在进程的用户空间里开辟一块几兆的栈空间，速度非常快。
    3. 切换效率极高：同一进程内的线程切换不需要切换页表，不会导致 TLB 失效，CPU 寄存器现场的保存和恢复也更加轻量。
  • 【坑点/缺点】：
    1. 并发锁竞争剧烈：因为内存是共享的，当多个线程同时写同一块内存时，必须引入互斥锁、自旋锁或读写锁。锁的使用稍有不慎，就会导致死锁或大量的锁竞争，CPU 会在加锁解锁中空转，效率急剧下降。
    2. 稳定性堪忧（一人犯错，全家升天）：线程没有独立的内存保护屏障。只要其中一个线程发生了段错误（如访问空指针），整个进程会被操作系统强制终止，其他上百个正在正常运行的线程也会陪葬。
    3. 受限于 4GB 空间极限：在 32 位系统下，一个进程最多只有 3GB~4GB 的虚拟空间，而每个线程通常默认需要 8MB 的栈空间，这限制了单个进程内所能创建的最大线程数。

3. 多进程与多线程的硬核对比表

对比维度 多进程 (Multi-Process) 多线程 (Multi-Thread)
内存空间 每个进程独立 4GB，互不干扰 共享进程的地址空间，各线程拥有独立栈区
创建开销 较重（需要分配页表、描述符表等，走 COW） 极轻（只需开辟独立栈空间）
切换开销 较大（需要切换页表，会导致 TLB 失效） 较小（不换页表，TLB 缓存有效）
数据共享 极其困难（必须使用 IPC 机制） 极其简单（直接读写全局变量/堆区）
同步控制 简单（一般不需要锁） 极复杂（锁竞争、死锁、竞态条件）
稳定性 极强（子进程挂了不影响其他进程） 极弱（一个线程挂，整个进程灰飞烟灭）

大厂选型原则：

  • 多进程是“立体交通系统”，虽然建造成本高，但各走各的路，互不干扰，绝不发生全局堵死。适用于稳定性要求极高的系统，如 Nginx 网关、PostgreSQL 数据库。
  • 多线程是“平面交通系统”，造价低廉，但如果红绿灯（锁）设置得不好，老是堵车，一出车祸整个交通全部瘫痪。适用于有海量共享数据、高频轻量级计算的系统，如游戏逻辑服务器、售票系统。

二、 操作系统调度理论底座

操作系统调度的本质，就是在“就绪队列”任务多于“处理核心数”时，通过一定的策略将 CPU 资源合理地分配给各个任务。

1. 三级调度体系

在多道程序系统中，一个作业从外存提交到最终运行完成，需要经历以下三级调度：

  • 高级调度（作业调度）：其主要任务是按一定的原则从外存上处于后备状态的作业中挑选一个或多个，给它们分配内存、输入输出设备等必要资源，并为它们建立进程，使其获得竞争处理器的权利。在多道批处理系统中配置，执行频率较低（几分钟一次）。
  • 中级调度（内存调度）：为了提高内存利用率和系统吞吐率，系统会将暂时不能运行的进程调至外存等待（称为挂起状态）。当中级调度发现内存空闲且这些挂起进程已具备运行条件时，就会把它们重新调入内存，并修改状态为就绪状态。
  • 低级调度（进程/线程调度）：按照某种算法从内存的就绪队列中选择一个进程/线程，将 CPU 分配给它。这是操作系统中最基本、最频繁的调度（几十毫秒一次）。

2. 调度方式：抢占与非抢占

  • 非剥夺调度方式（非抢占式）：一旦进程占用了 CPU，就会一直运行下去，直到该进程执行完毕，或者因为主动调用 sleep、I/O 阻塞而让出 CPU。这种方式简单，系统开销小，但无法用于分时系统和实时系统。
  • 剥夺调度方式（抢占式）：当有更重要的进程需要使用 CPU 时，系统会立即暂停当前运行的进程，强行收回其 CPU 使用权，分配给新来的紧急进程。抢占通常遵循：优先权原则、短进程优先原则或时间片原则。

3. 经典调度算法深度剖析与场景模拟

为了更直观地理解各种调度算法的特点，我们设计一个具体的测试场景。假设系统在同一时间收到了 3 个作业，它们的到达时间与预计运行时间如下：

  • 作业 A：到达时间 0ms，需要运行 10ms
  • 作业 B：到达时间 2ms，需要运行 4ms
  • 作业 C：到达时间 4ms，需要运行 2ms

下面我们使用不同的算法来调度，并计算各自的平均周转时间（周转时间 = 完成时间 - 到达时间）。

① 先来先服务 (FCFS - First-Come, First-Served)

谁先到达，谁就先占用 CPU 运行到结束。

  • 调度时间轴：
    • 0ms - 10ms：作业 A 运行并完成。
    • 10ms - 14ms：作业 B 开始运行并完成。
    • 14ms - 16ms：作业 C 开始运行并完成。
  • 周转时间计算：
    • 作业 A 周转时间：$10 - 0 = 10\text{ms}$
    • 作业 B 周转时间：$14 - 2 = 12\text{ms}$
    • 作业 C 周转时间：$16 - 4 = 12\text{ms}$
    • 平均周转时间：$(10 + 12 + 12) / 3 = 11.33\text{ms}$
  • 特点分析：极其有利于长作业，对短作业非常不公。作业 C 只需要 2ms 的时间，却被作业 A 堵在后面等了整整 10ms。

② 短作业优先 (SJF - Shortest Job First，非抢占)

当 CPU 空闲时，优先选择就绪队列中运行时间最短的作业。

  • 调度时间轴：
    • 0ms 时，就绪队列只有作业 A，作业 A 占用 CPU 运行。
    • 由于是非抢占式，作业 A 必须一口气跑完（到 10ms）。在此期间，作业 B（2ms 时）和作业 C（4ms 时）陆续进入就绪队列。
    • 10ms 时，作业 A 完成。此时就绪队列有作业 B (预计 4ms) 和作业 C (预计 2ms)。因为作业 C 较短，所以优先运行作业 C。
    • 10ms - 12ms：作业 C 运行并完成。
    • 12ms - 16ms：作业 B 运行并完成。
  • 周转时间计算：
    • 作业 A 周转时间：$10 - 0 = 10\text{ms}$
    • 作业 C 周转时间：$12 - 4 = 8\text{ms}$
    • 作业 B 周转时间：$16 - 2 = 14\text{ms}$
    • 平均周转时间：$(10 + 8 + 14) / 3 = 10.67\text{ms}$
  • 特点分析：平均等待和周转时间比 FCFS 明显缩短。但如果系统源源不断进入短作业，长作业（如 A）在就绪队列中可能会被无限“饿死”。

③ 高响应比优先 (HRRN - Highest Response Ratio Next)

每次调度时，计算就绪队列中所有任务的响应比，响应比最高的优先运行。
$$\text{响应比} (R_p) = \frac{\text{等待时间} + \text{预计运行时间}}{\text{预计运行时间}} = 1 + \frac{\text{等待时间}}{\text{预计运行时间}}$$

  • 调度时间轴：
    • 0ms 时，只有作业 A，作业 A 运行至 10ms 完成。
    • 10ms 时，作业 B 已经等待了 $10 - 2 = 8\text{ms}$，作业 C 已经等待了 $10 - 4 = 6\text{ms}$。
    • 计算此时的响应比：
      • 作业 B：$R_p(B) = 1 + 8 / 4 = 3.0$
      • 作业 C：$R_p(C) = 1 + 6 / 2 = 4.0$
    • 作业 C 响应比较高，因此 10ms - 12ms 调度运行作业 C。
    • 12ms - 16ms 运行作业 B。
  • 特点分析：完美平衡了短作业优先和避免饿死。短作业由于运行时间短，天然容易获得高响应比；长作业随着等待时间变长，其分子不断变大，响应比也会逐渐升高，直到被调度，防止被饿死。

④ 时间片轮转 (RR - Round Robin，设时间片为 2ms)

就绪队列的任务按 FCFS 排序，每个任务轮流跑一个时间片（2ms），时间片到期后强制放回队尾。

  • 调度时间轴：
    • 0ms - 2ms：作业 A 运行（还剩 8ms）。此时时间片到，作业 A 被放回队尾。因为在 2ms 时作业 B 到达，就绪队列此时为：B -> A。
    • 2ms - 4ms：作业 B 运行（还剩 2ms）。在此期间，作业 C（4ms 时）到达。队列变为：A -> C -> B。
    • 4ms - 6ms：作业 A 运行（还剩 6ms）。队列变为：C -> B -> A。
    • 6ms - 8ms：作业 C 运行（还剩 0ms，完成！）。队列变为：B -> A。
    • 8ms - 10ms：作业 B 运行（还剩 0ms，完成！）。队列变为：A。
    • 10ms - 16ms：作业 A 一口气跑完剩下的 6ms。
  • 周转时间计算：
    • 作业 C 完成时间 8ms，周转时间：$8 - 4 = 4\text{ms}$
    • 作业 B 完成时间 10ms，周转时间：$10 - 2 = 8\text{ms}$
    • 作业 A 完成时间 16ms，周转时间：$16 - 0 = 16\text{ms}$
    • 平均周转时间：$(4 + 8 + 16) / 3 = 9.33\text{ms}$
  • 特点分析：由于短作业能迅速通过频繁切换拿到 CPU 并完成，平均周转时间非常优秀，交互体验极佳。

⑤ 多级反馈队列 (MLFQ - Multilevel Feedback Queue)

  • 策略：
    1. 设置多个就绪队列，每个队列优先级不同。第 1 队列优先级最高，时间片最小。往后时间片依次加倍，优先级递减。
    2. 新进程进来，一律先放第 1 队列。如果当前队列的时间片用完前它还没运行完，就强制剥夺 CPU 降级到下一级队列。
    3. 只有高级队列全部为空时，低级队列的任务才能分到 CPU。
  • 优点：是现代操作系统（如 Linux/Windows）的调度核心演变方向。它不需要预先估计任务运行时间，能自动识别 I/O 繁忙型（因等待 I/O 主动让出 CPU，停留在高级队列）和 CPU 繁忙型（一直占着 CPU 导致被迫降级），保证系统整体响应极其流畅。

三、 Linux 进程的底层面貌

在 Linux 内核中，进程和线程在底层并没有严格的物理界限，它们都是用统一的结构体 struct task_struct 来表示的。

1. 进程描述符 task_struct 的核心成员

task_struct 结构体非常庞大，它完整地记录了一个进程生存期间的所有关键信息：

  • **进程状态 (State)**：记录进程当前是运行、阻塞还是僵死。
  • **进程标识符 (PID/PPID)**：唯一的进程号和父进程号。
  • **进程优先级 (Priority)**：静态优先级和动态优先级，以及 nice 值。
  • 内存相关信息：指向进程页表的指针（struct mm_struct *mm），定义了进程的虚拟地址空间。
  • 文件系统信息：进程当前的工作目录、打开的文件描述符表等。
  • 信号处理信息：当前挂起的信号、信号屏蔽字、信号处理函数指针数组等。

2. 进程状态机转移图

Linux 进程的生命周期就是状态机转移的过程，主要状态如下：

  • TASK_RUNNING（运行态/就绪态）：表示进程正在 CPU 上运行，或者已经准备好了，随时等待内核调度器选择它。
  • TASK_INTERRUPTIBLE（可中断阻塞态）：进程因等待某个事件（如等待套接字数据、等待定时器）而睡眠。它可以被信号唤醒。一旦收到信号，进程会退出休眠，进入运行队列排队。
  • TASK_UNINTERRUPTIBLE（不可中断阻塞态）：进程也处于睡眠状态，但它完全屏蔽信号。无论给它发什么信号（甚至是 kill -9），它都无动于衷，只有它所等待的特定硬件事件（如磁盘读写完毕）发生时，才会被唤醒。这可以保证一些关键的内核操作不被中断。
  • TASK_STOPPED（暂停态）：进程被暂停执行。收到 SIGSTOP、SIGTSTP 信号进入此状态，收到 SIGCONT 信号恢复。
  • EXIT_ZOMBIE（僵尸态）：进程已经调用了 exit 退出，占用的绝大部分内存和文件资源已被回收，但在进程表中还留着一个位置（task_struct 壳子），里面保存着退出的原因和状态，供父进程读取。

3. 内核态与用户态的隔离与陷入

CPU 将特权级别划分为 Ring 0 ~ Ring 3，Linux 系统利用了其中的两个级别：

  • 用户态（User Mode - Ring 3）：普通应用程序运行的特权级别。不能直接访问物理硬件，不能读写 3GB~4GB 的内核虚拟地址空间。
  • 内核态（Kernel Mode - Ring 0）：操作系统内核和驱动运行的地方，拥有最高的 CPU 特权，可以直接操控硬件、读写所有物理内存。

当用户程序需要执行读写磁盘、发送网络数据包、分配物理页面等操作时，它必须使用系统调用（System Call）切入内核态。

4. 进程上下文、中断上下文与原子上下文的底层细节

① 什么是上下文（Context）？

在计算机操作系统中，“上下文”指的是程序在执行时所处的完整物理与逻辑环境。进程的上下文可以分为三个部分：

  • 用户级上下文：包括进程的代码段（text）、数据段（data）、用户堆栈（user stack）以及映射的共享存储区。
  • 寄存器上下文：包括 CPU 的通用寄存器（如 EAX, EBX）、程序计数器（IP）、处理器状态寄存器（EFLAGS）以及栈指针（ESP/RSP）。
  • 系统级上下文：包括进程控制块 task_struct、内存管理信息（mm_struct、页表等）以及进程的内核栈（kernel stack）。

② 进程上下文 (Process Context)

当应用程序执行系统调用或触发硬件异常陷入内核时，内核代表当前进程在内核空间运行。此时，内核空间针对用户进程的这种代表就运行于进程上下文。

  • 物理机制：进程上下文是与特定进程密切绑定的，在此状态下，内核可以使用该进程的内核栈，并且可以发生睡眠和阻塞（例如调用 sleep() 或挂起等待磁盘 I/O 完毕）。由于与 current 进程直接关联，在进程上下文中引用 current 指针是有明确意义的。
  • 上下文切换：当系统发生进程调度切换时，需要完整地保存源进程的整个寄存器和页表映射，开销较大。而普通的系统调用只发生模式切换（Mode Switch），只需切换部分寄存器，不需切换页表，开销极小。

③ 中断上下文 (Interrupt Context)

当硬件设备（如网卡、磁盘、键盘）触发中断信号时，CPU 会暂停当前运行的进程，自动调用中断处理程序陷入内核。此时，内核代表硬件在内核空间运行，即处于中断上下文。

  • 物理限制：中断上下文与任何用户进程无关，它是随机打断当前 CPU 上正在运行的进程的。因此，在中断上下文中绝对不能发生任何阻塞、睡眠或调用可能引起睡眠的函数。一旦中断上下文睡眠（例如等待某个信号量），由于它不属于任何进程，系统调度器将永远无法唤醒它，这会直接导致内核崩溃死锁。
  • 无用户空间映射：中断上下文无法访问用户空间的虚拟地址（因为此时 current 进程是随机被打断的，其页表并不对应触发中断的软硬件环境）。
  • 不可抢占：中断上下文必须尽快执行完毕并返回，且不支持进程抢占。耗时任务必须交给底半部（Bottom Half，如工作队列或 Tasklet）延迟处理。

④ 原子上下文 (Atomic Context)

在内核编程中，如果代码处于不可中断、不可抢占的状态，我们称其处于原子上下文。原子上下文主要涵盖：

  1. 正在执行硬中断处理程序（Interrupt Handler）。
  2. 正在执行软中断（Softirq / Tasklet）。
  3. 进程处于内核态，但此时持有了自旋锁（Spinlock）（自旋锁被占用时，内核会自动禁用当前 CPU 的抢占机制）。

⚠️ 原子上下文的四大铁律：

  • 绝不能进入睡眠。
  • 绝不能访问用户空间虚拟内存。
  • 绝不能占用互斥锁（Mutex）（Mutex 拿不到锁时会使进程休眠，必须改用自旋锁 Spinlock）。
  • 绝不能执行任何耗时任务（否则会严重影响系统的实时性与吞吐量）。

⑤ 原子上下文的内核判定宏

Linux 内核提供了四个宏来判断当前 CPU 处于何种上下文中：

这些宏底层的物理依据是访问当前进程 thread_info 结构体中的位掩码成员 preempt_count。该变量在内核中的位结构如下：

  • PREEMPT_MASK (0-7位)：记录自旋锁的加锁深度，最大容许抢占深度为 256。
  • SOFTIRQ_MASK (8-15位)：记录软中断的禁用/嵌套深度，最大为 256。
  • HARDIRQ_MASK (16-27位)：记录硬中断的嵌套深度，最大为 4096。
  • 第 28 位为 PREEMPT_ACTIVE 标志，用来标识内核抢占是否正在进行。

凡是上面四个宏返回 1 的地方，都代表当前处于原子上下文中，抢占被禁用，不容许任何睡眠操作。

⚠️ 非抢占式内核下的 in_atomic() 陷阱：
在内核配置了非抢占模式（CONFIG_PREEMPT=n，常见于一些注重吞吐量的服务器内核）时，spin_lock() 底层并不会去修改 preempt_count 的值。这意味着即使你持有了自旋锁，in_atomic() 宏依然会返回 0，错误地告诉内核当前“不在原子上下文”，从而可能诱发严重的睡眠 Bug。因此，在编写可移植的内核模块时，绝对不能单纯依赖 in_atomic() 来防范自旋锁睡眠风险。
