微机三大总线、经典串行协议(UART/SPI/I2C/CAN)与以太网MII接口硬件指南

微机三大总线、经典串行协议(UART/SPI/I2C/CAN)与以太网MII接口硬件指南

在嵌入式系统级开发与底层硬件选型中，各种通信总线和协议就像是人体的神经网络，负责在 CPU、内存和千奇百怪的外设之间高速传输指令与数据。一个优秀的底层软件或驱动开发人员，不仅要懂得在系统空间里调用 read/write，更需要看懂示波器上的波形，理清信号线上的上拉电阻、时钟极性、以及碰撞仲裁等硬件逻辑。

本指南将自底向上，深度拆解计算机硬件总线体系：从微机三大总线基础，到四大经典串行外设接口协议（UART、SPI、I2C、CAN）的时序与硬件实现，再到工业级网络物理层 MII 总线与 MAC-PHY 之间的底层通信机理，为你搭建一套完备的硬件总线技术图谱。

第一部分：微机系统三大总线基础

在微型计算机体系结构中，系统总线（System Bus）是连接 CPU、内存和 I/O 接口的公共通信干线。根据传输信息的不同，系统总线被划分为经典的数据总线、地址总线与控制总线。

1.1 三大总线的职责与工作机制

1. 数据总线 (Data Bus, DB)

  • 职责：用来传输 CPU 与存储器、或者 CPU 与外设接口之间的指令代码、状态信息以及实际数据。
  • 物理特性：双向三态总线。CPU 既可以从总线上读取数据（输入），也可以向总线写入数据（输出）。
  • 性能指标：数据总线的宽度（位数）是微处理器的核心指标，通常与 CPU 字长一致（例如 16 位处理器 8086 的 DB 宽度为 16 位，32 位 ARM 处理器的 DB 宽度为 32 位）。

2. 地址总线 (Address Bus, AB)

  • 职责：专门用来传送 CPU 指示的内存单元地址或 I/O 端口地址。
  • 物理特性：单向三态总线。地址只能由主控端（通常是 CPU 或 DMA 控制器）发出，送往存储器或 I/O 接口，不能逆向传输。
  • 寻址空间计算：地址总线的宽度决定了 CPU 的最大寻址空间。若地址总线为 $n$ 位，则寻址空间为 $2^n$ 字节。
    • 8 位微机（16 位地址线）：最大寻址空间 $2^{16} = 64\text{ KB}$。
    • 16 位微机（20 位地址线）：最大寻址空间 $2^{20} = 1\text{ MB}$（如 8086 段寄存器寻址）。
    • 32 位微机（32 位地址线）：最大寻址空间 $2^{32} = 4\text{ GB}$。
    • 64 位现代处理器（64 位地址线，实际物理通常为 48 位）：理论最大寻址空间高达 16 EB。

3. 控制总线 (Control Bus, CB)

  • 职责：用来传送各种控制信号、时钟时序信号以及状态反馈。
  • 物理特性：控制总线的单根信号线通常是单向的，但作为整组总线，其信号流向是双向的。
    • 输出控制信号（CPU -> 外设）：例如内存读（MRD#）、内存写（MWR#）、I/O 读写控制、中断响应（INTA）等。
    • 输入状态信号（外设 -> CPU）：例如中断请求（IRQ）、总线请求（HOLD）、设备就绪（READY）等。

1.2 系统总线的四大基本特性

为了保证主板上林林总总的芯片能够稳定通信，总线在物理层面必须具备以下四大特性：

  1. 物理特性（机械特性）：规定了总线物理连接器的几何尺寸、形状、引脚个数、排列顺序以及插头插座的配合规范。
  2. 功能特性：明确指明每一根物理信号线的功能作用（例如哪几根是地址线，哪几根是时钟线、电源线或控制线）。
  3. 电气特性：定义了信号线上电平的物理范围、逻辑 1 和逻辑 0 对应的电压阀值。对于 TTL 电平，通常规定 $2.4\text{V} \sim 5\text{V}$ 为高电平（逻辑 1），$0\text{V} \sim 0.8\text{V}$ 为低电平（逻辑 0）。
  4. 时间特性（逻辑时序）：指明在总线读写操作过程中，各根信号线上电平发生跳变的先后顺序和时序窗口（即建立时间和保持时间），确保接收端在数据稳定时进行采样。

1.3 核心硬件概念：三态逻辑与高阻态 (High Impedance)

在总线设计中，最核心的问题是“总线竞争”——如果两个硬件输出端同时向一根数据线写数据（一个输出高电平，另一个输出低电平），会导致瞬间物理短路，烧毁芯片。
为了解决这一冲突，总线器件必须采用三态门（Three-State Gate）输出结构：

  • 三态定义：高电平、低电平和高阻态（High-Z）。
  • 高阻态原理：当控制端的使能信号无效时，输出引脚在物理上呈现极高的内阻（相当于开关断开，与总线隔离）。此时，该器件不对总线产生任何电平影响，允许其他器件驱动总线。
  • 工作流：主设备发起通信 -> 选通目的从设备 -> 目的设备打开三态门，驱动数据线 -> 通信结束 -> 目的设备输出变回高阻态，让出总线使用权。

1.4 总线带宽计算公式与实战分析

总线带宽是指单位时间内总线上能够传输的最大数据量（以字节/秒表示）。其计算公式为：
$$\text{总线带宽} = \text{总线工作频率} \times \text{总线宽度 (Bit)} / 8 \times \text{每个时钟周期传输的数据次数}$$

  • 典型实战案例：
    假设一条传统 PCI 总线，工作频率为 $33\text{ MHz}$，总线宽度为 $32\text{ 位}$，每个时钟周期传输 $1$ 次数据：
    $$\text{带宽} = 33\text{ MHz} \times 32\text{ Bit} / 8 \times 1 = 132\text{ MB/s}$$
    如果升级为 $66\text{ MHz}$、64 位的 PCI 总线，则带宽为：
    $$\text{带宽} = 66\text{ MHz} \times 64\text{ Bit} / 8 \times 1 = 528\text{ MB/s}$$

1.5 局部总线与扩展总线的演进历史

在个人电脑与嵌入式总线发展史中，总线架构经历了从同步并行到异步串行的几次重大演进：

  1. **ISA 总线 (Industry Standard Architecture)**：经典的 8 位/16 位系统扩展总线，工作频率仅为 $8.33\text{ MHz}$，带宽不足 $8\text{ MB/s}$。引脚极多且极慢，在现代主板上早已绝迹。
  2. VESA 局部总线：为了加速显卡访问，直接从 CPU 管脚引出，运行在 $33\text{ MHz}$ 下的 32 位局部总线，但因为干扰严重、负载能力差（只能挂 2-3 个设备）而被淘汰。
  3. **PCI 总线 (Peripheral Component Interconnect)**：Intel 推出的高性能同步并行总线，支持 32/64 位，工作频率 $33\text{ MHz}/66\text{ MHz}$。它独立于 CPU，通过桥接电路与 CPU 隔离，支持即插即用（PnP）与 DMA 传送，统治了 PC 主板数十年。
  4. 串行总线革命：当并行总线的时钟频率超过 $133\text{ MHz}$ 时，信号线间的“时钟偏斜（Clock Skew）”和电磁串扰成为无法逾越的物理障碍。因此，以 PCIe、USB 为代表的高速串行差分总线走上前台，完成了对并行总线的全面绞杀。

第二部分：四大经典串行总线协议深度对比

在板级外设通信中，并行总线由于走线繁多、电磁干扰严重，逐渐被串行总线取代。以下详细剖析 UART、SPI、I2C 和 CAN 这四种最经典、最常用的串行协议。

2.1 UART (通用异步收发器)

UART 是一种广泛应用的板级异步串行通信协议。因为“异步”，发送端与接收端没有共享的时钟同步线，全靠双方提前约定好的波特率（Baud Rate）来实现数据对齐。

  1. 物理连线：最简模式下只需两根线：发送端 TXD 和接收端 RXD，双方共地（GND）。
  2. 帧格式定义：数据以一帧（Frame）为单位发送：
    • **起始位 (Start Bit)**：1 位，逻辑 0。用于打破空闲状态的高电平，通知接收器开始准备采样。
    • **数据位 (Data Bits)**：通常为 5 ~ 8 位（常用 8 位，LSB 低位在前），表示实际数据。
    • **校验位 (Parity Bit)**：可选的 1 位奇偶校验，用于检测单比特数据传输错误。
    • **停止位 (Stop Bit)**：1 或 2 位，逻辑 1。将总线拉高，为下一帧的起始位跳变做准备。

2.2 SPI (串行外围接口)

SPI 是由摩托罗拉推出的高速、全双工、同步串行总线。

  1. 物理连线：经典的 4 线结构：
    • SCLK：串行时钟，由 Master 主机产生，驱动数据同步。
    • MOSI (Master Out, Slave In)：主机输出，从机输入。
    • MISO (Master In, Slave Out)：主机输入，从机输出。
    • CS# (Chip Select)：从机片选线，低电平有效。

2.3 I2C (内置集成电路总线)

I2C 是由飞利浦（NXP）推出的半双工、双线制双向串行总线。它最大的特点是“极省引脚”，仅用两根线即可挂载多达 127 个外设节点。

  1. 物理连线：
    • SDA (Serial Data)：双向数据线。
    • SCL (Serial Clock)：双向时钟线。
  2. 核心硬件：开漏输出 (Open-Drain) 与线与 (Wired-AND) 逻辑：
    • 硬件接法：I2C 挂载的每个设备，其 SDA 和 SCL 引脚内部均为漏极开路（Open-Drain）输出。引脚只能主动拉低电平（输出逻辑 0），而无法主动输出高电平。因此，总线物理上必须外接两个上拉电阻（通常为 $4.7\text{ k}\Omega$），将总线拉高到 Vcc。
    • 线与机制：如果所有设备都释放总线（不输出），总线由上拉电阻抬高为逻辑 1。只要任何一个设备输出逻辑 0，整条总线立刻被强行拉低。这种机制允许任何设备直接检测总线是否空闲，且即使多个设备同时输出也不会引发电气损坏。

2.4 CAN (控制器局域网总线)

CAN 总线是由德国博世公司为汽车监测与工业控制设计的现场多主总线，具备极高的抗电磁干扰性与硬件纠错机制。

  1. 物理差分信号：
    CAN 总线采用差分信号传输，两根物理线为 CANH 和 CANL。为了防止信号反射干扰，总线两端必须并接 $120\text{ }\Omega$ 的端接电阻。
    • **隐性状态 (Recessive, 对应逻辑 1)**：CANH 与 CANL 的差值几乎为 $0\text{V}$（均为 $2.5\text{V}$ 左右）。
    • **显性状态 (Dominant, 对应逻辑 0)**：CANH 升至 $3.5\text{V}$，CANL 降至 $1.5\text{V}$，差值达到 $2.0\text{V}$。
    • 冲突机制：显性（0）拥有绝对的覆盖权。如果一个节点发送逻辑 1（隐性），另一个节点发送逻辑 0（显性），总线呈现显性（0）。
  2. **无损逐位仲裁冲突竞争 (Non-destructive Bitwise Arbitration)**：
    CAN 采用多主结构。当多个节点在空闲时同时向总线发起通信时：
    • 每个节点在发送数据的同时，会实时读取总线上的实际电平。
    • 结论：报文标识符（ID）越小的帧，其优先级越高，且能抢占总线。

2.6 硬件流控制：RTS/CTS 与软件流控制逻辑

当串口通信速率较高且接收端 CPU 负载极重时，接收端可能无法及时从 UART FIFO 中读出数据，从而引发**溢出错误 (Overrun Error)**。为了解决这一痛点，通常采用流量控制：

  1. **硬件流控制 (RTS/CTS)**：
    引入额外的两根握手物理信号线：
    • RTS (Request to Send，输入/输出)：接收端输出给发送端。如果接收端缓冲区有空间，拉低 RTS，告知发送端“你可以发送”；若缓冲区快满，则拉高 RTS，要求发送端暂停。
    • CTS (Clear to Send，输入)：发送端检测此引脚。如果为低电平，发送数据；如果为高电平，立即挂起发送队列，等待接收端释放。
  2. **软件流控制 (XON/XOFF)**：
    不依赖任何额外的硬件走线，而是通过在数据流中插入特定的控制字符实现：
    • 当接收端感觉吃不消时，在 TXD 上向发送端送回一个特殊的 XOFF 字符（通常为 Ctrl+S, 十六进制 0x13）。发送端收到此字符后暂停发送。
    • 当接收端处理完积压数据有空闲时，送回 XON 字符（Ctrl+Q, 十六进制 0x11），发送端恢复传输。
    • 缺点：因为控制符占用了字符集空间，所以不能用于传输纯二进制文件，且由于网络延迟会导致流控响应滞后。

2.7 硬件布线与信号完整性实战指南

在高速板级电路设计中，忽略传输线效应往往会导致通信彻底死锁或偶发数据包校验错误。以下是四大总线布线中的核心硬件坑点与解法：

1. I2C 总线上拉电阻阻值选择与 RC 时间常数

  • 坑点：如果 I2C 总线挂载的从机设备过多（如超过 10 个），或者走线长达几十厘米，总线寄生电容 $C_b$ 会急剧升高（可达 $100\text{ pF} \sim 400\text{ pF}$）。若此时仍使用默认的 $10\text{ k}\Omega$ 上拉电阻，根据 RC 充电时序常数公式：
    $$\tau = R \times C_b$$
    信号线上电平从低变高的上升沿会变得极其缓慢和圆滑，导致高速时钟采样到错误的电平。
  • 实战解法：在快速模式（400 kbps）下，总线寄生电容较大时，应将上拉电阻阻值强行减小到 $2.2\text{ k}\Omega$ 甚至 $1\text{ k}\Omega$。较低的阻值提供了更大的充电电流，使波形重新变得陡峭。但是阻值不能过小（不能小于 $1\text{ k}\Omega$），否则会因为灌电流过大导致器件无法将总线电平拉至逻辑 0。

2. 高速 SPI SCLK 时钟反射与源端串联匹配电阻

  • 坑点：当 SPI 的 SCLK 时钟线运行在 $20\text{ MHz} \sim 50\text{ MHz}$ 时，由于走线阻抗不连续，在接收端会产生严重的信号反射。这在示波器上表现为严重的过冲（Overshoot）和回受（Ring-back），甚至会在时钟跳变沿产生毛刺，导致从机误判断为多次时钟脉冲，从而引发数据错位。
  • 实战解法：必须在主控芯片的 SCLK 输出引脚处，源端串联一个阻值在 $22\text{ }\Omega \sim 33\text{ }\Omega$ 之间的匹配电阻。该电阻可以有效吸收反射波，从而压制过冲波形，让时钟信号边缘顺滑、稳定。

3. CAN 总线共模电压与隔离收发器选型

  • 坑点：CAN 总线虽然采用差分传输，但如果两个互联节点物理距离过远（例如汽车前灯与后备箱控制器，相距几米），或者供电地电势相差极大，会导致差分线上的共模电压超出 CAN 收发器的允许范围（通常为 $-7\text{V} \sim +12\text{V}$），导致器件瞬间烧毁或锁死。
  • 实战解法：必须使用隔离式 CAN 收发器（如 TI ISO1050 或 CTM1051），利用内部磁耦或光耦将总线物理隔离，并引入专门的屏蔽地线（CAN_GND），彻底断开节点间的地电流回路。

2.8 SPI 硬件控制器的内核驱动机制与 DMA 搬运

在使用 Linux 驱动 SPI 核心控制器时，通常不会使用上述 bit-banging 方式，而是通过 Linux 内核的 SPI 核心框架来实现数据传输。

  • 核心结构体：内核用 spi_controller 抽象主机控制器，用 spi_device 抽象挂接的从设备，每次传输请求被包装为 spi_message，其中可以挂接多个不连续的 spi_transfer 内存缓冲区段。
  • 队列模型与异步提交：驱动通过 spi_async() 异步向核心层提交传输包，内核会将其挂载到控制器的内部链表队列中。控制器工作线程通过中断或者轮询逐一取出并执行。
  • DMA 加速搬运机制：
    当单次传输长度超过 FIFO 大小（通常为 64 字节）时，直接用 CPU 搬运会消耗大量 CPU 时间。此时驱动会自动配置 DMA 引擎：
    1. 驱动程序将 TX/RX 缓冲区映射为 DMA 地址空间。
    2. 启动 SPI 发送/接收 DMA 通道（例如 ARM 的 PL080 或 SDMA 控制器）。
    3. SPI 发送 FIFO 产生空信号（TX DRQ），直接触发 DMA 搬运内存数据到 SPI；当接收 FIFO 变满（RX DRQ）时，DMA 控制器自动将 SPI 数据搬移回内存缓冲区。
    4. 传输完成后产生 DMA 传输结束中断（TC），中断服务程序通知 SPI 驱动，完成一次免 CPU 负担的高速报文收发。

2.9 I2C 设备树匹配与内核架构关联

在现代 Linux 内核中，I2C 总线体系分为三层：I2C 核心层 (Core)、I2C 适配器驱动层 (Adapter Driver) 和 I2C 设备驱动层 (Client Driver)。

  • 结构体关联：i2c_adapter 代表主机硬件控制器物理实例（如 i2c0），i2c_algorithm 指明具体的字节/字节块传输物理算法（如 master_xfer）；挂在总线上的外设则抽象为 i2c_client。
  • 设备树实例化：
    在设备树 .dts 文件中，外设作为子节点直接定义在适配器节点内部： 当内核解析设备树时，系统会扫描 i2c0 下的所有子节点，自动为每个子节点创建一个 struct i2c_client 实例，并将 reg 的值写入 client->addr。最后，当有 compatible 属性与 atmel,24c256 匹配的 i2c_driver 注册时，内核就会调用其 probe 函数完成设备驱动绑定。

2.10 电平转换与高速信号完整性电路设计

许多现代低功耗传感器运行在 $1.8\text{V}$，而工业级主控 CPU 或单片机则工作在 $3.3\text{V}$ 或 $5\text{V}$，此时不能直接将它们的总线连接，否则会导致引脚烧毁或无法识别高电平。

  • **双向自适应电平转换芯片 (Auto-Direction Translation)**：
    对于 I2C 这种漏极开路的双向总线，常使用 TXS0108E 或 TXS0104E 转换芯片。该芯片内部集成了单发单收边缘加速电路和上拉电阻，能自动识别信号流动方向，且无需额外的方向控制引脚（DIR）。
  • **推挽输出电平转换芯片 (Push-Pull Translation)**：
    对于 SPI 这种单向、高速推挽总线，建议使用带有方向控制引脚的 TXB0104 芯片。该芯片不支持开漏，但能提供非常干净、陡峭的高频推挽沿，最高传输速率可达 $60\text{ Mbps}$，而 TXS0108E 用于 I2C 这种开漏总线时速度受上拉影响最大只能跑 $2\text{ Mbps}$。
  • 简易二极管物理夹钳法：
    在低速、成本极度敏感的场景下，也可以使用肖特基二极管与上拉电阻搭一个简单的 I2C 电平钳位电路，使得 $3.3\text{V}$ 端和 $5\text{V}$ 端能安全交互而无高压倒灌风险。

第三部分：以太网 MII 通信接口与 MAC-PHY 组成原理

在嵌入式网络设备开发中，网口并不是简单的插座，在 CPU 内部到网线之间，存在着清晰的 MAC 媒体访问控制器 和 PHY 物理层芯片 架构划分。

3.1 网口物理层三部分拓扑架构

对于 MAC 和 PHY 芯片，产业界根据性能和布线成本有三类集成方案：

  • 方案一：CPU 同时集成 MAC 与 PHY：高度集成，外部走线最简，但由于 PHY 属于模拟高频电路，容易受到 CPU 数字噪声的电磁干扰，仅在低速单片机常见。
  • 方案二：CPU 集成 MAC，PHY 采用独立芯片（最常用）：CPU 内部通过高性能 DMA 读写 MAC 寄存器，外部通过标准的 MII 接口排线与专用的独立 PHY 芯片通信。
  • 方案三：CPU 不集成 MAC/PHY，外部挂载集成网卡芯片：例如外接 DM9000、W5500 等芯片，CPU 通过 SPI 或局部总线访问网卡，传输效率受限制。

3.2 MII 标准总线引脚功能解析

MII 接口是 MAC 与 PHY 之间数据交换的标准并行总线。其核心物理引脚功能如下表所示：

MII 引脚名称 信号传输方向 引脚核心职责说明
TXD[0:3] MAC —> PHY 数据发送并行线。4 位数据，半字节同步传输。
TX_EN MAC —> PHY 发送使能控制。拉高时指示发送线上的数据有效。
TX_CLK PHY —> MAC 发送同步时钟。由 PHY 产生（10M网对应 2.5MHz，100M网对应 25MHz）。
RXD[0:3] PHY —> MAC 数据接收并行线。4 位数据，接收数据传输。
RX_DV PHY —> MAC 接收数据有效指示。接收到合法帧时拉高。
RX_CLK PHY —> MAC 接收同步时钟。由 PHY 芯片从接收的网线载波中恢复恢复出时钟。
MDC MAC —> PHY 管理接口时钟线（Management Data Clock）。配置引脚。
MDIO 双向 管理接口数据线（Management Data Input/Output）。配置接口。

3.3 MIIM (MII Management) 串行总线与 PHY 寄存器控制

除了通过 MII 的并行数据线搬运以太网报文外，CPU 的 MAC 还需要读取和设置 PHY 芯片内部的配置寄存器。这一控制通道称为 MIIM（MII Management），仅通过 MDC 和 MDIO 两根信号线工作。

根据 IEEE 802.3 Clause 22 规范，PHY 芯片内部配置寄存器（共 32 个，每个 16 位）的标准排布如下表所示：

寄存器号 寄存器名称 核心位控制属性与作用描述
Register 0 Control Register (控制寄存器) 💡 核心控制位：
- Bit 15: Soft Reset (软件复位)
- Bit 14: Loopback (环回测试，用于调试自检)
- Bit 13: Speed Selection (0=10Mbps, 1=100Mbps)
- Bit 12: Auto-Negotiation Enable (开启自适应协商)
- Bit 8: Duplex Mode (0=半双工, 1=全双工)
Register 1 Status Register (状态寄存器) 💡 只读状态位：
- Bit 5: Auto-Negotiation Complete (自适应协商完成)
- Bit 2: Link Status (💡 1=网线连接正常，0=网线已断开)
Register 2 & 3 PHY Identifier 存储芯片唯一的 OUI 厂商标识及芯片型号代码。
Register 4 Auto-Negotiation Advertisement 配置本网卡向对端交换机宣称的连接能力（如支持全双工、百兆等）。
Register 5 Link Partner Ability 只读对端能力。记录了网线另一端的交换机/设备所支持的最高速率和双工模式。

3.4 MDIO 串行数据帧物理格式

为了读写上述 16 位的寄存器，MDIO 总线上发送的每一帧配置命令都是一个 64 位的时序序列，其物理结构定义如下：

  • PRE (Preamble) 前导码：32 位连续的高电平（1），用于同步 PHY 的时钟恢复。
  • ST (Start of Frame) 起始符：2 位，固定为二进制 01。
  • OP (Opcode) 操作码：2 位。10 代表读操作，01 代表写操作。
  • **PHYAD (PHY Address)**：5 位。支持在同一组 MDC/MDIO 上级联最多 32 个物理 PHY 芯片。
  • **REGAD (Register Address)**：5 位。寻址被选中的 PHY 内部 32 个寄存器中的某一个。
  • TA (Turnaround) 转向周期：2 位。在读操作时，第一位为高阻，第二位由 PHY 芯片拉低，用于给物理器件进行数据输出状态的切换过渡。
  • DATA 数据域：16 位。写操作时由 MAC 发送，读操作时由 PHY 输出，用于读写 PHY 内部寄存器的核心参数。

3.5 接口变体演进：RMII 与 RGMII 架构对比

随着网络速率提升和芯片引脚精简要求，MII 接口变体总线广泛应用于现代嵌入式 Linux 系统：

  1. **RMII 总线 (Reduced MII)**：
    • 优化机制：将 MII 的 16 根引脚精简为 7 根。
    • 实现方法：将数据发送和接收线从 4 位减少为 2 位（TXD[0:1] 和 RXD[0:1]）。同时，将同步时钟频率从 MII 的 $25\text{ MHz}$ 提升两倍至 **$50\text{ MHz}$**，从而保证百兆以太网的总有效带宽仍然保持 $100\text{ Mbps}$ 恒定不变。且收发共享一个参考时钟 REF_CLK。
  2. **RGMII 总线 (Reduced Gigabit MII)**：
    • 优化机制：传统的千兆以太网 GMII 接口需要 24 根线，布线极难，RGMII 将其精简为 12 根。
    • 实现方法：数据线保留为 4 位。采用**双沿采样技术 (Double Data Rate, DDR)**，即在 $125\text{ MHz}$ 时钟的上升沿和下降沿同时采样数据。
    • 带宽计算：$125\text{ MHz} \times 4\text{ Bit} \times 2 (\text{双沿}) = 1000\text{ Mbps} = 1\text{ Gbps}$，成功用 12 根 line 支撑起了千兆网络传输。

3.6 MDIO Clause 45 协议格式与万兆网的演进

在万兆以太网（10G Base-T）或者光模块（SFP+/QSFP+）调试中，经典的 Clause 22 配置接口遇到了致命阻碍：它仅能支持 5 位的寄存器寻址，也就是说一个 PHY 芯片最多只能有 32 个寄存器。这在支持万兆速率、具备数千个参数寄存器的现代 PHY / MMD 面前完全不够用。为此，IEEE 802.3ae 引入了 Clause 45 标准：

  • 分步操作机制：Clause 45 摒弃了“单次直接读写”逻辑，将一次操作分为“地址设置”与“数据读写”两步：
    1. 第一步（设置地址）：发送一个地址帧，OP 码设为 00（Address Setup），将 16 位的寄存器内部地址写入特定的 MMD 寄存器。
    2. 第二步（数据操作）：紧接着发送一个读写帧，OP 码设为 01（Write）、11（Read）或者 10（Read and Increment），对刚才指定的寄存器执行读写。
  • 帧结构变动：
    • ST (Start of Frame) 字段由 01 改为 00，使老旧的 Clause 22 PHY 芯片能够自动忽略该配置帧，防止误触发。
    • DEVAD (Device Address) 代替了原本的 REGAD 字段，用于指示具体的功能子模块（如 1=PMA/PMD 物理媒介，3=PCS 物理编码子层，7=Auto-Negotiation 自适应层）。
  • 通过这种“两步寄存器定位”架构，Clause 45 支持高达 65536 个寄存器的超大寻址空间，完美护航了以太网从千兆到万兆乃至 100G 的无缝演化。

3.7 Linux 内核 PHYLIB 中的 Clause 45 驱动绑定与 Mixed 模式机制

在 Linux 内核以太网子系统中，物理层驱动主要集中在 /drivers/net/phy/ 的 PHYLIB 模块中：

  • Clause 45 在 phy_device 中的判定：
    内核在探测 PHY 芯片时，通过 MDIO 总线发送特定的读取指令。如果发现芯片是一个 Clause 45 设备，内核会在实例化 phy_device 时将其成员 is_c45 设为 true。同时，在 sysfs 节点下会暴露 phy_id 的 32 位标识（高 16 位为厂商 OUI，低 16 位为模块设备型号）。
  • 混合模式读写 helper API：
    为了屏蔽 Clause 22 和 Clause 45 在读写底层时序上的差异，内核抽象了以下统一接口：
    • 如果 is_c45 为 true，内核的 MDIO 驱动层会自动使用 Clause 45 的 00（设置地址）和 01/11（读写数据）两阶段事务传输；
    • 如果 is_c45 为 false 且芯片支持 MMD 间接寻址，内核会通过 Clause 22 的 13 号寄存器（MMD Access Control）与 14 号寄存器（MMD Access Address/Data）进行寄存器映射跳转读写。
  • 这种巧妙的框架封装，使得我们在编写具体的千兆或万兆 PHY 网卡驱动时，可以完全无需关心硬件接线是 Clause 22 还是 45，直接调用通用核心层 API 即可。

💡 板级总线与以太网接口物理层 Debug 避坑 Checklist

在进行板端总线调试前，建议执行以下硬核排查步骤：

  • I2C 总线电平：使用示波器探头点在 SDA/SCL 线上，观察上升沿时间常数是否圆滑。若上升沿圆滑且偶发通信挂死，将上拉电阻阻值减小（如从 4.7K 改为 2.2K）。
  • SPI 时钟反射：在 SCLK 线上测量波形，排查是否有超过 0.5V 的过冲与回受。如有，在主机侧串联一个 22Ω 阻尼电阻。
  • CAN 差分阻抗：断电后用万用表测量 CANH 与 CANL 之间的阻抗是否在 60Ω 左右。如果是 120Ω 说明漏接了一个端接电阻，若是物理断开（阻抗无限大）则两个端接电阻都漏接了。
  • MII 同步时钟：检查 PHY 芯片送回的 TX_CLK 与 RX_CLK 频率是否与网速匹配（例如百兆网络必须为 25MHz，十兆网络为 2.5MHz，千兆为 125MHz）。
  • MDIO 管理读写：若无法识别 PHYID，首先核对 PHYAD 硬件拉高/拉低状态，确认 MDIO 命令帧中的 PHY 地址是否与板级原理图一致。
