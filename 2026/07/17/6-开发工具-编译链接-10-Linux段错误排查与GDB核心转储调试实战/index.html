Linux 段错误排查与 GDB 核心转储调试实战

Linux 段错误排查与 GDB 核心转储调试实战

在 C/C++ 开发中，
段错误（Segmentation Fault）是令所有开发者头疼的致命崩溃故障。
程序突然异常终止，
控制台只留下一句冰冷的 Segmentation fault。

段错误的本质是程序违反了操作系统的内存管理规则。
通常是硬件 MMU 触发异常，
并由内核向进程发送 SIGSEGV 信号。
本指南将带你从段错误的底层物理机制出发，
深度解析如何启用和配置 Linux 核心转储（Core Dump）。
演示如何通过代码捕获信号自动打印崩溃调用栈。
并以实战代码演示如何利用 GDB 抓出引发崩溃的幕后黑手，
同时针对嵌入式系统提供跨平台的交叉 GDB 符号调试方案。

1. 深入理解段错误（SIGSEGV）的成因

段错误通常是指进程尝试访问其虚拟地址空间之外的物理内存，
或者尝试以非法方式访问合法内存（例如写入只读区域）。

1.1 常见的段错误根源

  1. 空指针解引用（Null Pointer Dereference）：
    试图读取或写入地址为 0x0（NULL）的内存空间。
    这是最典型的段错误原因。
  2. 野指针/悬空指针访问（Wild/Dangling Pointer）：
    访问未初始化或已被 free/delete 释放的内存区域。
  3. 缓冲区溢出（Buffer Overflow）：
    数组越界或指针加减失控，
    导致读写操作覆盖了进程未分配 of 虚拟页面或破坏了函数调用栈（Stack Corruption）。
  4. 内存写入权限冲突：
    试图修改只读内存段。
    例如修改常量字符串：
  5. 栈溢出（Stack Overflow）：
    深度递归或在栈上分配了过大空间的局部变量，
    超出了系统分配给该线程的栈容量空间限制。

1.2 异常内存访问相关系统信号对照表

信号名称 信号值 触发原因 典型场景
SIGSEGV 11 段错误 (Segmentation Violation) 空指针解引用、越界读写未分配的物理页、修改只读内存。
SIGBUS 7 总线错误 (Bus Error) 物理内存对齐错误（Alignment Error）、非法的物理地址访问。
SIGILL 4 非法指令 (Illegal Instruction) 执行了损坏的机器指令、栈破坏导致跳转到了无效的数据区。
SIGFPE 8 浮点异常 (Floating Point Exception) 除以零错误、溢出、浮点计算溢出。

2. 核心转储（Core Dump）机制与系统环境配置

当进程由于未捕获的严重信号（如 SIGSEGV,
SIGFPE,
SIGABRT 等）异常终止时，
操作系统可以将该进程当时的内存映像、CPU 寄存器状态以及系统调用堆栈等信息完整地转储（Dump）保存到一个文件中，
这个文件被称为 Core Dump（核心转储） 文件。

2.1 利用 ulimit 开启 Core Dump 功能

在大多数主流 Linux 发行版中，
为了防止突发崩溃占满磁盘空间，
系统默认将 Core 文件的最大限制设置为 0（即禁用转储）。

2.1.1 ulimit 命令行工具详解

ulimit 命令用于控制 Shell 启动的进程所占用的各种系统资源上限：

其常见选项如下：

  • -H：设定或显示硬性资源限制（只有 root 可以调大）。
  • -S：设定或显示弹性资源限制（不能超过硬性限制）。
  • -c：设置 Core 文件的最大容量限制（单位：512 字节的数据块 blocks）。
  • -s：设置线程最大栈空间（单位：Kbytes）。
  • -n：设置单个进程可开启的文件描述符最大数量。

7. 嵌入式跨平台交叉 Core Dump 符号解析实战

在嵌入式开发中，

此时，
我们需要在宿主机上使用交叉调试器来还原崩溃堆栈。

8. 内存崩溃排查进阶：AddressSanitizer (ASan) 诊断利器

8.1 什么是 AddressSanitizer？

虽然 GDB + Core Dump 非常强大，
但对于一些隐蔽的内存损坏（如越界读写、Use-After-Free），
Core Dump 可能发生在错误发生的很久以后，
甚至由于内存刚好合法而没有触发段错误，
导致难以定位根因。
AddressSanitizer (ASan) 是由 Google 开发的一款快速内存错误检测工具，
已经集成在 GCC (4.8+) 和 Clang 中。
它在编译期对代码进行插桩，
在运行期监控内存访问，
能在发生非法访问的瞬间立刻中断程序并打印极其详细的报告。

9. 段错误调试故障排查 FAQ

Q1: GDB 加载 Core 文件后显示 “No debugging symbols found” 怎么办？

  • 原因：当前加载的可执行二进制文件被去除了符号（stripped）。
  • 解决：在调试时，必须让 GDB 载入包含符号的未剥离版本（编译产物里的原始二进制），而不是拷贝到生产环境下的 stripped 版本。

Q2: 崩溃发生后，没有生成 Core 文件，ulimit -c 已经设置 unlimited 了？

  • 原因 1：磁盘空间不足或当前目录对当前运行用户无写入权限。
  • 原因 2：程序在崩溃前自己捕获了信号并且调用了 exit()，没有以异常终止方式结束。
  • 原因 3：系统使用了 Systemd 的 coredumpd 机制，core 文件被自动收集到了 /var/lib/systemd/coredump 或 journalctl 中。可以通过 coredumpctl list 命令查看。

Q3: Core 文件大小被截断 (Truncated core file) 怎么办？

  • 原因：尽管设置了 ulimit，但可写磁盘剩余空间小于 Core 文件大小，或者程序被强行终止导致写入未完成。
  • 解决：检查磁盘可用空间，或通过配置 core_pattern 将 Core 文件输出到内存文件系统（如 /dev/shm 或 /tmp）。

Q4: 如何在没有 GDB 的开发板上运行远程调试？

  • 解法：在开发板上运行轻量级的 gdbserver，指定端口并附着到进程上；在宿主机上运行交叉调试器（如 aarch64-linux-gnu-gdb），执行 target remote <IP>:<PORT> 进行远程联调。

10. 内存错误检测利器：Valgrind 动态分析

10.1 什么是 Valgrind？

Valgrind 是 Linux 下非常强大的动态二进制分析工具。
它包含一组工具，
其中最常用的是 Memcheck，
用于检测 C/C++ 程序中的各种内存错误：

  • 使用未初始化的内存
  • 内存泄露（malloc 申请后未释放）
  • 读写已释放的内存
  • 越界读写内存（堆/栈溢出）
  • 重复释放内存（Double Free）

10.2 Valgrind 的基本用法

使用 Valgrind 不需要重新编译程序，
可以直接分析二进制文件。
但为了看到具体出错的函数名和代码行号，
建议在编译时加上 -g 选项。
