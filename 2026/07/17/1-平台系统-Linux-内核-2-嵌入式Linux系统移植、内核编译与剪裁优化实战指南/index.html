嵌入式 Linux 系统移植、内核编译与剪裁优化实战指南

在嵌入式 Linux 产品开发中，系统移植与裁剪优化是连接底层硬件与上层应用的桥梁。一个未经裁剪的通用内核不仅体积庞大（数兆甚至数十兆字节），在有限的嵌入式 Flash 中极难容纳，而且启动速度迟缓，无法满足工业控制或消费电子对“瞬间开机”的严苛要求。

本指南将基于三星 S3C2410/2440 (ARM9) 平台，带你走过一条完整的嵌入式 Linux 系统构建全链路：从交叉编译环境搭建，到 U-Boot 板级支持包的深度修改，再到内核驱动移植、Busybox 根文件系统制作、Qt 与 tslib 触摸库移植，最后深入探讨如何通过内核裁剪与启动图调优，实现系统秒级自举启动的极限优化。

2.2 修改 start.S 与 nand.c 实现 Nand Flash 引导启动

原厂的 smdk2410 配置默认只支持从 Nor Flash 启动。由于 Nor Flash 价格昂贵，现代嵌入式系统大多采用 Nand Flash 启动。

4.4 常见文件系统挂载故障排除 (NFS 与 YAFFS2 实战)

在开发板自举的最后一阶段，挂载 Rootfs 经常遇到各种崩溃，以下是两类经典崩溃的深层原因及排查方案：

  1. 崩溃提示：Kernel Panic: VFS: Unable to mount root fs on unknown-block(2,0)
    • 根源分析：
      1. 内核没有编入对应的物理文件系统驱动（例如 YAFFS2 驱动未被勾选，却指定以 rootfstype=yaffs2 挂载）。
      2. U-Boot 环境变量指定的挂载分区编号错误（例如 Rootfs 在 Nand 第 3 分区，但 bootargs 里错写成 root=/dev/mtdblock2，导致内核去挂载只存有 uImage 的 kernel 分区）。
  2. 崩溃提示：NFS Boot error: DHCP/bootp failed, or Permission Denied
    • 根源分析：
      1. 宿主机中 NFS 服务器配置文件 /etc/exports 路径或 IP 网段配置不匹配。
      2. /etc/exports 中未加 no_root_squash 选项。如果缺此选项，嵌入式端以 root 用户身份建立虚拟节点时，主机会将其降权压制为 nobody 用户，导致权限不足直接挂载失败崩溃。

第五部分：内核启动时间与体积极致裁剪优化

当开发板的功能全部跑通后，最终的产品化开发必须要对内核进行极致的“瘦身”与“加速”。

5.1 initcall_debug 启动时间分析

为了揪出到底是哪个内核初始化函数拖慢了开机速度，我们在内核启动命令行中添加：

  1. 获取时间戳日志：内核会在控制台输出每一个初始化函数（initcall）开始和结束的时间戳（微秒级）。
  2. 生成启动分析图：
    拷贝板端串口输出的启动日志并保存为 boot.log。在宿主机上，进入 Linux 内核源码目录，运行编译脚本自带的 Perl 工具： 用矢量图浏览器（如 Inkscape 或 Chrome）打开生成的 boot.svg，整个内核初始化阶段的时间占比、探测（probe）阻塞热点一目了然。

5.2 极致裁剪技巧与配置优化

根据启动图和产品实际需求，我们进行深度的功能阉割：

  1. 精简驱动与硬件检测：
    • 通过查阅相关子目录的 Makefile 和 Kconfig，在 make menuconfig 中剔除所有开发板未焊接的驱动（例如：无用的 PCIE 驱动、USB 网卡驱动、多余的显示芯片支持）。
    • SMP 初始化：即使开发板只有一个 CPU 核心，内核默认可能也会启用对称多处理器支持（CONFIG_SMP）。关闭它！这不仅能直接让编译出的内核大小减少约 180KB，还能省去多核初始化握手的等待时间。
  2. 选用微型替代算法：
    • 将内核通用的伙伴系统/SLAB 物理页分配算法，替换为专门为嵌入式低内存设计的 CONFIG_SLOB 机制，可进一步优化页表常驻大小。
    • 使能 CONFIG_EMBEDDED，允许对内核的底层细节（如内核日志缓冲区大小 CONFIG_LOG_BUF_SHIFT）进行极限微调。

5.3 优化延时与控制台

  1. 预置 lpj（loops_per_jiffy）参数：
    内核每次引导启动时，为了校准 udelay() 精确延迟，都会执行一段耗时较长（约 200ms）的延迟循环校准（Calibrating delay loop）。
    • 实战做法：我们在第一次开机时，在内核串口终端中找到如下日志：
      Calibrating delay loop... 996.14 BogoMIPS (lpj=4980736)
    • 记录下这个 lpj 数值。随后在 U-Boot 中，将此参数固化到内核启动命令行（bootargs）中：
      setenv bootargs 'console=ttySAC0,115200 root=/dev/mtdblock3 lpj=4980736'
    • 效果：再次开机时，内核会直接跳过耗时的物理校准（显示 Calibrating delay loop (skipped) preset value），直接开机，瞬间节省近五分之一的开机耗时！
  2. 静默控制台：
    控制台串口输出（十分低速且会发生总线阻塞）在量产产品中并无用处。在内核命令行中添加 quiet 参数可以静音输出；或者在 menuconfig 中直接设置 CONFIG_PRINTK=n。这虽然会丢失 dmesg 功能，但能瞬间将引导时间再次缩短约 0.5 秒 以上。

5.4 各种压缩算法在嵌入式 ARM 平台上的性能测试

内核被搬运到 SDRAM 前，必须在 Flash 中以压缩包形式存放。不同的压缩算法在“解压速度（依赖 CPU 主频）”与“读取速度（依赖 Nand 接口速率）”之间各执一词。
以下是在基于 TI AM335x (ARM Cortex-A8, 主频 1 GHz) 平台下运行 Linux 5.1 的内核镜像实测数据：

压缩算法类型 生成内核镜像体积 (Bytes) Nand Flash 拷贝耗时 (s) 内核 CPU 解压启动耗时 (s) 综合启动耗时 (s)
Gzip (默认) 2,350,336 0.208 1.243 1.451
LZMA 1,777,000 0.158 2.009 2.167
XZ 1,720,120 0.154 1.845 1.999
LZO 2,533,872 0.224 1.192 1.416
LZ4 2,716,752 0.241 1.221 1.462
  • 结论分析：
    • XZ / LZMA 算法拥有极高的压缩比，内核体积最小（节省 Flash 空间），但解压阶段需要进行复杂的浮点和字典树计算，对于低主频 ARM 来说，CPU 解压时间反而大大拉长，得不偿失。
    • LZO / Gzip 在压缩体积和解压算法开销上取得了最佳平衡。在 1GHz CPU 下，LZO 凭借极速的解密算法，综合启动用时最短（1.416s）。
    • 在产品优化收尾阶段，建议根据自己开发板的 CPU 性能与 Flash 读取带宽，对各算法进行交叉对比，选出最契合自身硬件的解压缩算法。
