Linux System V IPC (共享内存、消息队列与信号灯集) 核心机制与同步实战

在 Linux 跨进程通信（IPC）中，除了基于文件描述符的管道外，还有一套功能极强、活动在内核级别的通信套件 —— System V IPC（包括共享内存、消息队列与信号灯集）。它们共享统一的命名空间设计（Key / ID 机制），提供了极高吞吐量的数据交换与复杂的进程同步手段。本文咱们就来彻底死磕这三大件的 API 底层细节，并提供三个工业级、可编译运行的高级进程同步协作代码。

一、 System V IPC 物理基础：全局 Key 与 ID 机制

与管道随进程退出而自动销毁不同，System V IPC 资源是随内核的（Kernel Persistent），只要不手动删除或关机，它们会一直驻留在内核内存里。

为了让两个完全独立的进程能够准确连接到同一个内核 IPC 对象，System V 引入了“命名与查找”机制：

  1. 全局外部键（Key）：一个 32 位整型数（key_t），用于系统级命名。
    • 生成 API：key_t ftok(const char *pathname, int proj_id);
    • 工作机制：ftok 提取指定文件/目录的 inode 号和设备号，结合 proj_id 的低 8 位，计算出一个唯一的 Key。只要路径和 ID 相同，不同进程算出来的 Key 必定相同。
  2. 内部标识符（ID）：通过 shmget / msgget / semget 系统调用将 Key 注册进内核，内核会返回一个局部整数 ID。后续进程的读写操作全部通过该 ID 来引用。
  3. 权限控制：每个 IPC 对象都拥有一个 ipc_perm 权限结构体，控制其读写所有权，与 Linux 文件权限类似（如 0666 允许任何人读写）。

二、 共享内存 (Shared Memory)：无拷贝的极限速度

共享内存是 Linux 所有 IPC 手段中速度最快的。它通过修改两个进程的页表映射，直接将同一块物理内存映射到它们各自的虚拟地址空间。进程读写共享内存就像读写本地普通数组一样，完全不需要任何系统调用内核态切换，更不需要在用户态与内核态之间来回拷贝数据。

🛠️ 核心 API

  1. 创建/获取共享内存：int shmget(key_t key, size_t size, int shmflg);
    • shmflg：通常传入 IPC_CREAT | 0666。
  2. 映射至虚拟地址空间：void *shmat(int shmid, const void *shmaddr, int shmflg);
    • shmaddr：一般传 NULL，由内核自动选择空闲的虚拟内存地址映射。
    • 返回值：成功返回指向共享物理内存的首地址指针，失败返回 (void *)-1。
  3. 解除映射：int shmdt(const void *shmaddr);
  4. 控制/销毁共享内存：int shmctl(int shmid, int cmd, struct shmid_ds *buf);
    • 销毁命令：shmctl(shmid, IPC_RMID, NULL);（注意：调用该命令后，共享内存不会立刻被内核抹去，而是处于待销毁状态，直到所有映射了它的进程都调用了 shmdt 解除映射后，它才彻底在内核消失）。

三、 消息队列 (Message Queue)：自带同步与类型的信箱

消息队列是内核中维护的一个消息链表。它具有两个非常亮眼的工业特性：

  1. 自带类型过滤器：每条写入的数据都打上了一个类型标号（mtype）。读取时，接收端不仅可以先进先出读取，还可以指定只读取某一种特定类型的消息，非常适合做业务分流。
  2. 自带线程/进程同步安全：如果消息队列中没有指定的消息，msgrcv 默认会自动将进程挂起休眠；当有新消息进来时自动唤醒，完全不需要手工加锁。

🛠️ 核心 API

  1. 创建/获取：int msgget(key_t key, int msgflg);
  2. 发送：int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);
    • msgp 必须指向一个自定义结构体，第一个成员固定为 long mtype。
  3. 接收：ssize_t msgrcv(int msqid, void *msgp, size_t msgsz, long msgtyp, int msgflg);
    • msgtyp：0 代表读取队列中任意第一条消息；> 0 代表只读取类型等于 msgtyp 的第一条消息。
  4. 控制/销毁：int msgctl(int msqid, int cmd, struct msqid_ds *buf);

🛡️ 高级实战二：多子进程并发竞争消费任务分发

四、 信号灯集 (Semaphore Array)：进程级同步计数器

System V 信号灯集（Semaphore Set）是进程间同步的最核心原语。需要特别注意的是：

概念避坑：System V 信号灯不是一个单一的锁，而是一个包含多个信号量的数组。而且，对信号量的 PV 操作是通过定义一个 struct sembuf 结构体，调用 semop 传入内核来原子执行的。

🛠️ 核心 API

  1. 获取/创建：int semget(key_t key, int nsems, int semflg);
    • nsems：该信号灯集里包含的信号量个数。
  2. 设置属性/初始化：int semctl(int semid, int semnum, int cmd, ...);
    • 动态初始化通常会设置信号灯的初始值：
  3. PV 原子操作：int semop(int semid, struct sembuf *sops, size_t nsops);
    • sembuf 结构体成员：
      • sem_num：信号灯在集合中的索引。
      • sem_op：操作值。
        • -1：P 操作（申请锁/资源，当资源为 0 时进程挂起阻塞）。
        • +1：V 操作（释放锁/资源，唤醒等待进程）。
      • sem_flg：通常设为 SEM_UNDO。如果进程未解锁就意外崩溃，内核会自动撤销该进程在此锁上的所有操作，有效防止产生永久死锁。

💡 核心应用场景扩展：利用信号灯集保证程序在系统内单例运行

在实际生产部署中，经常需要防止某个关键守护进程（Daemon）或后台服务进程被重复启动。除了传统的使用锁文件（Pidfile）外，使用 System V 信号灯集也是一种非常硬核且高可靠的解决方案：

  1. 程序启动时，首先通过相同的 Key 创建或打开一个信号灯集。
  2. 调用 semop，执行一个特殊的“等待 0（Zero Wait）”操作：如果信号量值为 0，则顺利通过；如果非 0（说明已有进程启动并将其加 1），则立即失败返回。为了防止阻塞，我们需要设置 IPC_NOWAIT 标志。
  3. 通过检测后，进程立即执行 V 操作（即 sem_op = 1），将信号量值改为 1（锁死单例通道），并设置 SEM_UNDO 标志。
  4. 如果有第二个进程实例企图运行，当它执行“等待 0”操作时，由于信号量值已经是 1，semop 会立即报错返回 EAGAIN。此时第二个实例就可以优雅提示并安全退出。
  5. 由于第一个进程在 V 操作中设置了 SEM_UNDO，当其退出或意外被 kill 终止时，内核会自动将信号量值恢复为 0，允许后续实例重新启动运行。

五、 System V 命令行维护与调试

当您的进程在运行期间发生死锁或意外中断崩溃时，它们在内核中占用的 System V 资源可能并不会释放，这会导致后续程序再次创建时报 EEXIST 或地址冲突错误。

请熟记以下清理与控制命令：

  1. 查看当前内核活动的 IPC 资源：
    • ipcs -m：只查看共享内存（Shared Memory）。
    • ipcs -q：只查看消息队列（Message Queue）。
    • ipcs -s：只查看信号灯集（Semaphores）。
  2. 手动物理强行清除残留的 IPC 资源：
    • ipcrm -m <shmid>：删除指定的共享内存 ID。
    • ipcrm -q <msgid>：删除指定的消息队列 ID。
    • ipcrm -s <semid>：删除指定的信号灯集 ID。
