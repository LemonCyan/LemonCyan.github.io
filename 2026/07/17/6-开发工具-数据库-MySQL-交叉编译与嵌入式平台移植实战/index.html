MySQL 交叉编译与嵌入式平台移植实战

MySQL 交叉编译与嵌入式平台移植实战

在物联网、工业网关以及智能车载终端等嵌入式设备开发中，我们有时需要在本地（如 ARM/ARM64 硬件开发板上）直接部署 MySQL 数据库，以满足大容量本地缓存和高效率关系型数据查询的业务需求。然而，由于 MySQL 本身逻辑极其庞大、构建期高度依赖 host（主机）本地的可执行程序来进行代码生成，且官方源码默认不支持平滑的交叉编译，这使得在 ARM 平台下交叉编译 MySQL 成为嵌入式开发中的一个硬骨头。

今天这篇实战指南，咱们就来完整拆解 MySQL 5.1.51 (基于 Autoconf) 以及 MySQL 8.0.22 (基于 CMake) 两代经典版本的交叉编译流程，同时分享我们在实际项目中的 6 大经典报错避坑排障套路，并附带生产级的多平台数据库高效迁移机制。

一、 嵌入式移植核心关注点

在动手编译之前，咱们作为架构师和底层开发人员，必须先搞明白交叉编译 MySQL 的几大核心“阻力”：

1.1 构建期二进制程序的宿主执行瓶颈

In MySQL 的编译过程中，编译器会实时生成并运行一些代码生成工具，用于动态生成 SQL 语法解析文件和字符集映射文件。这些工具包括：

  • gen_lex_hash：生成 lex 词法分析所需的哈希表表头。
  • uca9dump：转储和编译 Unicode 排序及映射字符集数据。
  • comp_sql：将系统初始化 SQL 脚本（如权限建表脚本）编译压缩成 C 语言数组常量。
  • gen_lex_token：根据词法生成解析 Token 标识符定义。

当进行交叉编译时，这些工具会被交叉编译器编译成目标板格式（如 ARM/AArch64）。当编译脚本在我们的 PC 开发机（如 x86_64 Ubuntu）上尝试执行 these 程序时，系统就会抛出 cannot execute binary file: Exec format error 报错导致构建崩溃。因此，我们必须采用“双路线构建”策略——先编译 PC 版获取原生辅助工具，再中途强行覆盖 ARM 构建目录中的对应程序。

1.2 ncurses 终端控制库依赖

MySQL 命令行客户端（即 mysql 终端工具）在运行和交互时，其控制台界面和交互输入底层完全依赖于 ncurses 库（终端绘图及字符处理库）。在交叉编译器环境中，默认只带了基础的 GCC/libc，缺少目标架构（ARM）下的 ncurses 静态库和头文件。因此在交叉编译 MySQL 前，必须先手动下载并交叉编译出目标架构的 libncurses.a 静态库，并将其路径显式传递给 MySQL 编译配置中。

1.3 静态链接库与密码库关联

现代 MySQL 8.0.x 版本在安全验证、SSL 通讯和 SASL 认证方面依赖 OpenSSL 和相关加密组件，在数据结构和基础库方面依赖 C++ Boost。在配置交叉编译时，必须理清这些外部依赖库在 ARM 平台下的对应物理路径。

4.3 编译中 6 大经典报错防线破局之策

在执行 make 后，你会遭遇以下一连串经典报错阻力，请严格按照以下对策进行破局：

报错 1：CMakelists.txt:1374 处 patchelf 校验报错

  • 原因：交叉编译脚本会检查宿主机系统上是否安装了用于动态库路径修改的 patchelf 工具。
  • 对策：如果本地开发机没有，可以直接去 CMakeLists.txt 的 1374 行附近，将 patchelf 的条件分支判断代码段直接注释掉。

报错 2：LIBEVENT version must be at least 2.1, found …

  • 原因：CMake 的检查宏对于交叉编译下的 libevent 版本判定有误，虽然 MySQL 源码自带了 2.1.11 的库，但依然报错终止。
  • 对策：进入相关 CMake 检测脚本（如 cmake/libevent.cmake），将检查版本抛出错误的那几行 message(FATAL_ERROR ...) 强行注释掉。

报错 3：cannot execute binary file: Exec format error (uca9dump 等)

  • 对策：

  • 原因：由于线程 ID 结构在不同 CPU 架构下的位宽不同，MySQL 底层调用的比较交换函数发生了签名冲突。
  • 对策：用全局搜索该关键字，在源码中将该函数的调用全部修改为通用的底层大整数比较交换函数 os_compare_and_swap_lint。

  • .frm：存储表结构定义。
  • .MYD：存储表内真实数据（Data）。

将这些文件拷贝到目标板的 /var/lib/mysql/db_name/ 目录下后，直接赋予 mysql 用户权限即可立刻使用，不需要任何 SQL 还原。
