Linux 内核虚拟内存空间管理与地址映射内核指南

Linux 内核虚拟内存空间管理与地址映射内核指南

在嵌入式 Linux 和驱动开发的世界里，仅仅停留在应用层 C 语言的 malloc 和 free 是远远不够的。当你的代码运行在内核态，或者当你需要手搓网卡、显卡等设备驱动时，你打交道的将是物理地址、MMU 页表、内核虚拟空间划归，以及外设 I/O 寄存器映射。

本指南将带你潜入 Linux 内核的物理底层，从内核用于描述虚拟空间的结构体（VMA）开始，梳理 ELF 可执行文件加载映射的全生命周期，深度拆解物理地址与虚拟地址转换 of 硬件原理，并详解驱动开发中 ioremap 物理外设寄存器映射的底层细节。

第一部分：内核虚拟内存区的核心抽象（VMA）

Linux 内核并不能直接利用页表去感知识别进程中各个段（如代码段、数据段）的功能差异。为了描述 and 管理进程用户空间中各个不同属性的虚拟内存区域，内核引入了 VMA（Virtual Memory Area，虚拟内存区） 这一高级抽象。

1.3 链表与红黑树的双轨制管理拓扑

为了兼顾“顺序遍历”和“快速随机检索”的要求，内核对进程中所有的 VMA 采用了双轨制管理架构：

  • **单向链表 (mm->mmap)**：所有的 VMA 按照虚拟内存地址由低向高排序，以单向链表形式链接。当需要遍历进程中所有的虚拟内存段（如程序退出注销空间）时，使用链表遍历非常高效。
  • **红黑树 (mm->mm_rb)**：所有 VMA 以虚拟地址作为 Key，挂载在红黑树上。当进程发生缺页中断（Page Fault）或者执行 mmap 分配新地址时，内核必须以极快的速度判定当前引发中断的虚拟地址落在哪个 VMA 区间内。红黑树能够将查找时间复杂度由 $O(n)$ 降为 $O(\log n)$，极大提升了内核的响应效率。

1.4 VMA 的合并与分裂（Merging & Splitting）

在进程运行过程中，虚拟内存区域并不是一成不变的。当发生 mmap()、munmap() 或者 mprotect() 时，内核会动态调整 VMA 节点：

  1. **VMA 合并 (VMA Merging)**：
    当新创建的虚拟内存段与相邻 VMA 在物理上连续，且它们的页保护权限、文件映射属性、标志位（vm_flags）完全一致时，内核会自动调用 vma_merge() 将它们合并为一个单一的 VMA 节点，以减少 VMA 节点的总数，节约内核内存开销。
  2. **VMA 分裂 (VMA Splitting)**：
    当调用 munmap() 释放某个现有 VMA 的中间某一部分，或者调用 mprotect() 强行修改 VMA 中间区域的读写权限时，内核会自动调用 split_vma()。这会将原有的一个 VMA 节点一拆为二或一拆为三，分别赋予它们不同的权限和范围。

第三部分：虚拟地址到物理地址的映射机制 (MMU & Paging)

进程在运行时访问的全部是虚拟地址。当 CPU 执行一条读写内存的指令时，它是如何将这个虚拟地址定位到真实的 DDR3/DDR4 物理内存条上的？

3.1 MMU 硬件与多级页表寻址机制

这一转换是由 CPU 内部的硬件单元 MMU（Memory Management Unit，内存管理单元） 实时完成的。
MMU 使用页表（Page Table）来维护虚拟页面（Page）到物理页帧（Page Frame，大小一般为 4KB）的映射关系。

在 32 位系统下，最经典的是二级页表映射机制。一个 32 位的虚拟地址被拆分为三部分进行寻址转换：

🔑 寻址流程：

  1. 一级寻址：CPU 控制寄存器 CR3 存放着当前运行进程的一级页表（页目录表）的物理基地址。MMU 以虚拟地址的最高 10 位作为索引，在页目录表中寻找到对应页表的物理基地址。
  2. 二级寻址：MMU 以虚拟地址的中间 10 位作为索引，在刚刚找到的页表中寻找到具体物理页帧的首地址。
  3. 最终寻址：物理页帧首地址加上虚拟地址的最低 12 位（即页内偏移量，最大对应 $2^{12} = 4\text{KB}$ 空间），就定位到了 DDR 内存条上的具体物理字节。

3.2 页表项（PTE）核心标志位功能表

MMU 不仅负责地址转换，还负责内存权限的安全拦截。页表项（Page Table Entry）中包含多位硬件控制标志：

标志位名称 标志位缩写 作用描述
Present (存在位) P 值为 1 代表该页已分配物理内存；值为 0 触发硬件缺页中断。
Read/Write (读写位) R/W 值为 1 代表可读可写；值为 0 代表只读（修改它触发页面写保护异常）。
User/Supervisor (用户位) U/S 值为 1 代表用户态程序可以访问；值为 0 代表仅限内核态访问。
Accessed (访问位) A 只要该物理页被读或写过，硬件自动置 1，用于内核页面置换算法（LRU）。
Dirty (脏位) D 只有该物理页被写入数据时，硬件自动置 1，提示换出时必须写回硬盘。
Global (全局位) G 值为 1 代表该页表项在 TLB 缓存中全局有效，进程切换时不被清空。

3.3 缺页中断（Page Fault）原理解密

在虚拟内存建立之初，系统并没有立刻为进程分配等大的物理内存。
当进程尝试读写一个尚未分配物理页帧的虚拟地址时，硬件 MMU 发现对应的页表项（PTE）的 Present 标志位为 0（代表没有对应的物理内存映射）。

这会立刻触发一个硬件异常——缺页中断（Page Fault）：

  1. CPU 暂停当前指令执行，保存当前现场，跳转到内核的缺页中断异常向量入口。
  2. 内核的异常处理程序（do_page_fault()）会被唤醒，它会读取 CR2 寄存器获取发生缺页的虚拟地址。
  3. 内核在进程的 mm_struct 红黑树中快速查找，判定该虚拟地址是否落在合法的 VMA 区间内。
    • 如果不合法：说明是野指针越界乱写，直接向进程发送 SIGSEGV 信号（段错误崩溃）。
    • 如果合法：说明是正常的懒分配。内核调用物理页分配器（如伙伴系统）申请一个 4KB 的物理页帧，将其物理首地址写入对应的页表项中，并将 Present 位置为 1。
  4. 异常处理程序退出，CPU 重新执行刚才引发缺页的那条机器指令。此时映射已建立，程序可以无缝继续运行。

🔑 TLB 缓存与多核同步 (TLB Shootdown)

为了提速，CPU 内部有一块极速的页表项缓存——**TLB (Translation Lookaside Buffer，旁路转换缓冲)**。
当多核 CPU 架构下，内核在某个核上释放或修改了页表映射（例如通过 munmap 或进程销毁），它必须立刻调用 **TLB 击落 (TLB Shootdown)**：通过核间中断（IPI，Inter-Processor Interrupt）通知其他所有 CPU 核心 invalidate 它们各自本地 TLB 中的过期条目，彻底杜绝多核内存数据不一致性。

对于想深入内核底层逻辑的极客，我们需要沿着 Linux 内核中 do_page_fault() 的执行路径一探究竟。以下是基于架构相关与独立层的核心伪代码分析：

在 handle_mm_fault() 内部，内核会沿着虚拟地址的高位，一级一级向下查找并自动补齐缺少的页表项：

  • PGD 级：pgd_offset(mm, address) 找到页全局目录条目。
  • PUD 级：pud_alloc(mm, pgd, address) 找到页上级目录条目。
  • PMD 级：pmd_alloc(mm, pud, address) 找到页中级目录条目。
  • PTE 级：pte_alloc_map(mm, pmd, address) 最终映射分配到具体的页表项上。

一旦定位到 PTE，内核会触发 handle_pte_fault()：

  • 如果该地址属于 写时复制 (Copy-On-Write, COW) 共享页，则调用 do_wp_page() 分配新物理页并把数据复制过去，打破共享。
  • 如果是匿名映射页（malloc 分配），直接调用 do_anonymous_page() 分配物理内存并全部清零。
  • 如果是文件映射页（mmap 读取文件），则通过调用 vma->vm_ops->fault 驱动接口去从磁盘同步读取数据到物理页中。

第四部分：内核空间映射与高端内存（High Memory）机制

在 Linux 系统中，1GB 的内核虚拟地址空间（0xC0000000 到 0xFFFFFFFF）是如何映射到物理内存上的？这是一个经典的底层开发痛点。

4.1 3G/1G 内存分割下的 896MB 局限性

如果系统中的物理内存（RAM）只有 512MB，内核可以直接采取“简单直接映射”的方式：把内核虚拟地址减去 0xC0000000 就可以得到对应的物理内存地址。
$$\text{物理地址} = \text{内核虚拟地址} - 0\text{xC0000000}$$
这非常高效，连复杂的二级页表转换都可以省掉，可以直接通过一级线性映射访问。

但是，如果系统的物理 RAM 大于 1GB，甚至是 4GB 呢？
由于内核虚拟地址空间只有 1GB，采用这种直接一对一映射，内核最大只能映射 1GB 的物理内存。剩余 of 3GB 物理内存，内核虚拟空间根本没有多余的地址去代表它们，导致这部分物理内存直接“失联”！

为了解决这个尴尬的问题，Linux 内核在 32 位系统下划分了高端内存（High Memory）机制。

4.2 划江而治：低端内存与高端内存的页表排布

Linux 内核将这 1GB 的内核虚拟空间强行切分为两块：

  • 低端内存（Low Memory，直接映射区）：占低端的 896MB 虚拟空间（0xC0000000 到 0xF7FFFFFF）。这部分空间与物理内存的 0 到 896MB 区间进行一对一直接线性映射。
  • 高端内存（High Memory，动态映射区）：占高端的 128MB 虚拟空间（0xF8000000 到 0xFFFFFFFF）。

当内核需要访问物理内存中大于 896MB 的区域时，有三种映射手段：

  1. vmalloc 映射区：
    vmalloc() 函数用于在内核中申请虚拟地址连续、但物理地址不连续的大块内存。它会在 0xF8000000 到 PKMAP_BASE 之间的虚拟区间分配 VMA，并通过修改页表把分散的物理页拼凑到一起。与之相对，kmalloc() 申请的内存物理和虚拟都连续，但最大大小受限。
  2. **永久内核映射 (kmap)**：
    通过 kmap(page)。它会借用 PKMAP_BASE 到 FIXADDR_START 之间的页表，将高端页面建立长期映射。可能会发生上下文阻塞，只能在非中断上下文使用。
  3. **临时/原子内核映射 (kmap_atomic)**：
    在中断处理函数等不能睡眠的场景下，通过 kmap_atomic(page)。它直接使用一组预留的固定映射（Fixmap）专用页表插槽，强行覆盖该 CPU 核的寄存器。分配速度极快、不阻塞，是驱动中处理高端内存的最爱。

第五部分：物理页物理内存分配算法（Buddy & Slab）

独立于虚拟地址映射之外，Linux 内核必须有一套机制来实际管理和分配有限的物理 RAM 页帧。这就是内核底层的伙伴系统与 Slab 分配器。

5.1 伙伴系统（Buddy System）算法

Linux 内核将所有的空闲物理页帧（Page Frames）交由伙伴系统进行分配与回收。

  • 工作机制：伙伴系统将空闲页面按照大小分为 11 个“阶”（Order，从 0 到 10）。阶为 0 代表单独的 1 个空闲页（4KB），阶为 1 代表连续的 2 个页（8KB），阶为 10 代表连续的 $2^{10} = 1024$ 个页（4MB）。
  • 分裂逻辑：当内核请求分配一个阶为 0 的页面时，如果阶为 0 的链表中没有空闲页，系统会寻找阶为 1 的链表。如果阶为 1 有空闲块，内核会将其一分为二（这两个半块互为“伙伴”），将前半部分分配出去，后半部分挂入阶为 0 的链表。
  • 合并逻辑：当释放一个物理页面时，系统会检查其相邻的“伙伴”页面是否也处于空闲状态。如果是，则将它们合并为一个更大的块，并向上挂载到高阶链表中，有效抑制了物理内存碎片。

5.2 Slab/Slub 分配器机制

虽然伙伴系统能很好地防止内存碎片的产生，但它分配物理内存的最小单位是“页（4KB）”。在内核中，我们经常需要分配几十字节或几百字节的小对象结构体（如 vm_area_struct 结构体只有 200 多个字节）。如果每次都找伙伴系统要一整页，会造成极大的内存浪费。

为了解决这个问题，内核在伙伴系统之上构建了 Slab（或 Slub）分配器：

  • 核心逻辑：Slab 会预先向伙伴系统索要整页内存，并将其精细切割成固定大小的小块（称为对象，Objects）进行缓存。
  • 状态维护：Slab 链表包含三种状态：
    1. Full（满）：Slab 中的所有小对象均已被内核模块分配使用。
    2. Partial（部分空闲）：部分小对象空闲，部分正在使用（分配时优先从此类 Slab 获取）。
    3. Empty（完全空闲）：里面的小对象全部空闲。当系统内存吃紧时，此类 Slab 占用的整物理页会被退还给伙伴系统。
      内核中的 kmalloc() 底层正是完全依赖于 Slab 分配器来完成快速小内存分配的。

第六部分：物理外设寄存器驱动开发与安全访问

在嵌入式开发中，SoC 外设（如 GPIO、I2C、串口、显卡）的物理控制寄存器都映射在特定的物理地址空间中。由于 Linux 运行在保护模式下，内核和应用层都无法直接读写物理地址，必须通过 ioremap 机制将其映射到内核虚拟空间。

6.1 内存映射 I/O (MMIO) 与 Port I/O 对比

  • **MMIO (Memory-Mapped I/O)**：外设寄存器的物理地址与内存条统一编址。访问外设就像读写普通的 DDR 内存单元一样，使用 readl()/writel() 等标准内存操作指令。ARM 和 MIPS 架构全系采用此方案。
  • **Port I/O (PIO)**：外设寄存器拥有独立的地址空间。必须使用特殊的 CPU 指令（如 x86 下的 in / out）进行访问。

6.2 硬件安全：为什么禁止通过指针直接读写 I/O 寄存器

在一些裸机开发中，程序员习惯直接通过强转指针解引用来修改寄存器（如 *(volatile unsigned int *)addr = val;）。但在复杂的 Linux 驱动中，这是绝对禁止的！必须使用内核封装好的 readl() / writel() 系列函数。

核心防崩防错原理解密：

  1. 编译器指令重排拦截：
    编译器在做 Release 编译优化时，会尝试将多条相邻的赋值指令进行合并或重新排序以提速。然而，外设寄存器的读写顺序代表着硬件状态机的跳转。直接解引用指针会导致优化后的写入顺序被打乱，引发芯片死锁。readl/writel 内部织入了内存屏障（Memory Barrier，如 ARM 下的 dsb/dmb 指令），强行拦截并禁止了编译器和 CPU 乱序执行。
  2. CPU 缓存绕行（Non-cacheable）：
    普通 RAM 内存读写会被 CPU 缓存（L1/L2 Cache）拦截以提速。但外设寄存器的数据是硬件实时改变的。如果读写被 Cache 拦截，CPU 读取到的将一直是缓存中的旧数据，写入的数据也无法实时送达芯片物理引脚。ioremap 映射出来的页表会被自动打上 Non-cacheable 标志，告知 MMU 必须绕过一切 Cache 直接发起物理总线读写。

第七部分：物理外设映射的高级话题：IOMMU 与 DMA 内存管理

在高性能计算中，外设需要越过 CPU 直接向物理 RAM 写入数据。这就是 **DMA (Direct Memory Access，直接内存访问)**。

7.1 DMA 的物理限制与内存一致性问题

  • 物理连续性限制：由于 DMA 控制器并不通过 CPU 内部的 MMU 译码，它只能识别连续的物理地址。如果驱动需要传输一个大缓冲区，而物理内存碎片化，分配连续的物理页面会极度困难。
  • Cache 一致性：CPU 读取内存是走 L1/L2 Cache 的。如果外设 DMA 默默把新的电信号写入了 DDR 物理内存，而没有通知 CPU，CPU 接下来继续从 Cache 中读取过期数据，会导致严重的业务逻辑混乱。
  • **一致性 DMA 分配 (dma_alloc_coherent())**：内核提供了该 API，能一步到位分配连续物理内存，并在页表中打上非缓存标志，彻底解决 Cache 一致性矛盾。
  • **分散-聚集 DMA 映射 (dma_map_sg())**：当物理页面散布在 RAM 各处（散列表形式 struct scatterlist）时，该接口可以通过配置 Scatter-Gather 链表将多个物理不连续的物理片一次性提交给硬件 DMA 搬运，避免了数据多次拷贝。

7.2 IOMMU 硬件虚拟化架构

为了解决 DMA 控制器必须使用连续物理内存的限制，现代芯片组引入了 **IOMMU (Input-Output Memory Management Unit，输入输出内存管理单元)**：

  • 地址转换：类似于 CPU 的 MMU 将虚拟地址转换为物理地址，IOMMU 负责将外设发出的 IOVA (I/O Virtual Address) 翻译为真正的物理地址。
  • 突破物理连续局限：有了 IOMMU，外设设备可以发起对非连续物理页帧的 DMA 传输。在设备看来，它读写的是一块连续的虚空间（IOVA），而 IOMMU 在中途自动把请求分发到散布在 DDR 内存条上的各个 Page Frame 中。
  • **安全防线 (DMA Isolation)**：IOMMU 还能对设备权限进行约束。一个越界写数据的 Bug 驱动程序，会被 IOMMU 强行拦截，阻止其非法写入其他内核进程的敏感内存页。

第八部分：内核内存性能调优：巨页与 OOM 机制

在生产环境的内核调优中，内存性能对应用吞吐量有着决定性的影响。

8.1 巨页（Huge Pages）降低 TLB Miss

默认的 4KB 页大小在面对 TB 级内存服务器时会遇到瓶颈。4GB 空间需要上百万个页表项，会导致 CPU 的 TLB 缓存频繁未命中（TLB Miss），大量 CPU 周期被空耗在多级页表寻址（Page Table Walk）中。

  • 内核巨页支持：Linux 内核支持 HugeTLB 机制，允许分配 2MB 甚至是 1GB 的巨型物理页。这能够将页表数量缩减几百倍，将 TLB 覆盖率提升几个数量级，数据库读写吞吐量可因此直接飞升 15%~20%。
  • **配置透明巨页 (THP, Transparent Huge Pages)**：
    可以通过 /sys/kernel/mm/transparent_hugepage/enabled 进行动态配置。

8.2 终极大杀器：OOM (Out Of Memory) Killer 调优

当物理内存耗尽，且 Swap 交换分区也被完全压垮时，为了防止整机内核崩溃挂起，Linux 内核会启动 OOM Killer（内存溢出杀手）。

1. 评分算法

OOM Killer 会对系统中的所有进程进行打分评级（得分保存在 /proc/<pid>/oom_score 中）。分值越大，代表该进程占用内存越多、运行时间较短、权重较低，越容易被内核选定并直接发送 SIGKILL 强行击杀！

第九部分：内核内存回收与 Swap 机制

当物理内存吃紧但还没有达到 OOM 爆发边缘时，内核会通过页面回收（Page Reclamation）机制尝试释放一些物理内存。

9.1 kswapd 内核守护线程与水位线 (Watermarks)

每个内存区域（Zone）都设有三条水位线：High（高水位）、Low（低水位）、Min（最小水位）。

  1. kswapd 唤醒：当空闲物理内存降至 Low 水位线以下时，系统会自动唤醒内核线程 kswapd。它在后台异步回收页面，直到空闲物理页回升到 High 水位以上。
  2. **直接内存回收 (Direct Reclaim)**：若进程申请内存速度过快，空闲内存直接跌穿了最低防线 Min 水位，此时进程分配会被强行卡住挂起，直接参与回收（Direct Reclaim），直到凑齐物理页。

9.2 LRU 链表与页面换出控制（Swappiness）

内核通过最近最少使用（LRU, Least Recently Used）链表来跟踪物理页面的活跃度：

  • LRU 分类：分为活跃链表（Active List）与非活跃链表（Inactive List）。冷页面会慢慢从活跃链表沉降到非活跃链表，并被内核优先挑选进行释放回收。
  • Swappiness 控制参数：我们可以通过修改 /proc/sys/vm/swappiness（取值为 0 至 100，默认通常为 60）控制页面的回收策略。
    • Swappiness = 0：最大限度使用物理内存，非必要绝不将匿名内存换出到 Swap 分区。
    • Swappiness = 100：极其激进地倾向于将冷匿名页面写入 Swap 交换分区，以腾出更多物理 RAM 空间缓存热点文件。
