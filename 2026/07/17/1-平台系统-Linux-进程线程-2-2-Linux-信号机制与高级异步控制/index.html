Linux 信号机制与高级异步控制

Linux 信号机制与高级异步控制

信号（Signal）是 UNIX/Linux 操作系统中用于处理异步事件的首要工具。在日常开发中，简单的信号捕获往往能应付基础需求，但在多进程高并发、系统级基础架构开发中，我们必须面对信号的未决、阻塞屏蔽、实时可靠信号队列、多信号并发优先级，以及被信号中断的系统调用重启等高阶问题。本文将全面深挖 Linux 信号机制的生命周期状态机、sigaction() 工业级 API 细节，并提供两个经典的多进程异步信号同步案例进行实战。

一、 信号的生命周期与底层状态机

在 Linux 系统中，每一个进程的控制块 task_struct 内部都维护着两个非常关键的 64 位位图（Bitmap）：未决信号集（Pending） 和 阻塞信号集（Blocked，也称信号屏蔽字）。

一个信号的生命周期通常经历以下三个物理阶段：

  1. 产生阶段（Generated）：
    • 硬件异常：如除以 0 导致 CPU 抛出 SIGFPE，非法内存访问（野指针）导致 SIGSEGV，终端按下 Ctrl+C 发送 SIGINT。
    • 软件触发：进程调用 kill()、raise()、alarm()、sigqueue()，或者内核因为管道断开抛出 SIGPIPE。
  2. 未决阶段（Pending）：
    • 信号已经被产生，但内核尚未传递给目标进程的这段时间。此时内核会将进程 task_struct 中 Pending 位图的对应位置为 1。
  3. 递达阶段（Delivered）：
    • 当进程被调度运行，且该信号在 Blocked 位图中没有被阻塞屏蔽（对应位为 0）时，内核会清除 Pending 位图中的对应位，并强制进程暂停当前执行流，去响应信号。

🛡️ 进程对信号的 3 种处理动作：

  • 执行系统默认动作（Default）：大多数信号的默认动作是终止进程并可能产生 Core Dump。
  • 忽略该信号（Ignore）：丢弃该信号，进程不采取任何动作（注：SIGKILL 和 SIGSTOP 绝对不能被忽略或捕获，它们是内核杀进程的底牌）。
  • 捕获并处理（Catch）：用户自定义一个回调函数，当信号递达时自动跳转执行该函数。

二、 信号的全景分类：可靠信号与不可靠信号

在终端输入 kill -l 可以查看系统支持的所有信号。Linux 系统移除了 16 号和 32 号信号，形成了 1-31 号的非实时（不可靠）信号和 34-64 号的实时（可靠）信号两大阵营：

1. 不可靠信号（1 ~ 31 号）

  • 信号可能丢失：如果进程阻塞了某个不可靠信号，在其解除阻塞之前，这个信号被连续发送了多次，内核只会在 Pending 位图中记录一次，其余的全部丢失。
  • 早期的 signal() 函数在信号被触发后，内核会自动将该信号的处理动作恢复为系统默认动作。如果要持续捕获，必须在信号处理函数开头重新注册，这会导致短暂的“竞态窗口”，容易在连续来信号时发生崩溃。

2. 可靠信号（34 ~ 64 号）

  • 支持排队：可靠信号也称实时信号（Real-time Signal）。如果一个可靠信号被阻塞并连续发送多次，内核会将其放入一个未决信号队列中进行排队，解除阻塞后，信号会按照发送的顺序依次被全部递送，一个也不会丢失。

3. 多个未决信号递送的优先级

如果一个进程同时有多个不同的未决信号等待递送：

  • 内核按照信号值从小到大的顺序依次递送（值越小优先级越高，如 2 号 SIGINT 优先于 10 号 SIGUSR1 递送）。
  • 对于同一类型实时信号，按照发送的时间顺序递送。
  • 如果不可靠信号与可靠信号同时处于未决状态，Linux 通常会优先递送不可靠信号。

三、 内核原理解析：信号拦截与用户栈跳转

信号的响应和处理有着独特的物理路径。进程在内核态运行时（如执行系统调用、处理硬件中断），软中断信号并不会立即起作用，必须等到检测时机到达：

🔍 内核检测与执行信号的处理时机

  1. 从内核态即将返回用户态前夕。
  2. 进程在内核中进入睡眠并即将被唤醒的时刻。

🔄 用户栈上“新建栈帧”的跳转魔术

信号处理程序是运行在用户空间的。为了确保安全，内核绝对不允许用户自定义处理函数在内核态下执行。因此，内核采用了一种巧妙的“跳转拦截”机制：

  1. 当进程从内核态准备返回用户态时，内核检测到有未决的信号需要捕获，它会在该进程的用户栈（User Stack）上强行构建一个新的临时栈帧。
  2. 内核将这个新栈帧的返回地址（PC）改写为用户自定义的信号处理函数首地址，并将原先即将返回的用户代码地址保存在该栈帧中。
  3. 进程返回用户空间时，由于栈顶返回地址已被篡改，执行流会自动跳转执行信号处理函数。
  4. 信号处理函数执行完毕后，执行临时的 sigreturn 系统调用重新陷入内核，由内核清理掉临时栈帧，再次将执行流恢复到最开始被中断的用户代码处继续运行。

四、 sigaction() 现代规范与信号集（sigset_t）操作

为了解决旧版 signal() 的硬伤，POSIX 规范引入了现代的 sigaction() 系统调用。

🛠️ 1. sigaction() API 原型

  • signum：要捕获的信号编号（除 SIGKILL 和 SIGSTOP 外）。
  • act：新的信号处理动作配置结构体。
  • oldact：传出参数，用于保存该信号之前的处理配置。

🔍 struct sigaction 核心结构体配置

  1. sa_mask（临时信号集）：在执行当前信号处理函数时，内核会自动将 sa_mask 集合中的所有信号加入进程的 Blocked 屏蔽字中。处理函数返回后，屏蔽字自动恢复。当前处理信号本身默认会被自动加入屏蔽字，防止同种信号重入。
  2. sa_flags（标志位）：
    • SA_RESTART：被信号中断的系统调用（如 read / write）在处理函数返回后会自动重启，防止程序抛出 EINTR 错误。
    • SA_RESETHAND（等同于旧的 SA_ONESHOT）：调用处理函数前，将该信号恢复为默认处理方式。
    • SA_NODEFER（等同于 SA_NOMASK）：处理该信号时不阻塞该信号的再次到来。
    • SA_SIGINFO：启用高级信号处理，使用 sa_sigaction 回调，可利用 siginfo_t 提取发送端 PID 等。

🛠️ 4. 发送与控制辅助函数

  • int kill(pid_t pid, int sig);：通过 PID 发送信号。
  • int raise(int sig);：向当前进程自身发送信号。
  • unsigned int alarm(unsigned int seconds);：定时器，超时向当前进程发送 SIGALRM。
  • int pause(void);：挂起进程等待任意信号，直到信号处理函数执行完毕后返回 -1 且 errno = EINTR。若信号默认动作是终止，则程序直接退出。若信号被忽略，pause 不会被唤醒，将继续睡眠。

五、 实战演练一：客车司机与售票员同步控制

下面我们编写一个经典的多进程异步信号协作实验：

  • 父进程模拟客车司机，子进程模拟售票员。
  • 当售票员关闭车门后，子进程向父进程发送 SIGUSR1 信号，唤醒司机启动客车。
  • 当司机到达目的地停车后，父进程向子进程发送 SIGUSR2 信号，唤醒售票员开门。
  • 当售票员处理完乘客下车后，向司机发送 SIGINT 信号，两者安全退出。

六、 实战演练二：SIGCHLD 异步自动收尸与信号屏蔽监测

在多进程后台任务设计中，如果不想让子进程退出后变成僵尸进程，最好的办法是捕获 SIGCHLD，并在其中调用 waitpid 来异步回收僵尸子进程，这不占用主程序的正常执行流程。
