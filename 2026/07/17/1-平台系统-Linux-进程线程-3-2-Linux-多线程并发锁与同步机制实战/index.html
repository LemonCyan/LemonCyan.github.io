Linux 多线程并发锁与同步机制实战

多线程最大的优势是共享进程的地址空间，这使得数据传递极其简单，指针随手一扔就能传数据。但这也带来了巨大的安全挑战：多线程并发竞态（Race Condition）。当多个线程在无保护状态下同时读写同一份共享数据时，就会引发脏数据和不可预期的运行结果。本文咱们就用实战代码复现并发 Bug，然后手撕 Linux 的并发控制锁机制。

一、 进程同步与临界区互斥的经典理论

在深入具体的 Linux 并发编程 API 之前，我们有必要先建立起操作系统关于“进程同步”的底层学术理论体系。

1. 临界资源与代码的四个执行区

在并发系统中，虽然多数资源可以共享，但有些资源一次只容许一个进程/线程使用，这类资源被称为临界资源（Critical Resource）（如打印机、共享的全局队列等）。
为了安全、互斥地访问临界资源，进程在逻辑上访问临界资源的代码被划分为四个区域：

  1. 进入区（Entry Section）：在进入临界区之前，进程要检查是否可以安全进入。若能进入，则设置“正在访问”标志（锁上锁），阻止其他进程进入。
  2. 临界区（Critical Section）：进程中真正执行访问临界资源、修改共享数据的代码段。
  3. 退出区（Exit Section）：临界区执行完毕后，负责清除占用标志（锁解锁），唤醒其他等待的进程。
  4. 剩余区（Remainder Section）：与访问临界资源无关的其他代码段。

2. 临界区互斥的硬件实现方法

在底层，为了防止上下文切换打断临界区的原子操作，硬件层面提供了两种最基本的低级互斥解法：

① 中断屏蔽方法

在进程准备进入临界区前，执行关中断指令（Disable Interrupts）；退出临界区后，再执行开中断指令（Enable Interrupts）。

  • 优缺点：极其简单。但由于关中断会屏蔽一切硬件事件（如磁盘读写、时钟中断），如果在临界区发生死循环或耗时任务，系统会彻底卡死。因此，关中断仅供操作系统内核级操作使用，用户态程序绝不能使用。

② 硬件原子指令法

现代 CPU 提供了特殊的硬件原语指令，它们能保证“读取-修改-写回”内存的操作是一口气原子完成的，不受任何多核或上下文切换打断。最著名的指令是 TestAndSet 和 Swap。

4. 管程（Monitor）的概念

由于手写信号量的 P/V 操作过于繁琐，且极易因为漏写 V 操作或者写错顺序导致死锁，计算机科学家发明了管程（Monitor）。

  • 定义：管程是一个特殊的软件模块，它把共享资源的数据结构以及对该数据结构进行操作的一组过程（函数）封装在一起。
  • 核心特性：
    1. 管程内部的共享数据只能通过管程内部的过程访问。
    2. 管程保证互斥是编译器自动处理的：同一时刻，只允许一个进程/线程在管程内部执行某个过程。如果另一个线程企图调用管程函数，它会被编译器自动生成的互斥代码阻断在管程外等待。
    3. 通过引入条件变量（Condition Variable）来实现管程内部的同步等待。

三、 互斥锁 (Mutex)：实现独占与排他

互斥锁（Mutex）是最基础的同步机制。它只有两个状态：锁定（Locked）和未锁定（Unlocked）。一旦一个线程拿到了互斥锁，其他试图获取锁的线程就会被阻塞挂起。

🛠️ 常用 API

  • 静态初始化：pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
  • 动态初始化：int pthread_mutex_init(pthread_mutex_t *mutex, const pthread_mutexattr_t *attr);
  • 加锁（阻塞）：int pthread_mutex_lock(pthread_mutex_t *mutex);
  • 加锁（非阻塞）：int pthread_mutex_trylock(pthread_mutex_t *mutex); —— 如果锁已被他人占用，立即返回 EBUSY，不挂起线程。
  • 解锁：int pthread_mutex_unlock(pthread_mutex_t *mutex);
  • 销毁锁：int pthread_mutex_destroy(pthread_mutex_t *mutex);

四、 死锁（Deadlock）的产生、判定与规避策略

当线程在执行过程中，因争夺资源而造成的一种互相等待的现象，若无外力作用，它们都将无法推进下去，这就是死锁。

1. 产生死锁的 4 个必要条件：

死锁的发生必须同时满足以下 4 个物理条件，破坏其中任意一个即可预防死锁：

  1. 互斥条件：一个资源在同一时刻只能被一个线程独占占用。
  2. 请求与保持条件：一个线程因请求新资源而发生阻塞时，对已获得的资源保持占用不放。
  3. 不剥夺条件：线程已获得的资源在未使用完之前，不能被其他线程强行剥夺，只能由占有线程自己释放。
  4. 循环等待条件：发生死锁时，必然存在一个线程-资源的环形链（如：线程 A 占有锁 1 等待锁 2，线程 B 占有锁 2 等待锁 1）。

2. 死锁的规避与处理策略

① 静态死锁预防（Prevention）

在程序运行前，通过严格的静态规则限制，破坏产生死锁的必要条件：

  • 破坏请求与保持条件：采用“一次性分配法”，要求线程在启动时必须一次性申请完全部所需的资源，否则不予运行。缺点是资源利用率极低。
  • 破坏循环等待条件：采用“资源顺序分配法”，给系统所有资源统一编号。规定线程必须严格按照编号从小到大的顺序去申请资源（例如必须先锁 1 锁，才能锁 2 锁）。这能彻底规避环形链的产生。

② 动态死锁避免（Avoidance）：银行家算法（Banker’s Algorithm）

在资源动态分配过程中，系统会首先通过算法模拟分配，检查系统是否会进入“不安全状态”。如果安全才予以实际分配，否则拒绝分配。

  • 数据结构定义：

    • Available[m]：长为 $m$ 的数组，表示当前 $m$ 种资源的可用物理实例数。
    • Max[n][m]：$n \times m$ 矩阵，定义每个进程对每种资源的最大需求量。
    • Allocation[n][m]：$n \times m$ 矩阵，定义当前已分配给每个进程的资源数。
    • Need[n][m]：$n \times m$ 矩阵，定义每个进程还需要的资源数（$\text{Need} = \text{Max} - \text{Allocation}$）。

  • 安全性检查算法（Safety Algorithm）：

    1. 设置临时工作向量 $\text{Work} = \text{Available}$，以及进程完成标志数组 $\text{Finish}[n] = \text{false}$。
    2. 在进程集合中寻找一个满足以下条件的进程 $P_i$：
      • Finish[i] == false
      • Need[i] <= Work（当前可用资源能满足其剩余最大需求）
        如果找到，执行步骤 3；否则执行步骤 4。
    3. 假设进程 $P_i$ 顺利运行完毕并释放资源，更新工作向量与状态：
      $$\text{Work} = \text{Work} + \text{Allocation}[i]$$
      $$\text{Finish}[i] = \text{true}$$
      返回步骤 2 继续检索下一个进程。
    4. 如果所有进程的 Finish 标志都成功变为了 true，则证明系统处于安全状态，模拟出的分配序列即为安全序列。否则，说明系统已进入不安全状态，容易诱发死锁，必须拒绝当前分配请求。

③ 死锁检测与资源分配图简化（Detection）

如果在程序运行中没有进行规避，可以通过资源分配图（Resource Allocation Graph）进行在线检测。

  • 结构：圆圈代表进程，方框代表资源。有向边“资源 $\to$ 进程”代表已分配；“进程 $\to$ 资源”代表正在申请。
  • 图的简化法：
    1. 找出一个非孤立的、且当前申请资源数小于对应资源空闲数的进程节点。
    2. 由于该进程可以顺利拿到资源并运行完，我们将其释放的边全部抹去（简化节点）。
    3. 重复此过程。如果最后能把图中所有的边全部消去，说明图是可完全简化的，此时系统无死锁。
    4. 如果最后图无法被完全简化（留有无法消除的环和边），则图不可简化，说明系统发生了死锁。

④ 死锁恢复技术（Recovery）

一旦检测到系统陷入死锁，必须通过外力手段强制解除：

  • 资源剥夺法：挂起某些死锁进程，强行抢占它们占有的资源分配给其他死锁进程。
  • 进程撤销法：强制终止一个或多个死锁进程（通常优先杀死消耗资源少、优先级低的进程），直到死锁环路断开。
  • 进程回退法：利用设置的检查点（Checkpoint）将死锁进程回退到之前的安全状态，并让其释放资源。

3. Pthread 死锁防范规避实战：

  • 保持锁的加锁顺序一致：所有线程在需要获取多个锁时，必须按照完全相同的顺序进行加锁。
  • **使用 pthread_mutex_trylock**：如果拿不到第二个锁，主动释放已占有的第一个锁，避免死等。

🎯 高阶用法：双锁交替（Ping-Pong）同步机制

我们还可以利用两个互斥锁在不同线程中交替解锁，来实现两个线程严格的交替运行。

五、 条件变量 (Condition Variable)：等待与唤醒机制

当线程需要等待某个特定的共享状态（例如：仓库不为空、队列有新任务）时，使用互斥锁不断进行轮询加锁会极度消耗 CPU 资源。条件变量允许线程在条件不满足时彻底挂起休眠，一旦条件具备，其他线程可以通过发信号的方式精确唤醒正在休眠的线程。条件变量必须与互斥锁配合使用！

🛠️ 常用 API

  • 等待条件：int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex);
    • 核心动作（必背）：
      1. 自动解锁传入的 mutex。
      2. 让线程进入休眠等待（这两步为原子操作，防止发生信号丢失）。
      3. 被唤醒时，在 pthread_cond_wait 返回前，自动重新加锁 mutex。
  • 单点唤醒：int pthread_cond_signal(pthread_cond_t *cond); —— 至少唤醒一个正在等待的线程。
  • 广播唤醒：int pthread_cond_broadcast(pthread_cond_t *cond); —— 唤醒所有正在等待该条件的线程。

六、 POSIX 信号量 (Semaphore)：资源计数器

信号量是一个非负整数计数器。它提供了经典的 PV 原子原语，我们可以利用它来实现线程同步或在有限缓冲区内的生产者-消费者队列。

🛠️ 常用 API

  • 初始化：int sem_init(sem_t *sem, int pshared, unsigned int value);
    • pshared：0 表示线程间共享，非 0 表示跨进程共享。
    • value：信号量的初始计数器值（可用资源数）。
  • **P 操作 (阻塞加锁)**：int sem_wait(sem_t *sem); —— 计数器减 1；如果计数器为 0，则线程进入阻塞休眠状态。
  • **P 操作 (非阻塞加锁)**：int sem_trywait(sem_t *sem); —— 计数器减 1；如果计数器为 0，不阻塞，直接返回错误码 EAGAIN。
  • **V 操作 (解锁释放)**：int sem_post(sem_t *sem); —— 计数器加 1，并唤醒阻塞等待的线程。
  • 销毁：int sem_destroy(sem_t *sem);

七、 读写锁 (Read-Write Lock)：读共享，写独占

读写锁适用于“多读少写”的高并发系统。只要没有线程加写锁，任意数量的读线程都可以同时持有读锁进行读取。

🛠️ 常用 API

  • 初始化：int pthread_rwlock_init(pthread_rwlock_t *rwlock, const pthread_rwlockattr_t *attr);
  • 加读锁（共享）：int pthread_rwlock_rdlock(pthread_rwlock_t *rwlock);
  • 加写锁（独占）：int pthread_rwlock_wrlock(pthread_rwlock_t *rwlock);
  • 统一解锁：int pthread_rwlock_unlock(pthread_rwlock_t *rwlock);
  • 销毁：int pthread_rwlock_destroy(pthread_rwlock_t *rwlock);

八、 线程间管道通信 (Pipe for Thread Communication)

虽然多线程间共享全局内存，通常使用锁同步，但是在一些基于事件驱动（如 Actor 模型）的架构设计中，使用管道作为线程间传递事件的信箱，能够完美复用系统的 select / poll / epoll 异步 I/O 监听机制，实现无锁的事件驱动循环。

  • 工作机制：在进程内部调用 pipe()。子线程持有读端（fd[0]）并阻塞在 read 系统调用上；主线程（或者其他生成事件的线程）向写端（fd[1]）写入命令，子线程随即被内核调度唤醒处理任务。
