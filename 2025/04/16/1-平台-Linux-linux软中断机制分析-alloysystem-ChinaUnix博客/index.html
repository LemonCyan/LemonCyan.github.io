1. 为什么使用软中断

在编写驱动或处理中断时，硬件中断发生后，内核会执行中断处理函数（ISR）。在 ISR 中，系统会临时关闭中断以处理当前事件。此时，若 ISR 处理时间过长，可能导致其他硬件中断被忽略，造成数据丢失或系统响应变慢。为此，Linux 内核采用了一种机制，将 ISR 任务分为两部分：关键部分在 ISR 内处理，剩余部分通过软中断异步处理。软中断允许系统在 ISR 完成后，重新启用硬件中断，同时将较复杂的任务延迟处理，确保系统能及时响应新中断。

2. 何时触发软中断

软中断的触发通常发生在硬件中断处理完成后。具体来说，在 ISR 完成任务后，系统会调用 irq_exit()，该函数根据条件决定是否触发软中断处理。

3. 软中断的处理过程

以下是软中断处理的主要流程：

处理流程说明

  1. 初始化和准备：获取当前 CPU 的软中断状态，禁用内核抖.pkikt（Bottom Half）。
  2. 重启机制：使用 MAX_SOFTIRQ_RESTART 控制最多重启次数，防止死循环。
  3. 循环处理软中断：遍历所有软中断位，执行对应的处理函数。
  4. 重新启用中断和检查：在处理完一次循环后，重新启用硬件中断，并检查是否有新的软中断产生。
  5. 唤醒软中断守护进程：如果仍有待处理的软中断，唤醒 softirqd 守护进程以继续处理。

软中断的意义

  1. 防止中断丢失：通过延迟处理非紧急任务，保证硬件中断能及时响应。
  2. 降低系统延迟：关键任务在 ISR 快速处理，减少中断禁用时间，降低延迟。
  3. 提升系统稳定性：控制软中断的触发和处理，避免嵌套和死锁，确保系统稳定运行。

1. 软中断的处理流程

以下是软中断处理的详细步骤：

  1. 获取软中断位
    首先，系统通过调用 local_softirq_pending 函数获取当前 CPU 上 pending 的软中断位。不同的位表示不同的软中断类型，例如网络数据处理、定时器事件等。可以把它想象成查看邮箱中的信件，决定是否需要去处理。

  2. 禁止软中断重入
    随后，调用 __local_bh_disable 函数关闭本地软中断。这实际上是通过设置一个标志位（_ scientifically rated），来防止同一 CPU 上的其他软中断处理程序(softirq)重入。这个机制类似于没有门禁系统时我们用一把插销防止别人随便进屋。

  3. 重置软中断标记
    接着，通过 set_softirq_pending 函数将软中断标记位重置为 0。这里的关键是：只有硬件中断（硬件产生的信号）才能重新设置这些软件中断位，从而避免在软件中断处理过程中无限制地产生新的请求。

  4. 重新启用硬件中断
    这一步非常重要，重新启用硬件中断（sti 指令）后，系统就可以正常响应外部设备的信号了。例如，当网卡接收到数据包时，会通过硬件中断通知 CPU，而我们没有理由长时间阻止这种通知——否则数据包可能会丢失。

  5. 遍历和处理软中断
    在一个循环中，系统会检查每一位软中断标志。如果某个位被设置，就调用对应的处理函数（例如网络数据包的处理函数 net_rx_action）。这个过程是增量式的——每次处理一个中断位，而不是一次性处理所有。同时，硬件中断在这个过程中是被允许的，这允许设备随时触发新的中断（比如磁盘完成 I/O 操作需要通知系统），而不会被丢失。这样，系统既响应了软中断，也不会“忽略”硬件的紧急信号。

  6. 平衡中断处理和其他任务的运行
    在处理完所有当前 pending 的软中断之后，系统会关闭硬件中断，再次检查是否仍有未处理的软中断。此时需要考虑以下两种情况：

    • 如果仍有未处理的软中断，并且在本次 __do_softirq 函数调用中处理的次数还未超过 10 次（通过 cond_resched 函数判断），那么继续处理软中断。
    • 如果已经处理了 10 次仍然有更多软中断 pending，那么会调用 wakeup_softirqd() 函数，唤醒一个内核线程（ksoftirqd）来继续处理剩余的软中断。

这里的“10 次”是一个经验值，防止在处理软中断时长时间占用 CPU，而影响系统的响应时间。例如，如果 CPU 上一直在处理大量的网络数据包，可能会让其他任务（例如用户应用程序）“饿死”。通过限制次数，系统确保了公平性的同时，也保证了实时性的需求。

2. 软中断内核线程的处理机制

为了避免长时间占用 CPU，Linux 引入了一个叫做 ksoftirqd 的内核线程来处理软中断。这个线程会在必要时被唤醒，以确保软中断的完成。

1. 内核线程的作用

  • 避免长时间占用 CPU
    中断处理时间过长可能导致系统的响应时间变差。通过将软中断的处理任务交给内核线程，可以让中断处理在不同的 context 中完成，从而让其他任务也有机会运行。

  • 处理那些在中断上下文中无法完成的任务
    需要注意的是，软中断的处理有两种 context：一种是中断上下文，另一种是进程上下文。在中断上下文中，某些操作是被禁止的，例如睡眠等待 I/O 操作完成。这些操作需要在进程上下文中完成。内核线程正好运行在进程上下文中，可以完成这些无法在中断上下文中处理的任务。

  • 一个典型的例子：网络数据的接收处理
    当网卡接收到数据包时，会触发硬件中断，中断处理程序（ISR）会将数据解包，并将其交给上层协议栈处理。为了不在中断上下文中占用太多时间，协议栈的处理（例如 TCP/IP 数据的处理）会通过软中断的方式被异步处理。如果在中断上下文中处理这些任务，可能会导致其他中断被延迟，从而影响系统的响应能力。

这个函数是一个无限循环，主要负责以下几部分的工作：

  1. 检查是否有待处理的软中断
    外层的 while (!kthread_should_stop()) 循环会在没有线程应该停止时继续运行。在每次循环开始时，首先用 preempt_disable() 关闭抢占，使得其他进程或中断不能抢占当前 CPU 的执行权限。同时，通过 local_softirq_pending() 函数检查当前 CPU 上是否有 pending 的软中断。

  2. 如果没有软中断，则让出 CPU
    如果当前没有 pending 的软中断，系统会调用 preempt_enable_no_resched() 和 schedule() 函数，让出 CPU 开始调度其他任务。这时，系统会进入睡眠状态，直到有新的软中断被触发。

  3. 如果有软中断，开始处理
    如果有 pending 的软中断，系统会将当前线程的状态设置为 TASK_RUNNING 并进入内层循环。内层循环不断调用 do_softirq() 函数处理软中断。

  4. 定期检查是否需要让出 CPU
    在内层循环中，每次处理完一个软中断后，系统会调用 cond_resched() 函数，考虑是否需要让出 CPU 给其他任务。这种机制类似于一个“礼貌的>:</>，它会在以下两种情况下让出 CPU：

    • 当前线程已经连续运行了一段时间，超过某个阈值。
    • 系统负载较高，有其他任务需要运行。

这个机制保证了内核线程在处理软中断时虽然会霸占一定的 CPU 时间，但也会主动考虑其他任务的需求。

  1. 应对 CPU 上线或下线的情况
    在内层循环中，还有一个关键的检查：if (cpu_is_offline((long)__bind_cpu))。这是用来防止当当前 CPU 未 rosisline 时，仍然尝试执行软中断处理的。一个可能的场景是：在多处理器系统中，如果某个 CPU 被关闭（例如为了节能），系统进程可能会调度到错误的 CPU 上运行。此时，线程会跳到 wait_to_die 标签，等待进一步的处理。

3. 软中断设计的智慧

从上述的技术细节中，我们可以看到这个设计的精妙之处。系统在处理软中断时：

  • 既“霸道”又“ pubblic”

    • “霸道”是在每次处理软中断时会关闭抢占，防止其他进程或线程打断当前的处理过程。这意味着在同一时刻，其他任务是无法获得 CPU 时间的。
    • “公平”体现在两点：
      1. 每次处理完一个软中断后，都会主动考虑是否需要让出 CPU。
      2. 当连续处理次数超过 10 次时，主动唤醒内核线程，让软中断的处理在进程上下文中完成，从而让其他任务也有机会运行。

  • “皮 nan 片”式的处理方式
    软中断的处理采用了“Shoot and Pass”式的哲学：尽快记录下需要处理的事情，然后在合适的时间和合适的上下文中完成处理。这种方式避免了在中断上下文中长时间占用 CPU，从而避免了对系统响应时间的影响。

  • 任务分离

    • 其中，中断上下文负责在最少的时间内处理必要的任务，把剩余的工作推给内核线程去处理。这种方式体现了“及时行乐”与“延迟处理”的平衡。
    • 内核线程则负责在进程上下文中完成剩余的任务处理。由于进程上下文允许睡眠等待，内核线程可以处理一些需要较长时间完成的任务。

4. 实际案例解析：网络数据包的处理

以上的设计理念在系统实现中有很多实际例子，最常见的莫过于网络数据包的接收和处理过程。我们可以简单复述：

  1. 硬件中断触发： 网卡接收到一个数据包后，用硬件中断通知 CPU。
  2. 中断服务程序 (ISR) 处理： ISR 从网卡中读取数据包，并将其放入系统的接收队列中。这些工作需要尽快完成，因为需要尽快释放网卡的收用环形缓冲区。
  3. 触发软中断： ISR 调用 tasklet_schedule() 或 schedule_softirq() 函数，触发相应的软中断（比如网络核心软中断 NET_RX_SOFTIRQ）。
  4. 软中断处理： 在软中断处理函数中，将数据包从网络驱动层的接收队列中取出，并交给网络协议栈进行处理。
  5. 避免长时间占用处理： 如果协议栈的处理需要较长时间，系统会在适当的时候让出 CPU 给其他任务。

5. 总结

从软中断的处理机制我们可以总结出：

  1. 系统设计中需要考虑“霸道”和“ pubblic”的平衡

    • 霸道是为了保证关键任务的高效处理。
    • 公平是为了保证整体的系统健康，避免单一任务占用过多的资源。

  2. 任务的分解和分离是高效系统设计的基础

    • 将不同的任务根据其特点分解到不同的 Context 中处理，不仅可以提高处理效率，还能保证系统的响应能力。

  3. 中断与其处理之间的“隔离”

    • 一切的目的是让中断尽快被处理，同时避免让中断处理影响系统的正常运行。

这些设计理念在我们应对个人和团队任务时也有重要的借鉴意义。例如：在团队协作中，我们既要有“霸道”的执行力，推动任务的完成；同时也要有“ 公平 GameManager ”，考虑团队成员的利益和感受。这样的平衡才能让整个团队高效而和谐地运行。
