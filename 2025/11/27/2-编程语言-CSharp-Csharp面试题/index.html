1、维护数据库的完整性、一致性，你喜欢用触发器还是自写业务逻辑？为什么？

优先选择自写业务逻辑，复杂场景辅助触发器，核心原因是「可维护性」和「灵活性」：

  • 「自写业务逻辑」的优势：
    1. 透明性强：逻辑在应用代码中，调试、排查问题直观（比如日志可跟踪）；
    2. 低耦合：不依赖数据库特性，换数据库（如 MySQL→PostgreSQL）时迁移成本低；
    3. 灵活扩展：复杂业务（如跨系统校验、关联第三方接口）可直接集成，触发器难以实现；
    4. 便于测试：可通过单元测试覆盖逻辑，触发器测试需依赖数据库环境。
  • 「触发器」的适用场景：
    1. 强制底层约束（如跨表数据同步、日志自动记录），避免应用层遗漏；
    2. 简单规则（如更新时间自动填充、字段值联动修改），减少重复代码。
  • 总结：触发器是「数据库级别的兜底方案」，业务逻辑是「应用级别的核心方案」，优先用业务逻辑保证代码可维护性，仅在需要强制约束时用触发器。

2、什么是事务？什么是锁？

事务

  • 定义：数据库中一组不可分割的 SQL 操作集合，要么全部执行成功，要么全部执行失败（回滚），核心是保证数据一致性。
  • 核心特性（ACID）：
    • 原子性（Atomicity）：操作不可拆分；
    • 一致性（Consistency）：执行前后数据符合业务规则（如转账后总金额不变）；
    • 隔离性（Isolation）：多个事务并发执行时，相互不干扰；
    • 持久性（Durability）：事务提交后，数据变更永久生效。

锁

  • 定义：数据库为保证事务隔离性，防止并发读写冲突的机制（本质是资源占用权的控制）。
  • 核心作用：解决并发问题（脏读、不可重复读、幻读），按粒度可分为「表锁」「行锁」「页锁」；按类型可分为：
    • 悲观锁：假设冲突必然发生，直接锁定资源（如 select ... for update）；
    • 乐观锁：假设冲突不发生，通过版本号 / 时间戳校验（如 where version = ?）。

3、什么是索引？有什么优点？

索引

  • 定义：数据库中优化查询效率的「数据结构」（主流是 B + 树，哈希索引用于等值查询），本质是对表中一列或多列数据排序后的 “快速查找目录”。

优点

  1. 提升查询效率：避免全表扫描，通过索引直接定位数据，尤其适合 where 条件、join 关联、order by 排序场景；
  2. 降低 IO 开销：B + 树结构将数据层级压缩（通常 3-4 层），一次查询只需 3-4 次磁盘 IO；
  3. 辅助约束：唯一索引可保证字段唯一性，替代部分业务校验；
  4. 加速聚合计算：sum「count「max 等聚合函数可通过索引直接获取结果，无需遍历全表。

（补充：索引缺点是占用存储空间，且插入 / 更新 / 删除时需维护索引，会降低写操作效率，需合理设计）

4、视图是什么？游标是什么？

视图（View）

  • 定义：基于 SQL 查询结果集的「虚拟表」，仅存储查询逻辑（不存储实际数据），数据随原表动态变化。
  • 核心用途：简化复杂查询（将多表关联逻辑封装为视图）、隐藏敏感字段（如只暴露用户姓名，隐藏手机号）、控制权限（授予视图访问权，不暴露原表结构）。

游标（Cursor）

  • 定义：数据库中处理结果集的「指针」，用于逐条读取查询结果集的行数据，支持对单行数据的单独操作。
  • 适用场景：复杂逐行处理（如批量更新时需判断每行数据逻辑），但效率较低（需频繁与数据库交互），非必要不推荐使用（优先用 SQL 批量操作替代）。

5、什么是存储过程？有什么优点？

存储过程

  • 定义：预编译后存储在数据库中的「SQL 语句集合」，可接收参数、包含逻辑判断（if「loop），支持重复调用。

优点

  1. 执行效率高：预编译一次，后续调用无需重新解析 SQL；
  2. 封装复杂逻辑：将多步 SQL 操作（如查询 + 更新 + 插入）封装为一个过程，简化应用层代码；
  3. 减少网络传输：应用层只需传递调用参数（而非大量 SQL 语句），降低网络开销；
  4. 安全性高：可通过权限控制只允许调用存储过程，不直接操作表，避免 SQL 注入；
  5. 代码复用：多个应用 / 模块可共享同一个存储过程，减少重复开发。

6、什么是触发器？

  • 定义：数据库中「自动执行的特殊存储过程」，无需手动调用，由特定事件（insert「update「delete）触发。
  • 核心要素：
    • 触发事件：对表 / 视图的 DML 操作（部分数据库支持 DDL 触发）；
    • 触发时机：before（操作前执行，如数据校验）、after（操作后执行，如日志记录）；
    • 触发对象：通常是表（视图触发需特定数据库支持）。
  • 用途：维护数据完整性（如级联更新关联表数据）、自动记录日志（如操作审计日志）、数据同步（如跨表数据复制）。

7、C# 中的委托是什么？事件是不是一种委托？

委托（Delegate）

  • 定义：C# 中「类型安全的函数指针」，本质是一种引用类型，用于封装具有相同签名（参数列表 + 返回值类型）的方法。
  • 核心作用：实现方法的 “间接调用” 和 “回调机制”，解耦调用者与被调用者（如多线程中传递执行方法、事件通知）。

事件（Event）

  • 结论：事件是一种特殊的委托实例，是委托的 “安全封装版本”。
  • 核心区别：
    • 委托：可在类外部直接调用（delegateInstance()）、赋值（delegateInstance = 方法）；
    • 事件：仅允许在类内部触发（Invoke()），外部只能通过 += 订阅、-= 取消订阅，无法直接调用或赋值，保证封装性和安全性。
  • 关系：事件基于委托实现，遵循 “发布 - 订阅” 模式（发布者触发事件，订阅者响应事件）。

8、概述三层结构体系

三层结构是软件架构的「分层设计模式」，核心是「解耦」，将系统分为 3 层（通常含实体层辅助数据传递）：

层级 核心职责 示例操作
UI 层（表示层） 用户交互：展示数据、接收用户输入，不处理业务逻辑，仅调用 BLL 层 WinForm 表单、Web 页面（ASP.NET）
BLL 层（业务逻辑层） 核心层：处理业务规则、数据验证、逻辑判断，协调 UI 层与 DAL 层的数据流转 登录校验、订单计算、权限判断
DAL 层（数据访问层） 数据交互：仅负责与数据库的 CRUD 操作（查询、插入、更新、删除），不包含业务逻辑 select * from Users、insert into Orders
Model 层（实体层） 数据载体：定义实体类（如 User「Order），映射数据库表或业务对象，用于层间数据传递 public class User { public int Id { get; set; } public string Name { get; set; } }
  • 优点：代码复用、易维护（修改某层不影响其他层）、便于测试（可单独测试 BLL 层）。

9、什么是装箱和拆箱？什么是重载？

装箱（Boxing）与拆箱（Unboxing）

  • 装箱：将「值类型」（如 int「bool「struct）转换为「引用类型」（object 或其派生类）的过程。原理：在堆上分配内存，复制值类型的内容到堆中，返回堆内存地址（引用），有性能开销。示例：int a = 10; object b = a;（a 是值类型，b 是引用类型）

  • 拆箱：将「引用类型」（必须是装箱后的值类型）转换为「值类型」的过程。原理：先校验类型兼容性（如不能将装箱的 int 拆为 string），再从堆中复制数据到栈上，有性能开销。示例：object b = 10; int a = (int)b;

重载（Overload）

  • 定义：在「同一个类中」，方法名相同，但「参数列表不同」（参数个数、类型、顺序不同）的方法定义。
  • 核心规则：
    1. 与返回值类型、访问修饰符无关；
    2. 编译器通过 “参数匹配” 确定调用哪个方法；
  • 示例：public int Add(int a, int b) {} 和 public double Add(double a, double b) {} 是重载。

10、概述序列化

  • 定义：将对象的「状态信息」（属性、字段值）转换为「可存储 / 传输的格式」（如二进制、XML、JSON）的过程；反之，将该格式还原为对象的过程称为「反序列化」。
  • 核心用途：
    1. 对象持久化：将对象保存到文件、数据库（如游戏存档、配置存储）；
    2. 网络传输：跨进程、跨服务器传递对象（如接口调用中传递 JSON 数据）。
  • 常见序列化类型：
    • 二进制序列化：高效、紧凑，仅支持.NET 平台（需标记 [Serializable] 特性）；
    • XML 序列化：易读、跨平台，适合结构化数据；
    • JSON 序列化：轻量、通用，广泛用于 HTTP 接口（如 Newtonsoft.Json、System.Text.Json）。

11、UDP 和 TCP 连接有何异同？

相同点

  • 均为 TCP/IP 协议栈的「传输层协议」，用于实现端到端的数据传输。

核心差异（表格更清晰）

对比维度 TCP（传输控制协议） UDP（用户数据报协议）
连接性 面向连接（三次握手建立连接，四次挥手断开） 无连接（直接发送数据，无需建立 / 断开连接）
可靠性 可靠（重传丢失数据包、排序、流量控制、拥塞控制） 不可靠（无重传机制，可能丢包、乱序、重复）
效率 开销大（需维护连接状态、校验信息），效率低 开销小（仅包含源 / 目的端口、数据长度、校验和），效率高
数据边界 字节流（无边界，需应用层定义分隔符） 数据报（有边界，一次发送一个完整数据报）
适用场景 需可靠传输的场景（HTTP/HTTPS、FTP、邮件、文件传输） 实时性要求高的场景（视频 / 音频直播、DNS、游戏、物联网）

12、分库、分表应用场景？以及带来的问题？

一、应用场景（核心触发条件：数据量 / 并发量达到瓶颈）

  1. 数据量过大：
    • 单表数据量超 1000 万行（查询全表扫描慢，索引维护成本高）；
    • 单库存储超 50GB（备份 / 恢复耗时，磁盘 IO 压力大）。
  2. 并发量过高：单库连接数上限（如 MySQL 默认 151）无法满足高并发请求（如秒杀、峰值 QPS 超 1 万）。
  3. 业务拆分需求：按业务模块垂直拆分（如电商系统拆分为用户库、订单库、商品库），降低模块耦合。

二、分库分表类型

  • 垂直分库 / 分表：按「业务 / 字段」拆分（如用户表拆为用户基本信息表、用户详情表）；
  • 水平分库 / 分表：按「数据行」拆分（如订单表按用户 ID 哈希拆到 10 个分表，按时间范围拆到不同年份的分表）。

三、带来的问题

  1. 分布式事务：跨库 / 跨表操作难以保证 ACID（如订单创建同时扣减库存，涉及订单库和库存库）；
  2. 跨库查询复杂：多表 join、分页（如查询 “所有用户近 3 个月的订单”）需聚合多个分库 / 分表结果；
  3. 全局 ID 生成：需保证分库 / 分表中 ID 的唯一性（不能用自增主键）；
  4. 数据迁移与扩容：新增分库 / 分表时需重新分配数据，可能导致服务不可用；
  5. 运维成本增加：多库多表的监控、备份、故障排查难度提升；
  6. 一致性问题：缓存与数据库、各分表数据同步可能出现不一致。

13、公司组织架构数据库设计（示意）

核心设计「树形部门结构」和「员工 - 部门 - 上级」关联，3 张核心表即可：

1. 部门表（Departments）

字段名 类型 说明
dept_id int 部门 ID（主键，自增）
dept_name varchar(50) 部门名称（如 “技术部”）
parent_dept_id int 父部门 ID（自关联，顶级部门为 0）
manager_id int 部门负责人 ID（外键关联员工表）
create_time datetime 部门创建时间
remark varchar(200) 备注（可选）

2. 员工表（Employees）

字段名 类型 说明
emp_id int 员工 ID（主键，自增）
emp_name varchar(20) 员工姓名
gender tinyint 性别（1 = 男，2 = 女）
position_id int 职位 ID（外键关联职位表）
dept_id int 所属部门 ID（外键关联部门表）
superior_id int 直接上级 ID（自关联，无上级为 0）
entry_time datetime 入职时间
phone varchar(11) 手机号（可选）
status tinyint 状态（1 = 在职，2 = 离职）

3. 职位表（Positions）

字段名 类型 说明
position_id int 职位 ID（主键，自增）
position_name varchar(30) 职位名称（如 “后端开发工程师”“部门经理”）
remark varchar(200) 职位描述（可选）
  • 核心关联：部门树形结构通过 parent_dept_id 自关联；员工与部门通过 dept_id 关联；员工汇报关系通过 superior_id 自关联。

14、两数之和（编程题）

题目分析

  • 要求：在数组中找到和为目标值的两个不同元素，返回其值（题目示例输出 [2,5]，顺序不影响）。
  • 核心思路：用「哈希表」优化时间复杂度（暴力枚举 O (n²)→哈希表 O (n)），遍历数组时记录已访问元素，快速匹配目标差值。

思路解释

  1. 遍历数组中每个元素 num，计算目标差值 complement = target - num；
  2. 检查哈希表中是否存在 complement：存在则说明找到两个数（complement 和 num），返回结果；
  3. 若不存在，将当前 num 加入哈希表（避免后续重复匹配自身）；
  4. 时间复杂度 O (n)（仅遍历一次数组），空间复杂度 O (n)（哈希表存储最多 n-1 个元素）。
