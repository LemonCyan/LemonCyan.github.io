面向 C++程序员的 C#教程

C# 大致可以看作是 C++ 和 Java 的结合体。其核心概念是引用类型（Reference Types）。这一特性借鉴自 Java，如今已被多种主流语言采用，值得深入理解。它是一套巧妙的机制，支持隐式指针和堆内存使用，并且实现了自动垃圾回收（Garbage Collection）。

C# 比 C++ 更冗长，添加和修改小型功能时更顺手，但整体风格难以明确界定。第一部分介绍 C# 中的语法差异与语法糖，第二部分讲解引用类型，第三部分则介绍那些理解引用类型后才合理的更多差异与语法糖。最后第四部分是关于 C# 编程文化的一些说明。

1 与 C++的细微差异

1.1 杂项

  • 移除了 if(n=3) 这类问题。C# 不允许任何类型隐式转换为 bool，因此会将其视为语法错误。你需要将所有 if(n) 替换为 if(n!=0)，不过这并不算麻烦。
    但 if(b=true) 仍然是可怕的逻辑错误，且指针仍会隐式将空/非空转换为真/假：if(ptr) 等价于 if(ptr!=null)。

  • 类型转换语法相反，类型必须放在括号内：使用 (int)n 而非 int(n)。风格借鉴自 Java。

  • char 类型占 2 字节（采用 UTF-16 编码格式，基本兼容 ASCII）。
    若需要将 char[] 用作字节数组，C# 提供了 byte 类型。

  • 点号（.）兼具作用域解析运算符的功能。C++ 中的 c1.age = Cat::maxAge 在 C# 中写成 c1.age = Cat.maxAge。
    更极端的例子：Cat::primeCat.age 会变成 Cat.primeCat.age。设计初衷是让初学者只需学习一个“属于”运算符。
    双冒号运算符并未被移除，而是被重新用途化。global:: 可以作为路径的起始部分。如果你想了解它，它被称为命名空间别名限定符。

  • 没有 typedef。但 using 关键字被重载，可实现有限的类型定义功能：
    using intList = System.Collections.Generic.List<int>;（注意此处使用点号进行作用域解析）。但这种定义仅在当前文件有效，且不能用于基本类型：using tailLen t = int; 会报错。奇怪的是，函数指针有专门的类型定义语法（delegate）。

  • 不支持宏。#define sqr(x) (x*x) 是不允许的。

  • 使用未初始化变量会导致编译错误。编译器对此检查非常严格，有时甚至会在无法跟上你的逻辑时，误判变量未初始化。
    类似地，将可能未初始化的变量传递给函数、传递包含未初始化字段的结构体，都会导致编译错误——无论函数是否会使用这些变量/字段。其目的是确保 C# 程序永远不会读取无效的垃圾值，也不会因此出现难以复现的间歇性错误。

  • var n = 8; 或 var w = "cow"; 会自动推断类型，类似 C++ 的 auto。
    本质上编译器会将 var 替换为具体类型，且该类型必须能被立即确定。理论上这是个小众特性，但在 C# 代码中却被大量使用，甚至毫无意义地替换短类型名。

  • 命名空间中只能包含类和枚举，不能包含变量或函数。
    你需要将这些变量/函数定义为静态成员，放在一个“占位类”中。
    class stringFuncs { // 占位类 public static string bestWord; public static int compareSomehow(string w1, string w2) {… } } 注意，使用时仍通过点号访问：stringFuncs.bestWord = “cow”;`。

  • using 指令的使用受到限制：只能放在文件顶部，且不能指定命名空间中的单个成员。
    最多只能写成 using System.Collections.Generic;，无法直接指定 LinkedList。
    using 可用于命名空间或类，但引用类时需要额外添加 static 关键字：using static stringFuncs;。
    补充说明：using 还有第三种完全不同的用途——using 代码块。当代码块执行完毕时，会立即调用对象的析构函数（用于资源释放）。

  • 没有持久化局部变量：
    void A() { static int n=0; ...} 这种写法不存在。这其实不算问题——C# 鼓励你编写小型类，用成员变量作为持久化存储。

  • C# 支持与 C++ 类似的指针用法，但只能在“不安全模式（UnSafe mode）”下使用，而大多数运行环境并不支持不安全模式。因此，实际上 C# 中没有显式指针，相关功能由引用类型替代。

  • 数组的变化：

    • 方括号是类型的一部分。这一点非常便捷，示例：
      int[] A, B; // 两个 int 类型数组
      int[] func_takeArray_returnArray(int[] A) { ... }// 参数和返回值同理：

    • 数组自带长度信息：通过 Length 成员函数获取。无需再手动传递 。

    • 索引范围检查：若 A[n] 访问越界，会抛出运行时错误。我知道，这让语言看起来像是给初学者用的，但实际编写快速原型代码时很实用——你可以随意使用数组，即使出错也会得到清晰的异常信息和堆栈跟踪。

    • 除了 [][] 交错数组，还支持矩形多维数组。

    • 交错数组的语法与 C++ 相反。它并非简单的“数组的数组”组合，而是有额外规则，必须从左到右声明。 再举一个示例（源自 StackExchange），更清晰地说明这种“从左到右”规则：

  • 函数调用时可选择性地指定参数名。max(3,4) 和 max(x:3, y:4) 完全等价。有时用于自注释代码，或灵活调整参数顺序，实际用途是在众多可选参数中只指定部分：

  • 函数指针的工作方式与 C++ 大致相同。以下代码声明 ff 为指向“接收两个 string 参数、返回 int 的函数”的指针，注意返回类型在末尾而非开头，且使用 System.Func 而非 C++ 风格的 System::Func。使用方式相同，但无需（也不能）添加引用（&）或解引用（*）运算符：

    • 无输入参数时，省略尖括号：System.Action f3; 表示无参无返回值的函数指针。
    • 还有一种旧语法，需先通过 delegate 定义类型别名（类似 typedef）：两种方式完全互通。正因这种旧语法，C# 中的函数指针也被称为“委托（delegate）”。

  • 支持匿名函数（称为 Lambda 表达式，但并非严格意义上的 Lambda——不支持上下文捕获）。函数体语法与普通函数一致。参数部分无需指定类型（编译器可自动推断），返回类型也会自动推断。=> 是匿名函数的关键字。若函数体仅含单个返回语句，可省略 return 和大括号。还有一种使用 delegate 关键字的过时语法，无使用必要，但仍可能在旧代码中见到。

  • C++ 模板函数的功能在 C# 中被拆分为两个特性：
    • 泛型（generics）：语法类似 C++ 模板，但不支持“鸭子类型”——函数体中只能假设类型支持赋值运算符（=），这是一个很大的限制。你可以指定类型必须是类（where T : class），此时函数体可使用 == 进行指针比较，但仍不够实用。 最终可限制输入类型为某个类的子类（where T : Animal），例如从 Animals 列表中筛选 Dog 或 Cat：T aa = Animals[i] as T;。显然，C++ 模板函数的许多实用功能在泛型中无法实现。
    • dynamic 关键字：将变量标记为动态类型，类似模板类型 bool longEnough(dynamic n) { return n.Length > 10; }。限制是：无法简洁地强制两个输入参数为同一类型，也无法直接声明该类型的局部变量（据说可通过 typeof 规避，但较为繁琐）。
  • 不允许用内部作用域变量隐藏外部作用域变量。明确来说，这是将不良编程风格直接列为错误。int i = 8;for(int i=0; i<10; i++) {} // C# 报错：内部 i 隐藏外部 i
  • 有限的“真正引用变量”：不允许 int& n = C[i].age;。但可声明 ref int n，且只能通过“按引用返回”的函数赋值。注意必须添加所有冗余的 ref 关键字。限制是：被引用的对象必须在堆上（但可以是类中的值类型，例如 Cat 中的 age）。
  • 没有头文件或函数原型。但也无需这些——C# 环境会自动扫描所有文件查找依赖。
  • 容器类：
    • List 替代 C++ 的 Vector，基于数组实现。奇怪的是，它包含部分时间复杂度为 O(n) 的成员函数，且没有 pop_back 方法。
    • LinkedList 对应 C++ 的链表类。
    • 哈希表映射类被重命名为 Dictionary（对应 C++ 的 unordered_map）。
    • ArrayList 是 Java 风格的旧版混合类型数组，存储 Object 类型（所有类型的基类），不推荐使用。
  • foreach 循环语法糖：可遍历数组和大多数容器：适用于任何支持外部迭代器的类型（实际上是实现了标准“向前迭代器”接口的类型）。
  • 序列化（Serialization）：无需复杂操作，任何类都可序列化为 XML 格式，之后再反序列化恢复。通常图形界面（GUI）开发中会用到这一特性。标准序列化假设字段是简单的“自有数据”，会递归序列化所有嵌套数据。若数据结构包含指针，则需要额外大量工作。
  • 反射（Reflection）：类的 GetMethods 方法会返回所有函数的签名和名称（包括私有函数），之后可通过这些句柄调用函数。适用于一些特殊场景：例如为任何类快速制作可视化编辑器，或调用编译时未知的类。有时你会看到代码“违规”访问私有变量/函数——它们正是通过反射实现的。反射对实现已知功能没有帮助。

1.2 类/结构体、继承差异

  • 结构体（struct）和类（class）的默认访问修饰符是 private，且 public 不会“继承传递”——必须为每个成员显式添加 public。纯数据结构体的写法继承自 Java 的设计：良好的面向对象编程（OOP）应让开发者“费力”才能暴露破坏封装的公共变量。
  • 类中的所有成员变量都会被自动初始化（本质上是默认构造函数的作用），但只要你自定义了任何构造函数，系统就不会再自动生成默认构造函数。类的成员变量可在定义时指定初始值，编译器会秘密将这些初始化逻辑注入到所有构造函数（包括默认构造函数）中。示例：

  • 不支持类的多重继承。借鉴自 Java 的解决方案是：可实现任意数量的“接口（interface）”。接口是纯契约，本质上是只包含抽象虚函数（=0 函数）的抽象类：

  • 子类中未重写的函数应添加 new 关键字（表示隐藏父类函数），但这是默认行为，因此很少有人显式添加。

    2. 这并非实用特性——建议所有需要重写的函数都显式添加 override。
    3. 官方设计理由：假设 Cat.eat 最初是子类中的普通函数，之后父类 Animal 独立添加了 virtual eat 函数。该规则可帮助发现：旧的 Cat.eat 可能不满足 Animal.eat 的后置条件，因此不应被重写。

  • 没有友元函数（friend functions）。

  • 扩展方法（Extension methods）：纯粹的语法糖——本质是普通函数，但调用方式类似成员函数。它无法访问私有变量，也不能是虚函数——除调用语法外，与普通函数完全一致。定义时需在第一个参数前添加 this。调用方式：c1.passTime(4);。唯一用途是允许通过“对象.方法名”的方式调用，方便在 IDE 中通过下拉列表查找。注意：参数列表中的 this 是关键字的又一次复用。

  • 属性（Properties）：简化 getter/setter 方法对的编写。使用时完全像访问变量，无需 n = c1.getYears(); 和 c1.setYears(4);，直接 n = c1.years; 和 c1.years = 4;。
  • 属性的使用非常普遍，被视为定义简单变量的“标准方式”。系统会为左值（赋值操作）调用 set，为其他场景调用 get。
  • 无逻辑的空 getter/setter 极为常见，因此有专门的“自动属性（auto-properties）”

2 引用类型

这是 C# 与 C++ 最核心的差异，借鉴自 Java。其设计目标是：隐藏指针和堆/栈的细节，为指针和非指针使用提供统一语法，并支持自动垃圾回收。

2.1 工作原理

  • 类型分为两类：不能使用指针的类型，和必须使用指针的类型。
    • 第一类很简单：基本类型（int、float、string、char、bool 等）永远不能有指针——C# 中不存在指向这些类型的指针。
    • 第二类称为引用类型（Reference Types），包括所有自定义类。它们只能是指针，且只能通过 new 在堆上创建——无法像 C++ 那样直接声明类的实例。正因为这一限制，语言可以简化其使用：无需额外语法声明为指针，且会自动解引用。
  • Cat c1 = new Cat(); 本质是堆内存分配，看似开销较大，但由于自动垃圾回收，C# 开发者无需关心堆内存的管理细节。术语说明：c1 被称为“引用（reference）”。

因为 Cat 是引用类型，所以所有指向 Cat 的指针都叫引用。关于该机制的几点观察：

  • 指针永远不能指向栈数据。栈上只能存储“值类型”，且无法创建指向值类型的指针。
  • 没有取地址（&）、解引用（*）或箭头（->）运算符。这些运算符在普通类型中被禁用，在引用类型中则通过隐式方式实现。
  • 即使不需要，也必须使用指针和 new。大多数时候我们想要的是 C++ 中“自动存储期”的 Cat，但 C# 强制我们声明堆上 Cat 的指针。因此，C# 程序会产生大量看似“无意义”的垃圾。从 C++ 开发者的视角来看，C# 的内存管理似乎完全没有章法。
  • 没有二级指针（Cat**）。若需类似功能，需创建一个包含单个引用类型成员的小型类。
  • 没有销毁（destroy）命令。好处是不会出现悬空指针——垃圾回收器会定期通过“可达性分析”回收内存：只要有指针指向某个对象，它就不会被销毁。

3 杂项特性

引用传递

  • C# 支持标准的按引用传递（pass-by-reference），分为“仅输出”和“输入输出”两种专门形式。用 ref 替代了 C++ 的 &，且调用时必须为变量添加 ref 修饰。设计初衷是让调用者明确知道参数会被修改（但实际作用有限）。

仅输出形式用 out 替代 ref（声明和调用时都需替换），并修改了错误规则：

  1. 实参未初始化不会报错（正常情况下这是编译错误）；
  2. 函数内必须为该参数赋值；
  3. 函数内尝试读取该参数会报错。

核心用法就是这些，但从文化角度看，按引用传递属于高级话题。一方面，几乎所有对象都是按指针传递的，无需按引用传递：

另一方面，术语容易混淆——C# 开发者从一开始就将 Cat c1 称为“引用”，因此“按引用传递引用类型”“按引用传递值类型”等表述非常绕口。微软官方文档曾一度错误地声称“所有类都是按引用传递的”。

此外，命名空间中还无理由地包含了常见的列表查询函数——例如“返回所有满足谓词的元素”等。但这些函数都不能修改原始列表，且大多数会创建延迟计算的对象——通常需要在末尾添加 .ToArray() 或 .ToList() 才能立即获取结果。

4 编程文化

  • 他们热衷于新特性，这对一门商业语言来说很奇怪。需注意：你可能会看到对某个“微不足道的语法糖”的热烈追捧。
  • 官方文档是微软的 C# 网站。该文档很多地方不完整，但这似乎被认为是可接受的。
  • 网上普遍认为“只有标记为 interface 的才是接口”，“只有标记为 get/set 的才是访问器”。他们对“隐藏成员变量”的理解非常字面化：一个私有 int 加空实现的 get/set 就被视为“隐藏”。因此，当你看到某些关于 C# 设计的言论觉得奇怪时，不妨思考：“这个人是否不理解面向对象编程首先是一种概念，其次才是实现？”
  • 特性淘汰速度快于谷歌索引更新速度。你经常会发现高排名的搜索结果指向不算太旧的页面，却在讲解两个版本前的繁琐特性。作者自己也可能混淆——部分讲解当前特性的页面中，代码仍混合了过时语法。
