5. 持续演化

System 不动
User 多轮演进
否定压制坏味道
在“感觉对了”之后才写实现

① 多轮演进 Prompt（这是 Vibe Coding 的主循环）

❌ 传统做法（错）

  • 一次 prompt

  • 一次生成

  • 不满意 → 推翻

这是一次性思维。

✅ Vibe Coding 做法（对）

同一个 System，不断修 User

你不是在“重新生成”，
你是在演进一个设计决策。

✅ 正确的多轮节奏（非常重要）

第 1 轮（只看结构）

项目用途： 网络连接状态检测工具 本次关注点： 模块边界与扩展方式 额外约束： 新增检测对象不修改核心

👉 只看目录结构
👉 不看实现

第 2 轮（针对“别扭点”）

在上一次设计的基础上： 我不希望出现模糊职责的模块； 希望能一眼看出检测流程从哪里开始，到哪里结束。

👉 这是审美修正，不是需求变更

第 3 轮（确认稳定点）

请明确哪些模块是稳定核心， 哪些是预期会频繁变化的扩展点。

👉 你在逼 AI 显式表达架构意图

🧠 判断标准

当你发现自己开始说：

“这个结构我能接受长期维护”

说明结构阶段结束。

② 否定式 Prompt 库（压制 AI 坏习惯）

这是 工程师和普通人拉开差距的地方。

🔥 为什么“否定”这么重要？

因为 AI 默认会：

  • 教学化

  • 抽象过度

  • 用 core / common / utils 糊你

你必须提前立禁令。

✅ 否定式 Prompt 的正确放置

  • ❌ 不放 user

  • ✅ 放 system（长期有效）

✅ 推荐你直接加入 system 的“否定条款”

你避免以下设计： - 模糊职责的通用模块（core / common / utils） - 为了扩展而扩展的空壳插件 - 仅在文档中存在、但无法直观使用的扩展点 - 把业务流程隐藏在抽象层之后

👉 这是在约束 AI 的工程审美

🧠 你以后看到“别扭”的正确反应

不是：

哪里不对？

而是：

我要禁止这种味道

③ 什么时候从“结构”切到“实现”（非常关键）

这是 90% 人最容易犯错的地方。

❌ 错误信号（还不能写代码）

  • “我们先把 main.cpp 写出来看看”

  • “先跑起来再说”

  • “逻辑可以边写边调”

👉 这会直接毁掉结构阶段

✅ 正确切换信号（必须同时满足）

只有当你能回答这 3 个问题，才进入实现阶段：

  1. 新增一个设备，要加哪一个文件？

  2. 删除一个模块，不会影响谁？

  3. 核心流程是否不依赖具体实现？

如果你能用一句话回答，说明结构稳了。

🚀 正确的实现顺序（Vibe 专用）

  1. 先实现最小闭环

    • 不是所有功能

    • 只是一条“完整流程”

  2. 实现核心接口，而不是具体逻辑

  3. 最后才写具体设备 / IP / URL 实现

👉 顺序一旦反过来，结构必烂。

1️⃣ ai_client.py

# ai_client.py import requests class AIClient: def __init__(self, api_key, endpoint): self.api_key = api_key self.endpoint = endpoint def run_task(self, system_prompt, task_prompt): payload = { "system": system_prompt, "user": task_prompt } headers = {"Authorization": f"Bearer {self.api_key}"} response = requests.post(self.endpoint, json=payload) response.raise_for_status() return response.text # 原始 content

2️⃣ spec_parser.py

职责：

  • 接收 AI 返回的字符串

  • 校验 JSON

  • 确保每个 file 有 path/action/content

  • 返回 Python 对象

# spec_parser.py import json def parse_spec(content: str): content = content.strip() if not content: raise RuntimeError("AI returned empty content") try: spec = json.loads(content) except json.JSONDecodeError as e: raise RuntimeError(f"AI output is not valid JSON:\n{content}") from e if "files" not in spec or not isinstance(spec["files"], list): raise RuntimeError(f"'files' missing or invalid in spec:\n{spec}") for f in spec["files"]: if not all(k in f for k in ("path", "action", "content")): raise RuntimeError(f"File entry missing required keys:\n{f}") return spec

3️⃣ file_manager.py

职责：

  • 支持 dry-run / apply

  • 安全写入文件（先备份）

  • 自动创建目录

# file_manager.py import os from pathlib import Path import shutil def apply_spec(spec, dry_run=True): for f in spec["files"]: path = Path(f["path"]) content = f["content"] action = f["action"] if dry_run: print(f"[DRY-RUN] {action.upper()} {path}") continue # 确保目录存在 path.parent.mkdir(parents=True, exist_ok=True) # 备份已有文件 if path.exists(): shutil.copy(path, str(path) + ".bak") # 写入文件 with open(path, "w", encoding="utf-8") as fp: fp.write(content) print(f"{action.upper()} {path}")

4️⃣ main.py

CLI 入口：

# __main__.py import argparse from ai_client import AIClient from spec_parser import parse_spec from file_manager import apply_spec def main(): parser = argparse.ArgumentParser() parser.add_argument("--dry-run", action="store_true") parser.add_argument("--task-file", required=True) parser.add_argument("--system-file", required=True) args = parser.parse_args() with open(args.system_file) as f: system_prompt = f.read() with open(args.task_file) as f: task_prompt = f.read() client = AIClient(api_key="YOUR_KEY", endpoint="YOUR_ENDPOINT") raw_content = client.run_task(system_prompt, task_prompt) spec = parse_spec(raw_content) apply_spec(spec, dry_run=args.dry_run) if __name__ == "__main__": main()

你当前的 Schema（我们刚定的）：

  • 易解析

  • 适合「整体生成」

❌ 但它有 3 个限制

  2. 无法删除文件

  3. 不强迫你马上实现所有能力

👉 所以：
Schema 可以“描述更多”，实现可以“慢慢来”

三、升级后的「工程级 JSON Schema（推荐版）」

这是我给你的一版 长期可用、但不臃肿 的结构：

{ "intent": "add | modify | refactor | delete | explain", "scope": "entities/mine", "summary": "为 MineEntity 增加基础金币产出", "changes": { "files": [ { "path": "entities/mine/MineEntity.h", "operation": "create | replace | delete", "content": "完整文件内容（create / replace 必须）" }, { "path": "entities/mine/MineEntity.cpp", "operation": "patch", "patch": { "type": "unified", "content": "--- a/...\n+++ b/...\n@@ ..." } } ] }, "constraints": [ "不修改 core", "不引入新模块" ], "confidence": 0.93 }

四、为什么这个结构「不别扭」

我们一项一项说 设计理由。

1️⃣ operation 而不是 action

原来：

"action": "modify"

现在：

"operation": "replace | patch | delete"

👉 这是工程术语升级

  • replace：我给你整个新文件

  • patch：我只改一部分

  • delete：我确认这个文件要删

不会产生歧义

2️⃣ patch 是“能力描述”，不是强制实现

{ "operation": "patch", "patch": { "type": "unified", "content": "..." } }

你现在可以先不支持 patch：

  • 看到 patch → 报错 or 忽略

  • 等你准备好了，再实现

👉 Schema ≠ 立即实现

这是高级工程师常用技巧。

3️⃣ changes 是“变更域”

为什么不直接 files？

因为以后你可以加：

"changes": { "files": [...], "directories": [...], "configs": [...], "deps": [...] }

👉 结构为扩展留了“自然位置”

4️⃣ constraints 是“自我审计”

这个字段不是给你用的，而是：

  • 你可以在日志里打印

  • 你可以在 CI 里检查

  • 你可以将来做对比分析

👉 这是 VibeCoding 的“思维留痕”

5️⃣ confidence 是一个“工程信号”

"confidence": 0.93

不是准不准的问题，而是：

  • < 0.6 → 自动 dry-run

  • < 0.4 → 拒绝 apply

👉 你在用“概率”约束 AI

这是非常现代的做法。

五、你现在“立刻就该做”的最小升级（重要）

✅ 第一阶段（现在就做）

  • operation: replace

  • 忽略 / 报错：

    • delete

❌ 不要现在就做的事

  • AST patch

  • 智能冲突解决
